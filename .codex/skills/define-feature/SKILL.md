# Define Feature

新機能の要件を徹底的にヒアリングし、実装者が迷わず着手できるレベルまで落とし込む。
要件定義書・実装手順書・GitHub Issueを成果物として作成する。
セッションをまたいだ途中保存・再開に対応する。

**ユーザーとの対話はすべて日本語で行うこと。**

## 並行作業とworktree前提の注意

- このスキルは要件定義と実装手順書を作るためのもの。実装作業そのものは原則として専用worktreeで行う前提で計画する。
- 同じ `docs/features/<機能名>/` を複数セッションで同時に編集しない。途中状態のファイルがある場合は、その内容を読み、どのセッションの作業か確認してから続行する。
- `implementation-plan.md` には、実装時に専用worktreeを使うこと、同じファイルを別セッションで同時編集しないこと、commit/push/PR作成前に確認することを前提としてタスクを分割する。
- 複数レイヤーにまたがる機能は、タスクごとに変更対象ファイルを明確にし、並行実装しても衝突しにくい単位へ分ける。

## Step 0: 起動モードの判定

### 機能名や概要が指定されている場合

ユーザーの依頼で指定された機能名に対応する `docs/features/<機能名>/` ディレクトリを探す。

- **ディレクトリが存在し、`requirements.md` の `status` が `completed` でない場合**: 途中再開モード（Step 0a へ）
- **ディレクトリが存在し、`requirements.md` の `status` が `completed`、かつ `implementation-plan.md` の `status` が `completed` でない場合**: 実装手順書の途中再開モード（Step 0b へ）
- **ディレクトリが存在しない場合**: 新規作成モード（Step 1 へ）
- **両方とも `completed` の場合**: 「この機能の要件定義は完了済みです。実装スキルを追加済みなら `$implement <機能名>` で実装に進めます。」と伝える

### 機能名や概要が指定されていない場合

`docs/features/` 配下のディレクトリを走査する。

- **途中状態（`status` が `completed` でない）のものがある場合**: 一覧表示してユーザーに選択してもらう。

```text
途中状態の要件定義:
1. <機能名A> - ステータス: ヒアリング中（次: 技術設計）
2. <機能名B> - ステータス: ドラフトレビュー中
どの機能の要件定義を再開しますか？ または新しい機能名を入力してください。
```

- **途中状態のものがない場合**: 「新しい機能の要件定義を始めます。機能名または概要を教えてください。」

### Step 0a: ヒアリング途中からの再開

`requirements.md` のフロントマター（`status`, `completed_sections`, `next_section`）を読み取る。
ユーザーに簡潔に伝える。

```text
前回は「<機能名>」の要件定義で、<completed_sections> まで完了しています。
<next_section> のヒアリングから再開します。
```

該当する Step 2 のセクションから続行する。

### Step 0b: 実装手順書の途中再開

`requirements.md` は完了済み。`implementation-plan.md` のフロントマターを読み取り、途中から再開する。

```text
前回は「<機能名>」の要件定義書は完了しており、実装手順書の作成途中です。
続きから再開します。
```

Step 4b から続行する。

## Step 1: 機能概要の把握

ユーザーの依頼または会話のコンテキストから機能の概要を把握する。
情報が不足している場合はユーザーに質問する。

機能名が確定したら、`docs/features/<機能名>/` ディレクトリを作成する。

## Step 2: 徹底ヒアリング

`grill-me` 方式で、以下の領域を順に掘り下げる。各領域で不明点がなくなるまで次に進まない。
ヒアリング中、既存のコードベースを調査し「既存の○○を拡張する形が良さそうです」のような提案も積極的に行う。

**各領域の完了時に自動保存を行う（Step 2e 参照）。**

### 2a. ユーザーストーリー

- 誰がこの機能を使うのか
- 何を実現したいのか
- なぜこの機能が必要なのか
- ユーザーにとってのゴールは何か

完了したら自動保存する。

### 2b. 機能要件

- 画面の動き・ユーザー操作の流れ
- 入力と出力（フォーム項目、バリデーション、表示内容）
- ビジネスルール・制約条件
- エラーケース・境界条件

完了したら自動保存する。

### 2c. 技術設計

- APIエンドポイント（URL、メソッド、リクエスト/レスポンス）
- DBテーブル・カラムの追加・変更
- フロントエンドのコンポーネント構成
- 既存コードの拡張 or 新規作成の判断

完了したら自動保存する。

### 2d. 影響範囲

- 既存機能への影響
- 共通コンポーネント・ユーティリティへの影響
- API・DBスキーマの互換性

完了したら自動保存する。

**重要:** 実装者が迷わず着手できるレベルまで落とし込む。曖昧さが残っていれば追加質問する。

### 2e. 自動保存の仕組み

各領域が完了するたびに、`docs/features/<機能名>/requirements.md` を以下の形式で書き出す（上書き保存）。

```markdown
---
status: ヒアリング中
completed_sections: [ユーザーストーリー, 機能要件]
next_section: 技術設計
---
# <機能名> 要件定義書（ドラフト）

## 1. 概要
（確定した内容）

## 2. ユーザーストーリー
（確定した内容）

## 3. 機能要件
（確定した内容）

## 4. 技術設計
（未着手）

## 5. 影響範囲
（未着手）

## 6. 設計判断の根拠
（未着手）
```

- 確定済みセクションには内容を記載する
- 未着手セクションには「（未着手）」と記載する
- フロントマターの `completed_sections` と `next_section` を更新する

### 2f. 中断の検知

ユーザーが以下のような発言をした場合、中断と判断して現時点の内容を保存する。

- 「今日はここまで」
- 「また明日」
- 「一旦終わり」
- 「保存して終わり」
- 「続きは後で」
- 「ここで中断」

中断時の処理:

1. 現時点の内容を `requirements.md` に保存する（Step 2e の形式）
2. ユーザーに以下を伝える

```text
現在の進捗を保存しました。
- ファイル: docs/features/<機能名>/requirements.md
- ステータス: ヒアリング中
- 完了済み: <completed_sections>
- 次回: <next_section> から再開

再開するには `$define-feature <機能名>` または `/define-feature <機能名>` を実行してください。
```

## Step 3: 要件定義書ドラフトの提示

全ての不明点が解消されたら、`requirements.md` のステータスを更新する。

```yaml
---
status: ドラフトレビュー中
completed_sections: [ユーザーストーリー, 機能要件, 技術設計, 影響範囲]
next_section: null
---
```

要件定義書のドラフトをユーザーに提示する。
以下の構成で提示し、ユーザーの承認を得る。修正指示があれば反映して再提示する。

```markdown
# <機能名> 要件定義書

## 1. 概要
- 目的
- 背景・動機

## 2. ユーザーストーリー
- 対象ユーザー
- ユーザーの目的
- 利用シナリオ

## 3. 機能要件
### 3.1 画面仕様
- 画面レイアウト・操作フロー
- 入力項目・バリデーションルール
- 表示項目・表示ルール

### 3.2 ビジネスルール
- 処理ルール・制約条件
- エラーケース・例外処理

## 4. 技術設計
### 4.1 API設計
- エンドポイント一覧（URL、メソッド、リクエスト/レスポンス）

### 4.2 DB設計
- テーブル定義（新規・変更）
- カラム定義

### 4.3 フロントエンド設計
- コンポーネント構成
- 状態管理

### 4.4 バックエンド設計
- クラス構成（Controller / Service / Repository / Entity / DTO）
- 処理フロー

## 5. 影響範囲
- 変更が必要な既存ファイル
- 既存機能への影響

## 6. 設計判断の根拠
- 主要な設計判断とその理由
```

ユーザーの承認を得たら、`requirements.md` のステータスを `completed` に更新し、フロントマターを以下に変更する。

```yaml
---
status: completed
---
```

ドラフトレビュー中にユーザーが中断を申し出た場合も、Step 2f と同様に保存して中断する。

## Step 4: 成果物の作成

### 4a. 要件定義書の確定

Step 3 で承認済みの `requirements.md` がそのまま確定版となる（すでにファイルに保存済み）。

### 4b. 実装手順書の作成

`docs/features/<機能名>/implementation-plan.md` に実装手順書を作成する。

作成開始時にまず以下のフロントマター付きでファイルを作成する。

```yaml
---
status: draft
---
```

構成:

```markdown
---
status: draft
---
# <機能名> 実装手順書

## 実装タスク

## 並行作業ルール
- 実装は専用worktree（例: `C:\tmp\impl-<summary>`）で行う
- メイン作業ツリーでは原則として編集しない
- 複数セッションで同じファイルを同時編集しない
- commit / push / PR作成前に変更内容とテスト結果を確認する

### タスク1: <タスク名>
- [ ] 完了
- **概要:** タスクの説明
- **変更対象ファイル:**
  - `path/to/file1` - 変更内容
  - `path/to/file2` - 変更内容
- **依存タスク:** なし / タスクN
- **対応Issue:** （Issue作成後に記入）

### タスク2: <タスク名>
...

## 実装順序
1. タスク1（依存なし）
2. タスク2（タスク1に依存）
3. ...
```

実装手順書が完成したら、ユーザーに内容を提示して承認を得る。
承認後、ステータスを `completed` に更新する。

```yaml
---
status: completed
---
```

実装手順書の作成中にユーザーが中断を申し出た場合、`status: draft` のまま保存し、Step 2f と同様の中断メッセージを表示する。

### 4c. GitHub Issue の作成

**前提条件:** `requirements.md` と `implementation-plan.md` の両方が `status: completed` であること。

Issue作成前に、これから作る親Issue・子Issueのタイトルと本文要約をユーザーに提示し、承認を得る。

**親Issue:**

```bash
gh issue create --title "[Feature] <機能名>" --label "enhancement" --body "$(cat <<'EOF'
## 概要
<機能の概要>

## 要件定義書
`docs/features/<機能名>/requirements.md`

## 実装手順書
`docs/features/<機能名>/implementation-plan.md`

## タスク一覧
- [ ] #<子Issue番号>: タスク1
- [ ] #<子Issue番号>: タスク2
...
EOF
)"
```

**子Issue（各タスクごと）:**

```bash
gh issue create --title "<タスク名>" --label "enhancement" --body "$(cat <<'EOF'
## 親Issue
#<親Issue番号>

## 概要
<タスクの説明>

## 変更対象ファイル
- `path/to/file1` - 変更内容
- `path/to/file2` - 変更内容

## 依存タスク
なし / #<依存する子Issue番号>

## 完了条件
- <具体的な完了条件>
EOF
)"
```

子Issueを全て作成した後、親Issueのタスク一覧に子Issue番号を反映する。
実装手順書の各タスクにも対応するIssue番号を記入する。

## Step 5: 完了報告

ユーザーに以下を報告する。

- 作成した要件定義書のパス
- 作成した実装手順書のパス
- 親IssueのURL
- 子Issue一覧（番号・タイトル・URL）
- 「実装スキルを追加済みなら `$implement <機能名>` で実装に進めます」
