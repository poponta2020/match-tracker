# Updatefile

直近の実装変更に基づき、以下の3つのドキュメントを最新の状態に更新する。

## 対象ファイル

1. `docs/SPECIFICATION.md` - 仕様書
2. `docs/DESIGN.md` - 設計書（DB設計・API設計含む）
3. `docs/SCREEN_LIST.md` - 画面一覧

## 手順

### 1. 差分の把握

以下を並列で実行し、直近の変更内容を把握する。

- `git diff HEAD --name-only` で変更ファイル一覧を取得
- `git diff HEAD` でコードの差分を確認
- 直前のコミットログ（`git log --oneline -5`）も確認

### 2. 影響範囲の特定

変更内容から、以下のカテゴリに影響があるか判定する。

- **画面の追加・削除・パス変更**: `SCREEN_LIST.md` を更新
- **APIエンドポイントの追加・変更**: `DESIGN.md` の API設計セクションを更新
- **DBテーブル・カラムの追加・変更**: `DESIGN.md` の DB設計セクションを更新
- **機能仕様の変更（ロール・フロー・認証等）**: `SPECIFICATION.md` を更新
- **ER関連の変更**: `DESIGN.md` の ER図セクションを更新

### 3. ドキュメント更新

影響があるドキュメントのみを更新する。更新不要であればスキップする。

更新時のルール:
- 既存のフォーマット・文体・セクション構成に合わせる
- テーブルの番号体系を崩さない（セクション番号 `4.2.1` のように既存の間に挿入）
- ファイルパス一覧（`SCREEN_LIST.md` 末尾のツリー）も忘れずに更新する
- 不要になった記述は削除する（古い画面パスなど）

### 4. 結果を報告

以下の形式でユーザーに報告する。

```text
ドキュメント更新結果:
- SPECIFICATION.md: 更新済み（変更箇所の概要）/ 変更不要
- DESIGN.md: 更新済み（変更箇所の概要）/ 変更不要
- SCREEN_LIST.md: 更新済み（変更箇所の概要）/ 変更不要
```

## 注意事項

- ドキュメントは大きいので、該当箇所を特定してから部分更新する（全文書き換えしない）
- 推測で書かない。実際のコード（エンティティ・コントローラ・ページコンポーネント）を確認してから更新する
- コミットは自動では行わない。ユーザーに確認を取る
