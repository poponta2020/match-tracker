## 2. 環境変数の設定

### 2.1 LINE_ENCRYPTION_KEY の生成

チャネルシークレットとアクセストークンをDBに暗号化保存するためのキーです。
任意の文字列を指定できます（内部でSHA-256ハッシュされて32バイトキーに正規化されます）。

**生成方法（いずれか）:**

```bash
# 方法1: opensslでランダム文字列を生成（推奨）
openssl rand -base64 32

# 方法2: uuidを使う
uuidgen

# 方法3: pythonで生成
python3 -c "import secrets; print(secrets.token_urlsafe(32))"
```

**出力例:** `K7x2mP9qRt4wYz1aBcDeFgHiJkLmNoPqRsTuVwXyZ0=`

### 2.2 環境変数一覧

| 環境変数 | 必須 | デフォルト値 | 説明 |
|----------|------|-------------|------|
| `LINE_ENCRYPTION_KEY` | **Yes** | （なし） | DB暗号化キー。**一度設定したら変更不可**（既存データが復号できなくなる） |
| `LINE_MONTHLY_MESSAGE_LIMIT` | No | `200` | チャネルあたりの月間送信上限 |
| `LINE_RECLAIM_INACTIVE_DAYS` | No | `90` | 未ログインでチャネル回収警告する日数 |
| `LINE_RECLAIM_GRACE_DAYS` | No | `7` | 回収警告後の猶予日数 |

### 2.3 設定場所

**ローカル開発（application.properties に既に定義済み）:**
```bash
# .envファイルや起動時に環境変数として渡す
export LINE_ENCRYPTION_KEY="ここに生成したキーを設定"
```

**Render.com（本番）:**
1. Renderダッシュボード → karuta-tracker-api → Environment
2. 「Add Environment Variable」で以下を追加:
   - Key: `LINE_ENCRYPTION_KEY`
   - Value: 生成したキー文字列

> **重要:** `LINE_ENCRYPTION_KEY` は秘密情報です。Gitにコミットしたり、他者に共有しないでください。

## 3. LINEチャネルの作成

ユーザー1人に対し1つのLINE Messaging APIチャネルが必要です。
以下の手順を**必要なチャネル数分**繰り返します。

### 3.1 LINE Developersコンソールへのアクセス

1. https://developers.line.biz/console/ にアクセス
2. LINEアカウントでログイン

### 3.2 プロバイダーの作成（初回のみ）

1. 「プロバイダー」→「作成」
2. プロバイダー名を入力（例: `かるた練習管理`）
3. 「作成」をクリック

> 1つのプロバイダーの下に複数のチャネルを作成できます。

### 3.3 Messaging APIチャネルの作成

1. プロバイダーを選択
2. 「新規チャネル作成」→「Messaging API」を選択
3. 以下を入力:

| 項目 | 入力内容 | 備考 |
|------|---------|------|
| チャネルの種類 | Messaging API | 固定 |
| チャネル名 | `かるた通知01` など | ユーザーに表示される名前。後から変更可能 |
| チャネル説明 | `かるた練習管理の通知用アカウント` | 任意 |
| 大業種 | `個人` or `団体` | 適切なものを選択 |
| 小業種 | `スポーツ・フィットネス` など | 適切なものを選択 |
| メールアドレス | 管理者メールアドレス | LINE Developers登録のメール |

4. 利用規約に同意して「作成」

### 3.4 必要な情報の収集

チャネル作成後、以下の3つの情報を控えます:

#### (A) チャネルID
- 「チャネル基本設定」タブ → **チャネルID**

#### (B) チャネルシークレット
- 「チャネル基本設定」タブ → **チャネルシークレット**

#### (C) チャネルアクセストークン（長期）
- 「Messaging API設定」タブ → 下部の「チャネルアクセストークン（長期）」
- 「発行」ボタンをクリックして取得
- **発行後は必ずコピーして控えてください**（再表示不可、再発行は可能）

### 3.5 友だち追加URL・QRコードの取得（任意だが推奨）

- 「Messaging API設定」タブ → **QRコード**（画像を右クリック保存 or URLをコピー）
- 友だち追加URL: `https://line.me/R/ti/p/@{ベーシックID}`
  - ベーシックIDは「Messaging API設定」タブの上部に表示
  - 例: `https://line.me/R/ti/p/@123abcde`

> QRコード画像は外部からアクセス可能なURLに置く必要があります。
> GitHub Issueに画像を貼り付けてURLを取得する方法や、S3/Cloudinaryなどが使えます。

### 3.6 応答設定の変更

1. 「Messaging API設定」タブ → 「LINE公式アカウント機能」の「応答メッセージ」→「編集」
2. LINE Official Account Manager が開く
3. 「応答設定」で以下を設定:

| 設定項目 | 値 |
|---------|-----|
| 応答メッセージ | **オフ** |
| あいさつメッセージ | **オフ**（任意。オンのままでも動作に影響なし） |
| Webhook | **オン** |

> 応答メッセージをオフにしないと、ユーザーがメッセージを送るたびに自動返信が発生し、送信数を消費します。

## 4. Webhook URLの設定

### 4.1 Webhook URLの形式

各チャネルのWebhook URLは以下の形式です:

```
https://{あなたのAPIドメイン}/api/line/webhook/{チャネルのDBのID}
```

**重要:** `{チャネルのDBのID}` は LINE のチャネルIDではなく、アプリのDBに登録した後に振られる `line_channels.id`（AUTO_INCREMENT）です。

### 4.2 設定手順

1. まず[手順6](#6-チャネルのアプリ登録)でチャネルをアプリに登録する
2. 登録後、管理画面でチャネルのDB IDを確認（例: `1`）
3. LINE Developersコンソール → 対象チャネル → 「Messaging API設定」
4. 「Webhook URL」に以下を入力:
   ```
   https://karuta-tracker-api.onrender.com/api/line/webhook/1
   ```
5. 「Webhookの利用」を **オン** にする
6. 「検証」ボタンをクリックして接続確認（200が返ればOK）

### 4.3 Webhook URLの対応表（例）

| DB ID | チャネル名 | Webhook URL |
|-------|-----------|------------|
| 1 | かるた通知01 | `https://{domain}/api/line/webhook/1` |
| 2 | かるた通知02 | `https://{domain}/api/line/webhook/2` |
| 3 | かるた通知03 | `https://{domain}/api/line/webhook/3` |
| ... | ... | ... |

## 5. データベースの準備

### 5.1 マイグレーションの実行

`ddl-auto=update` を使用しているため、アプリ起動時にテーブルは自動作成されます。

手動で作成する場合は以下のSQLファイルを実行:

```bash
psql -U karuta -d karuta_tracker -f database/line_notification_schema.sql
```

### 5.2 デフォルトスケジュール設定の確認

マイグレーションSQLにはデフォルトのスケジュール設定が含まれています:
- 参加予定リマインダー: 1日前に送信
- 締め切りリマインダー: 3日前と1日前に送信

`ddl-auto=update` の場合はデータが自動挿入されないため、管理画面（`/admin/line/schedule`）から初期設定を行ってください。

## 6. チャネルのアプリ登録

### 6.1 管理画面から個別登録

1. アプリにSUPER_ADMINでログイン
2. ハンバーガーメニュー → 「LINEチャネル管理」
3. 「チャネル追加」ボタンをクリック
4. 以下を入力:

| フィールド | 入力内容 | 例 |
|-----------|---------|-----|
| 管理名 | 識別用の名前 | `かるた通知01` |
| チャネルID * | 手順3.4(A)で控えたID | `1234567890` |
| チャネルシークレット * | 手順3.4(B)で控えた値 | `abcdef0123456789...` |
| アクセストークン * | 手順3.4(C)で控えた値 | `eyJhbGciOi...` |
| 友だち追加URL | 手順3.5のURL | `https://line.me/R/ti/p/@123abcde` |
| QRコード画像URL | QRコード画像のURL | `https://qr-official.line.me/...` |

5. 「登録」をクリック

> 入力したシークレットとアクセストークンは `LINE_ENCRYPTION_KEY` で暗号化されてDBに保存されます。

### 6.2 API経由で一括登録

多数のチャネルを一度に登録する場合:

```bash
curl -X POST https://{domain}/api/admin/line/channels/import \
  -H "Content-Type: application/json" \
  -H "X-User-Role: SUPER_ADMIN" \
  -H "X-Player-Id: 1" \
  -d '[
    {
      "channelName": "かるた通知01",
      "lineChannelId": "1234567890",
      "channelSecret": "secret1",
      "channelAccessToken": "token1",
      "friendAddUrl": "https://line.me/R/ti/p/@aaa",
      "qrCodeUrl": "https://example.com/qr1.png"
    },
    {
      "channelName": "かるた通知02",
      "lineChannelId": "1234567891",
      "channelSecret": "secret2",
      "channelAccessToken": "token2",
      "friendAddUrl": "https://line.me/R/ti/p/@bbb",
      "qrCodeUrl": "https://example.com/qr2.png"
    }
  ]'
```

### 6.3 登録後の確認

- 管理画面のチャネル一覧でステータスが「空き」になっていることを確認
- この時点でDB IDが確定するので、[手順4](#4-webhook-urlの設定)に戻ってWebhook URLを設定

## 7. 動作確認

### 7.1 チャネル割り当てテスト

1. 一般ユーザーでアプリにログイン
2. ハンバーガーメニュー → 「LINE通知設定」
3. 「LINE通知を有効にする」をクリック
4. QRコードまたはURLが表示されることを確認
5. 管理画面で対象チャネルのステータスが「割当済」に変わることを確認

### 7.2 友だち追加テスト

1. スマホのLINEアプリでQRコードを読み取り、友だち追加
2. LINE通知設定画面で「ステータス更新」をクリック
3. ステータスが「LINE通知: 有効」（緑色）に変わることを確認
4. 管理画面で対象チャネルのステータスが「連携済」に変わることを確認

### 7.3 通知送信テスト

1. ADMIN以上のユーザーでログイン
2. 抽選結果画面 → 「LINE通知送信」ボタンをクリック
3. LINEに抽選結果メッセージが届くことを確認

### 7.4 Postbackテスト（キャンセル待ち連絡）

1. キャンセル待ちの状態を作成（テスト用に定員を少なめに設定）
2. 当選者がキャンセル → 繰り上げ発生
3. LINE上に「参加する」「辞退する」ボタン付きメッセージが届くことを確認
4. ボタンをタップして、アプリ側のステータスが変わることを確認
5. 確認メッセージがLINEに届くことを確認

## 8. チャネル量産の手順

ユーザー数に合わせてチャネルを大量に作成する場合の効率的な手順です。

### 8.1 準備するもの

- LINEアカウント（1アカウントで100チャネルまで作成可能）
- 必要に応じて複数のLINEアカウント
- スプレッドシート（情報管理用）

### 8.2 効率化のコツ

1. **命名規則を決める**: `かるた通知01`〜`かるた通知99` のように連番
2. **スプレッドシートで管理**: チャネルID・シークレット・トークンを一覧で記録
3. **一括登録APIを活用**: スプレッドシートからJSON配列を生成して一括POST
4. **Webhook URLは登録後にまとめて設定**: DB IDが確定してから一括設定

### 8.3 推奨チャネル数

| ユーザー規模 | 推奨チャネル数 | 備考 |
|-------------|-------------|------|
| 〜10人 | 12〜15 | アクティブユーザー + 余裕分 |
| 〜30人 | 35〜40 | 同上 |
| 〜50人 | 55〜60 | 同上 |

> チャネル回収機能（90日未ログインで自動回収）があるため、ユーザー数より少し多めに用意すれば十分です。

### 8.4 作業の流れ（まとめ）

```
1. LINE Developersでチャネルを作成（手順3を繰り返す）
    ↓
2. 各チャネルの情報をスプレッドシートに記録
    ↓
3. 環境変数 LINE_ENCRYPTION_KEY を設定（手順2）
    ↓
4. アプリにチャネルを一括登録（手順6.2）
    ↓
5. DB IDを確認し、Webhook URLを各チャネルに設定（手順4）
    ↓
6. 動作確認（手順7）
```

## 9. トラブルシューティング

### Webhook検証で失敗する

- アプリが起動しているか確認
- URLのDB IDが正しいか確認（LINEのチャネルIDではなく、DBのAUTO_INCREMENT ID）
- HTTPSであることを確認（HTTPは不可）
- Render.comの場合、初回アクセスでコールドスタートに時間がかかることがある → 少し待ってリトライ

### LINE通知が届かない

- 管理画面でチャネルステータスが「連携済」になっているか確認
- ユーザーのLINE通知設定で対象の通知種別がONになっているか確認
- 月間送信数が200に達していないか確認（管理画面のチャネル一覧で確認可能）
- サーバーログで `LINE push message` 関連のエラーを確認

### 「利用可能なLINEチャネルがありません」エラー

- ステータスが「空き(AVAILABLE)」のチャネルが0 → チャネルを追加するか、不要な割り当てを解除

### Postback（参加/辞退ボタン）が反応しない

- Webhook URLが正しく設定されているか確認
- LINE Developersコンソールで「Webhookの利用」がオンか確認
- 応答メッセージがオフか確認
- サーバーログで `postback` 関連のエラーを確認

### 暗号化キーを変更したい

**暗号化キーの変更は既存データの復号ができなくなるため、基本的に推奨しません。**
やむを得ず変更する場合:
1. 管理画面で全チャネルを削除
2. 環境変数 `LINE_ENCRYPTION_KEY` を新しいキーに変更
3. アプリを再起動
4. チャネルを再登録

## 補足: LINE Messaging APIの制約

| 項目 | 制約 |
|------|------|
| 無料プランの月間メッセージ数 | 200通/チャネル |
| Push API | ユーザーのlineUserIdが必要（友だち追加済み） |
| Webhook | HTTPSのみ。自己署名証明書不可 |
| チャネルアクセストークン | 有効期限なし（長期トークン） |
| 1リクエストあたりのメッセージ数 | 最大5つのメッセージオブジェクト |
| Confirm Template | テキスト + 2ボタンまで |
