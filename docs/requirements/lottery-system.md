## 1. 概要

練習参加者が定員を超えた場合に抽選を行い、公平に参加者を決定するシステムを導入する。
落選者にはキャンセル待ち番号を付与し、当選者がキャンセルした際に番号順に繰り上げ参加の機会を提供する。

### 1.1 背景・目的

- 練習の参加希望者が定員を超えるケースに対応する
- 抽選により公平な参加機会を提供する
- キャンセル待ちの仕組みにより、枠を無駄にしない運用を実現する

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| セッション | 1日の練習全体。日付で一意に特定される |
| 試合 | セッション内の個別の試合（match_number: 1〜N） |
| 定員 | セッションに設定される参加上限人数。各試合に共通で適用される |
| 参加希望 | 締め切り前にプレイヤーが登録する参加意思表示 |
| 当選 | 抽選の結果、参加が確定した状態 |
| 落選 | 抽選の結果、参加できなかった状態 |
| キャンセル待ち | 落選者に割り当てられる繰り上げ順位付きの待機状態 |
| 繰り上げ | キャンセル発生時にキャンセル待ちの人が参加権を得ること |
| 締め切り | 参加希望登録の期限。翌月分の練習に対し、前月末日の0時 |
| `todayTaken` | そのセッション（1日）で取れた試合数。抽選中に当選が確定するたびに+1する。ルール1（その日の一巡保証）の基準値 |
| `recentTaken` | そのセッション日付から遡って直近30日間に、同一団体で取れた試合数（WON件数）。暦月リセットはなく常にローリングで算出する。ルール2（直近30日の重み付き抽選）の基準値 |
| ルール1（その日の一巡保証） | 定員超過の試合で、`todayTaken` が最小の候補者から確定的に座席を割り当てる仕組み。1日1試合も取れない人が構造的に発生しないことを保証する |
| ルール2（直近30日の重み付き抽選） | `todayTaken` が同数で並ぶ候補内でのみ、`recentTaken` が少ない選手ほど当選しやすくなる重み付き抽選 |
| キャップ（重みの上限） | ルール2の重み計算で用いる上限値。その試合の候補者の `recentTaken` 分布から、システム設定 `lottery_weight_cap_percentile`（団体別、デフォルト30）が示すパーセンタイルとして毎回算出する。キャップ以上の候補者は全員同一の重みになる |

## 3. 全体業務フロー

```
[締め切り前]          [締め切り・抽選]         [抽選後]              [練習当日]
     │                     │                    │                    │
  参加希望登録          自動抽選実行          結果確認・通知          練習実施
  (自由に変更可能)      (前月末日 0:00)       当選者→参加確定         │
     │                     │                落選者→キャンセル待ち     │
     │                     │                    │                    │
     │                     │              [キャンセル発生時]          │
     │                     │                キャンセル待ち            │
     │                     │                1番に通知                 │
     │                     │                  ↓                      │
     │                     │              24h以内に応答               │
     │                     │              承諾→当選                   │
     │                     │              辞退/無応答→次の番号へ      │
     │                     │                                         │
     │                     │              [当日キャンセル]             │
     │                     │                アプリでキャンセル手続き   │
     │                     │                →直接連絡で調整           │
     │                     │                →連絡を受けた人が         │
     │                     │                  アプリで参加登録        │
```

### 3.1 タイムライン例（4月の練習の場合）

| 時期 | 状態 | 操作可能な内容 |
|------|------|-------------|
| 〜3/30 23:59 | 登録受付中 | 参加希望の登録・変更・取消が自由 |
| 3/31 00:00 | 締め切り・抽選実行 | 登録不可。システムが自動抽選を実行 |
| 3/31 00:00〜 | 抽選結果確定 | 結果確認。定員未到達かつキャンセル待ちなしの試合は自由登録可。当選者はキャンセル可 |
| 練習当日 | 当日モード | 当日キャンセルはアプリで手続き後、直接連絡で調整 |

## 4. 機能要件

### 4.1 参加希望登録（締め切り前）

現行の月次一括登録機能を維持する。

- プレイヤーは翌月のセッション・試合を選択して参加希望を登録する
- 締め切り前であれば何度でも変更・取消が可能
- 既存のチェックボックスUIを継続使用する
- 登録されたデータのステータスは「参加希望（PENDING）」とする

### 4.2 自動抽選処理

#### 4.2.1 実行タイミング

- **自動実行**: 前月末日の0時（例：4月分は3月31日 00:00:00）
- スケジューラにより自動実行する

#### 4.2.2 対象

- 翌月の全セッションのうち、以下の条件を満たす試合が抽選対象:
  - セッションに定員（capacity）が設定されている
  - 当該試合の参加希望者数が定員を超えている
- 以下は抽選対象外（全員当選）:
  - 定員が未設定のセッション
  - 参加希望者数が定員以下の試合

#### 4.2.3 抽選アルゴリズム

**処理順序**:
1. セッションを日付昇順で処理する
2. 各セッション内で試合を番号昇順で処理する

**各試合の抽選処理**（詳細な選抜手順は `docs/features/lottery-fair-share-rotation/requirements.md` §3.3〜§3.4 を出典とする）:

```
入力: 当該試合の参加希望者リスト、定員、管理者指定優先選手ID、todayTaken（このセッション内で各選手が既に取れた試合数）、
      recentTaken（各選手の直近30日の取得数）

1. 定員判定
   - 定員から当該試合の既存当選者数（WON/OFFERED）を差し引く
   - 参加希望者数が定員以下の場合、全員当選して終了

2. 座席の割り当て（2プールに分けて同じ選抜手続きを適用）
   a. 管理者指定優先プール: 希望を出している優先選手を先に選抜する。優先選手だけで定員を超える場合は優先選手同士に下記の選抜手続きを適用する
   b. その他プール: 残り定員に対して同じ選抜手続きを適用する

   選抜手続き（座席が埋まるまで1人ずつ繰り返す）:
   - ルール1（確定）: プール内で todayTaken が最も少ない人たちを候補とする
   - ルール2（確率、候補が2人以上のときのみ）: 候補の recentTaken からパーセンタイル（システム設定 lottery_weight_cap_percentile、デフォルト30）でキャップを算出し、
     recentTaken が少ないほど当選しやすくなる重み付き抽選（重み = 1 / (min(recentTaken, キャップ) + 1)）で1人を選ぶ
   - 選ばれた人を当選とし、プールから除外して todayTaken・recentTaken をそれぞれ+1し、座席が埋まるまで繰り返す

3. 当落の確定
   - 座席に入った人: status = WON
   - 座席に入らなかった人: status = WAITLISTED

4. キャンセル待ち番号の付与
   - 落選者に対して手順2と同じ選抜手続きを続行し、引かれた順に1から連番でキャンセル待ち番号を付与する
   - バケット順は当選時と同じ（管理者指定優先プール→その他プール）
   - 前試合からのキャンセル待ち番号の引き継ぎは行わない（毎試合この手続きのみで順番を決める）
```

#### 4.2.4 抽選実行の信頼性

- サーバー障害等で抽選が実行されなかった場合に備え、起動時に未実行の抽選がないかチェックし、あれば実行するリトライ機構を設ける
- 抽選の実行履歴を記録する（実行日時、対象月、処理結果）

### 4.3 抽選結果の管理

#### 4.3.1 ステータス一覧

| ステータス | 説明 |
|-----------|------|
| PENDING | 参加希望（抽選前） |
| WON | 当選（抽選当選、管理者指定優先による当選、定員以下による自動当選を含む） |
| WAITLISTED | キャンセル待ち（キャンセル待ち番号付き） |
| OFFERED | 繰り上げ通知済み（応答待ち） |
| DECLINED | 繰り上げ辞退（明示的辞退または応答期限切れ） |
| CANCELLED | 当選後キャンセル |

#### 4.3.2 ステータス遷移図

```
PENDING ──[抽選: 当選]──→ WON ──[キャンセル]──→ CANCELLED
   │                       ↑
   │                       │[承諾]
   └──[抽選: 落選]──→ WAITLISTED ──[繰り上げ通知]──→ OFFERED
                                                       │
                                                       ├──[承諾]──→ WON
                                                       └──[辞退/期限切れ]──→ DECLINED
```

### 4.4 キャンセル・繰り上げフロー（当日以外）

#### 4.4.1 キャンセル

- 当選者は練習前日（23:59）までいつでもキャンセルが可能
- キャンセル時、当該試合にキャンセル待ちの人がいる場合、繰り上げフローを開始する

#### 4.4.2 繰り上げフロー

```
1. 当選者がキャンセル
2. キャンセル待ち1番に通知（アプリ内通知 + Web Push通知）
3. 応答期限: min(通知から24時間, 練習日前日の23:59) のうち早い方
4. 期限内に応答:
   - 「参加する」→ ステータスをWONに変更。フロー終了
   - 「参加しない」→ ステータスをDECLINEDに変更。次の番号（2番）に通知。手順3へ
5. 期限切れ（無応答）:
   - ステータスを自動的にDECLINEDに変更。次の番号に通知。手順3へ
6. 全キャンセル待ち者が辞退/無応答の場合:
   - 枠は空きのまま
   - キャンセル待ちリストが空になるため、当該試合は自由登録可能となる
```

#### 4.4.3 複数同時キャンセル

- 同一試合で複数の当選者が同時期にキャンセルした場合、それぞれのキャンセルに対して順番にキャンセル待ちの次の人へ通知する
- 例: 待ち1番にはキャンセルAの通知、待ち2番にはキャンセルBの通知

### 4.5 当日キャンセル

- 練習当日（0:00以降）のキャンセルには、アプリによる繰り上げ通知フローを適用しない
- 当選者はアプリ上でキャンセル手続きを行う（ステータスをCANCELLEDに変更）
- キャンセルした人がキャンセル待ちの人に直接連絡して調整する
- 連絡を受けた人はアプリ上で参加登録を行う
  - 当日はキャンセル待ちのステータス制約を解除し、通常の空き枠への登録として扱う
  - キャンセル待ちの番号順の制約は無視する（待ち3番の人が先に登録しても良い）
  - キャンセル待ちに入っていなかった人でも参加登録が可能

### 4.6 抽選後の自由登録

抽選後であっても、以下の条件を**両方**満たす試合には自由に参加登録・登録解除が可能:

1. 現在の参加者数が定員に到達していない
2. 当該試合にキャンセル待ちの人がいない（全員が繰り上げ済み or 辞退済み、またはそもそも抽選がなかった）

- キャンセル待ちの人が1人でも残っている場合、その試合への新規自由登録は不可とする
- 自由登録した人のステータスはWONとする

### 4.7 管理者機能

#### 4.7.1 再抽選

- 管理者はセッション単位で再抽選を手動実行できる
- 用途: 抽選後に不備が発覚した場合の即時やり直し
- 再抽選の対象:
  - 当該セッションに紐づく全参加者（当選者＋キャンセル待ち者）
  - ただし、**キャンセル待ちから繰り上がって承諾した人は維持**（再抽選の対象外）
  - 明示的にキャンセルした人は対象外
- 再抽選のルール:
  - 通常の抽選アルゴリズムと同じ（試合番号昇順、ルール1「その日の一巡保証」・ルール2「直近30日の重み付き抽選」を適用）
  - `recentTaken`（直近30日の取得数）は他セッションでの実際の取得実績から算出されるため、月内の落選経験を別途集計する仕組みは持たない
- 再抽選は当該セッションのみに影響し、他セッションの結果には波及しない

#### 4.7.2 手動編集

- 管理者は以下の操作が可能:
  - 参加者の追加（定員超過も可）
  - 参加者の削除
  - キャンセル待ち順番の変更
  - 当選↔キャンセル待ちのステータス変更
- 手動編集は抽選結果の上書きとして扱う

#### 4.7.3 締め切り後のセッション追加

- 締め切り後に管理者が追加したセッションは抽選対象外とする
- 先着順で参加登録を受け付ける（定員に達するまで自由登録）

### 4.8 通知機能

#### 4.8.1 アプリ内通知

- 通知一覧画面を新設する
- 未読通知数をヘッダー等に表示する（バッジ）
- 通知対象イベント:
  - 抽選結果の確定（当選・落選）
  - キャンセル待ちからの繰り上げ連絡
  - 繰り上げの応答期限切れ警告（期限の一定時間前）

#### 4.8.2 Web Push通知

- Service Workerを導入し、ブラウザのPush APIを利用する
- ユーザーがブラウザの通知許可をONにする必要がある
- Push通知の許可を求めるUIフローを設ける（初回ログイン時 or 設定画面）
- Push通知はアプリ内通知と同じイベントで送信する
- Push通知を許可していない場合でも、アプリ内通知は常に機能する

## 5. 画面要件

### 5.1 既存画面の変更

#### 5.1.1 練習参加登録画面（PracticeParticipation.jsx）

**締め切り前**:
- 現行通り。チェックボックスで試合を選択し、参加希望を登録する
- 締め切り日時を画面上に表示する

**締め切り後（抽選済み）**:
- 各試合について、自分のステータスを表示する:
  - 当選: 「当選」ラベル＋キャンセルボタン
  - キャンセル待ち: 「キャンセル待ち N番」ラベル
  - 未申込で自由登録可能な試合: チェックボックスで登録可能
  - 未申込で自由登録不可の試合: 「定員到達」等の表示

#### 5.1.2 練習一覧画面（PracticeList.jsx）

- 各試合に抽選ステータス（抽選済み/未抽選）を表示する
- 管理者向け: 再抽選ボタン、手動編集リンクを追加する

#### 5.1.3 ホーム画面

- 次回参加予定の情報に加え、抽選結果サマリーを表示する
- 未応答の繰り上げ連絡がある場合、目立つ形で表示する

#### 5.1.4 ヘッダー/ナビゲーション

- 通知ベル（未読バッジ付き）を追加する

### 5.2 新規画面

#### 5.2.1 抽選結果確認画面

- 月別に全セッション・全試合の抽選結果を一覧表示する
- 自分の各試合のステータスを確認できる
- 各試合の当選者一覧、キャンセル待ち一覧（番号付き）を閲覧できる

#### 5.2.2 キャンセル待ち状況確認画面

- 自分がキャンセル待ちになっている試合の一覧を表示する
- 各試合のキャンセル待ち番号、現在の状況（何番まで進んでいるか等）を表示する

#### 5.2.3 繰り上げ参加承認画面

- 繰り上げ通知を受けた際に遷移する画面
- 対象の練習日・試合の情報を表示する
- 応答期限を表示する
- 「参加する」「参加しない」ボタンを配置する

#### 5.2.4 通知一覧画面

- 全通知を時系列で一覧表示する
- 未読/既読のステータス管理
- 通知をタップすると関連画面（繰り上げ承認画面等）に遷移する

#### 5.2.5 通知設定画面

- Web Push通知の許可/拒否を設定する
- ブラウザの通知許可を求めるフローを提供する

#### 5.2.6 管理者用: 抽選管理画面

- セッション単位で抽選状態を管理する
- 再抽選の実行ボタン
- 各試合の参加者・キャンセル待ちの手動編集機能
  - 参加者の追加・削除
  - キャンセル待ち順番の変更
  - ステータスの手動変更

## 6. データ設計（概要）

### 6.1 既存テーブルの変更

#### practice_participants テーブル

新規カラムの追加:

| カラム名 | 型 | 説明 |
|---------|------|------|
| status | ENUM('PENDING','WON','WAITLISTED','OFFERED','DECLINED','CANCELLED') | 参加ステータス |
| waitlist_number | INT | キャンセル待ち番号（WAITLISTED時のみ） |
| lottery_id | BIGINT | 紐づく抽選実行ID |
| offered_at | DATETIME | 繰り上げ通知日時 |
| offer_deadline | DATETIME | 繰り上げ応答期限 |
| responded_at | DATETIME | 繰り上げ応答日時 |

- 既存データのマイグレーション: 既存レコードは `status = 'WON'` として移行する

### 6.2 新規テーブル

#### lottery_executions（抽選実行履歴）

| カラム名 | 型 | 説明 |
|---------|------|------|
| id | BIGINT PK | 抽選実行ID |
| target_year | INT | 対象年 |
| target_month | INT | 対象月 |
| execution_type | ENUM('AUTO','MANUAL_RELOTTERY') | 実行種別 |
| executed_by | BIGINT | 実行者（自動の場合はNULL） |
| executed_at | DATETIME | 実行日時 |
| status | ENUM('SUCCESS','FAILED','PARTIAL') | 実行結果 |
| details | JSON/TEXT | 処理詳細（各試合の結果サマリー等） |

#### notifications（アプリ内通知）

| カラム名 | 型 | 説明 |
|---------|------|------|

## 9. エッジケース・ルール一覧

### 9.1 抽選関連

| # | ケース | ルール |
|---|--------|--------|
| 1 | 定員未設定のセッション | 抽選なし。全員当選 |
| 2 | 参加希望者が定員以下の試合 | 抽選なし。全員当選 |
| 3 | 同セッション内で試合1に落選し、試合3も定員超過 | 試合1で取れていない（`todayTaken`=0のまま）ため、試合3でもルール1の最優先候補になる。空きが無い場合は他の候補との重み付き抽選（ルール2）で当落が決まる |
| 4 | 同セッション内で試合1に落選し、試合3は定員以下 | 試合3は全員当選 |
| 4b | 同セッション内で試合1に落選し、試合3は定員超過だが空き枠あり | 試合3は当選しやすい（`todayTaken`が最小のためルール1の候補になる） |
| 5 | 直近30日の取得数（`recentTaken`）が少ない人 | 他の候補との重み付き抽選（ルール2）で当選しやすくなる（当選を保証するものではない） |
| 6 | 管理者指定優先選手が定員を超える場合 | 優先選手同士に同じルール1・ルール2を適用して抽選。一般申込者は全員落選 |
| 7 | 月を跨いだ取得実績 | リセットしない。そのセッション日付から遡る直近30日のローリングで算出し続ける（暦月の境界に依存しない） |

### 9.2 キャンセル・繰り上げ関連

| # | ケース | ルール |
|---|--------|--------|
| 8 | 当選者が前日までにキャンセル | 繰り上げフロー開始。キャンセル待ち1番に通知 |
| 9 | 当選者が当日キャンセル | アプリでキャンセル処理のみ。繰り上げフロー不開始。直接連絡で調整 |
| 10 | 繰り上げ通知の応答期限 | min(通知から24時間, 練習日前日23:59)の早い方 |
| 11 | 繰り上げ辞退 or 無応答 | 自動DECLINED。次のキャンセル待ち番号に通知 |
| 12 | 全キャンセル待ち者が辞退/無応答 | 枠は空きのまま。当該試合は自由登録可能に |
| 13 | 複数の当選者が同時キャンセル | 各キャンセルに対し、順番にキャンセル待ちの次の人に通知 |
| 14 | 当日にキャンセル待ちの制約 | 当日は番号順制約を解除。誰でも空き枠に登録可能 |

### 9.3 抽選後の登録関連

| # | ケース | ルール |
|---|--------|--------|
| 15 | 定員未到達＋キャンセル待ちなし | 自由登録可 |
| 16 | 定員未到達＋キャンセル待ちあり | 自由登録不可（キャンセル待ちが優先） |
| 17 | 定員到達 | 自由登録不可 |
| 18 | 締め切り後に追加されたセッション | 抽選なし。先着順（定員まで自由登録） |

### 9.4 管理者操作関連

| # | ケース | ルール |
|---|--------|--------|
| 19 | 再抽選実行 | セッション内全参加者を対象に再抽選（繰り上げ承諾者は維持） |
| 20 | 管理者による手動追加 | 定員超過も許可 |

## 10. 非機能要件

### 10.1 信頼性

- 抽選処理はトランザクション内で実行し、途中失敗時はロールバックする
- 抽選実行履歴を保持し、監査可能とする
- サーバー障害時の抽選リトライ機構を設ける

### 10.2 パフォーマンス

- 月次一括抽選はバックグラウンドで実行し、ユーザー操作をブロックしない
- 繰り上げ期限チェックのスケジューラは軽量に実装する

### 10.3 セキュリティ

- 抽選結果の改ざん防止（実行履歴の記録）
- 管理者操作のログ記録
- Web Push通知のVAPIDキーの安全な管理
