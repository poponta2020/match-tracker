# Google OAuth 本番公開移行計画書

## 1. 概要

現在、Google Calendar同期機能のOAuth同意画面は**テストモード**で運用しており、以下の制約がある:
- 手動登録したテストユーザー（最大100人）のみ利用可能
- 管理者がGoogle Cloud Consoleでメールアドレスを1件ずつ追加する必要がある

本計画では、**本番環境に公開**し、Googleアカウントを持つ全ユーザーが自由にカレンダー同期機能を利用できるようにする。

## 2. 現状

| 項目 | 現在の値 |
|------|----------|
| アプリ名 | 競技かるた記録 |
| OAuth同意画面ステータス | テスト（外部） |
| 使用スコープ | `https://www.googleapis.com/auth/calendar.events` |
| スコープ分類 | **センシティブスコープ**（Google審査必要） |
| フロントエンドURL | Vercel（要確認） |
| バックエンドURL | https://match-tracker-1zdd.onrender.com |
| GitHubリポジトリ | https://github.com/poponta2020/match-tracker |

## 3. 審査に必要な準備物

### 3.1 プライバシーポリシーページ（必須）

アプリ内に `/privacy` ページを新設し、以下の内容を記載する。

**記載内容:**
- アプリ名・運営者情報
- 取得する情報
  - Googleアカウントのメールアドレス
  - Googleカレンダーへの予定の読み書き権限
- 利用目的
  - 練習予定をユーザーのGoogleカレンダーに同期するためのみに使用
- データの保存
  - アクセストークンはサーバーに保存しない（同期処理中のみ一時的に使用）
  - 同期マッピング情報（どの練習がどのカレンダーイベントに対応するか）のみDBに保存
- 第三者提供
  - ユーザーデータを第三者に提供・販売しない
- データの削除
  - ユーザーはいつでもGoogle側でアプリのアクセス権を取り消し可能
  - アカウント削除時にマッピング情報も削除される
- 問い合わせ先

### 3.2 アプリのホームページURL（必須）

フロントエンドのデプロイ先URL（Vercel）を設定する。

### 3.3 スコープの利用目的説明（審査時に記入）

> 本アプリは競技かるたの練習管理ツールです。ユーザーが参加予定の練習日程をGoogleカレンダーに同期する機能を提供しています。`calendar.events`スコープは、ユーザーが手動で「同期」ボタンを押した際に、練習予定の作成・更新・削除を行うために必要です。アクセストークンはサーバーに保存せず、同期処理中のみ一時的に使用します。

## 4. 実施手順

### Phase 1: プライバシーポリシーページ作成

| # | タスク | 詳細 |
|---|--------|------|
| 1 | フロントエンドにページ作成 | `/privacy` ルートにプライバシーポリシーページを追加 |
| 2 | ログイン不要でアクセス可能に | PrivateRouteの外に配置（Googleの審査担当者がアクセスできる必要がある） |
| 3 | デプロイ | Vercelにデプロイして公開URLを確認 |

### Phase 2: Google Cloud Console設定

| # | タスク | 詳細 |
|---|--------|------|
| 1 | OAuth同意画面の情報を更新 | アプリ名、サポートメール、ロゴ（任意）を確認・更新 |
| 2 | プライバシーポリシーURLを設定 | Phase 1で作成したURLを入力 |
| 3 | アプリのホームページURLを設定 | フロントエンドのデプロイ先URLを入力 |
| 4 | 「アプリを公開」をクリック | 審査リクエストを送信 |

### Phase 3: Google審査対応

| # | タスク | 詳細 |
|---|--------|------|
| 1 | 審査待ち | 通常数日〜数週間 |
| 2 | 追加質問への回答 | Googleから質問が来た場合に対応 |
| 3 | 動画デモの提出（求められた場合） | スコープの使い方を示す短い動画を録画 |

### Phase 4: 公開後の確認

| # | タスク | 詳細 |
|---|--------|------|
| 1 | テストユーザー以外でのログイン確認 | 新規Googleアカウントでカレンダー同期をテスト |
| 2 | 同意画面の表示確認 | アプリ名・スコープの説明が正しく表示されることを確認 |
| 3 | テストユーザーリストのクリーンアップ | 不要になったテストユーザーを削除（任意） |

## 5. 審査でよく指摘される事項

| 指摘内容 | 対策 |
|----------|------|
| プライバシーポリシーにアプリ名が記載されていない | アプリ名「競技かるた記録」を明記する |
| データ削除の方法が記載されていない | 削除手順を明記する |
| スコープの使用が過剰 | `calendar.events`のみ使用（必要最小限） |
| ホームページURLがアクセスできない | デプロイ済みURLであることを確認 |
| OAuth同意画面のアプリ名とポリシーのアプリ名が不一致 | 統一する |

## 6. 審査不要の代替案（参考）

`calendar.events`はセンシティブスコープのため審査が必須。もし審査を避けたい場合:

- **スコープを`calendar.events.owned`に変更** — 自分が作成したイベントのみ操作可能なスコープ。ただしこれもセンシティブ扱いのため審査は必要。
- **テストモードのまま運用** — 100人以下なら現実的。メールアドレスの登録はCSVなどで一括管理し、Google Cloud ConsoleのAPIで自動化することも可能（ただしAPIでのテストユーザー追加は非公式）。
