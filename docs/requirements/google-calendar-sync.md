# Google Calendar連携機能 要件定義書

## 1. 概要

### 1.1 目的
アプリに登録された練習日の予定を、ユーザーの個人Google Calendarに手動で同期する機能を実装する。

### 1.2 背景
- 練習日の予定がアプリ内にしか存在せず、普段使いのカレンダーアプリで確認できない
- 練習の予定を手動でGoogle Calendarに転記する手間が発生している
- アプリからワンタップで自分の参加予定をカレンダーに反映したい

### 1.3 用語定義
| 用語 | 説明 |
|------|------|
| 同期 | アプリ内の練習日データをGoogle Calendarのイベントとして作成・更新・削除すること |
| アクセストークン | Google OAuth2で取得する一時的な認証トークン（約1時間有効） |
| GIS | Google Identity Services。フロントエンドでのGoogle認証用ライブラリ |
| プライマリカレンダー | Googleアカウントのデフォルト（メイン）カレンダー |

---

## 2. 現状分析

### 2.1 既存システムの状態

#### データベース
- `practice_sessions`: 練習日（日付・時間帯・会場・試合数）を管理
- `practice_participants`: 練習日への参加者を管理（session_id + player_id）
- `venues`: 会場マスタ（会場名・デフォルト試合数）
- `venue_match_schedules`: 会場ごとの試合時間帯

#### 外部連携
- 伝助（Densuke）からのスクレイピングによる練習日・参加者インポート機能が既存
- Google等の外部OAuth連携は未実装

#### 認証
- 簡易的なロールベース認証（SUPER_ADMIN / ADMIN / PLAYER）
- JWTは未実装（localStorageにダミートークンを保存）
- Google OAuth2とは独立して動作可能

### 2.2 課題
- 練習予定の確認がアプリ内に限定されている
- ユーザーが普段利用するカレンダーアプリとの連携手段がない

---

## 3. 要件

### 3.1 機能要件

#### FR-1: Google OAuth2 認証
| 項目 | 内容 |
|------|------|
| 概要 | ユーザーがGoogleアカウントでログインし、Calendar APIへのアクセスを許可する |
| 認証場所 | フロントエンド（ブラウザ上） |
| OAuthスコープ | `https://www.googleapis.com/auth/calendar.events` |
| トークン管理 | フロントエンドのメモリ上で保持（localStorageには保存しない） |
| トークン保存 | バックエンドにはトークンを永続化しない（毎回フロントから送信） |

**認証フロー:**
```
1. ユーザーが「Google Calendarと同期」ボタンを押下
2. Google OAuth2 同意画面がポップアップ表示
3. ユーザーがCalendar書き込み権限を許可
4. アクセストークンをフロントエンドが受け取る
5. アクセストークンをバックエンドAPIに送信して同期実行
```

#### FR-2: 練習日の同期
| 項目 | 内容 |
|------|------|
| 同期方向 | アプリ → Google Calendar（一方向） |
| 同期対象 | ログインユーザーが参加予定の練習日（PracticeParticipantに登録あり） |
| 同期先 | ユーザーのプライマリ（メイン）カレンダー |
| 同期タイミング | ユーザーが手動で「同期」ボタンを押下したとき |
| 冪等性 | 同じ練習日を複数回同期しても重複イベントを作成しない |

**同期処理フロー:**
```
1. フロントエンドがGoogleアクセストークンを取得（未認証なら認証フロー開始）
2. バックエンドAPI POST /api/google-calendar/sync を呼び出し
   - リクエスト: アクセストークン + ユーザーID
3. バックエンドが該当ユーザーの参加予定練習日を取得
4. 練習日ごとに以下を判定:
   a. google_calendar_eventsに未登録 → Google Calendar APIでイベント作成
   b. 登録済みかつ練習日に変更あり → イベント更新
   c. 参加取消済み（Participantから削除済み） → イベント削除
5. 同期結果をフロントエンドに返却
```

**作成されるイベントの内容:**
| フィールド | 値 |
|------------|-----|
| タイトル | `わすら@{会場名}` |
| 開始日時 | `PracticeSession.startTime` |
| 終了日時 | `PracticeSession.endTime` |
| 場所 | `Venue.name` |
| 説明 | `{試合数}試合` |

#### FR-3: 同期状態の確認
| 項目 | 内容 |
|------|------|
| 概要 | ユーザーが最終同期日時と同期済み件数を確認できる |
| 表示場所 | 同期ボタンの近くにテキスト表示 |
| 表示内容 | 最終同期日時、同期済みイベント数 |

#### FR-4: 同期結果の表示
| 項目 | 内容 |
|------|------|
| 概要 | 同期実行後に処理結果をユーザーに表示する |
| 表示形式 | トースト通知またはモーダル |
| 表示内容 | 追加件数・更新件数・削除件数（例:「3件追加、1件更新、1件削除」） |

### 3.2 非機能要件

| 項目 | 内容 |
|------|------|
| セキュリティ | アクセストークンはメモリ上のみで管理し、localStorage/DBに永続化しない |
| パフォーマンス | 同期処理は非同期で実行し、UIをブロックしない |
| エラーハンドリング | トークン期限切れ時は再認証フローを自動起動 |
| API利用制限 | Google Calendar APIの日次クォータ（1,000,000クエリ/日）を超えない設計 |
| 可用性 | Google API障害時はエラーメッセージを表示し、アプリ本体の機能に影響しない |

## 4. データベース設計

### 4.1 新規テーブル

#### `google_calendar_events`
| カラム | 型 | 制約 | 説明 |
|--------|----|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 主キー |
| player_id | BIGINT | FK → players.id, NOT NULL | ユーザーID |
| session_id | BIGINT | FK → practice_sessions.id, NOT NULL | 練習日ID |
| google_event_id | VARCHAR(255) | NOT NULL | Google Calendar側のイベントID |
| last_synced_at | TIMESTAMP | NOT NULL | 最終同期日時 |
| created_at | TIMESTAMP | NOT NULL | レコード作成日時 |
| updated_at | TIMESTAMP | NOT NULL | レコード更新日時 |

**制約:**
- ユニーク制約: `(player_id, session_id)`
- 外部キー: `player_id` → `players.id`, `session_id` → `practice_sessions.id`

**インデックス:**
- `idx_gcal_events_player_id` ON `(player_id)`
- `idx_gcal_events_session_id` ON `(session_id)`

## 5. API設計

### 5.1 新規エンドポイント

#### `POST /api/google-calendar/sync`
同期を実行する。

**リクエスト:**
```json
{
  "accessToken": "ya29.xxxx..."
}
```
- ユーザーIDはログインセッションから取得

**レスポンス（成功）:**
```json
{
  "created": 3,
  "updated": 1,
  "deleted": 1,
  "lastSyncedAt": "2026-03-16T15:30:00"
}
```

**エラーレスポンス:**
| ステータス | 条件 |
|------------|------|
| 401 | アクセストークンが無効または期限切れ |
| 403 | Calendar APIへの権限が不足 |
| 500 | Google API呼び出し失敗 |

#### `GET /api/google-calendar/status`
同期状態を取得する。

**レスポンス:**
```json
{
  "syncedEventCount": 5,
  "lastSyncedAt": "2026-03-16T15:30:00"
}
```

## 6. UI設計

### 6.1 同期ボタンの配置
- **ホーム画面**に「Google Calendarと同期」ボタンを設置
- ボタン横に最終同期日時を表示

### 6.2 操作フロー
```
[ホーム画面]
  └→ 「Google Calendarと同期」ボタン押下
      ├→ 未認証の場合 → Googleログインポップアップ → 認証完了後に同期実行
      └→ 認証済みの場合 → 直接同期実行
          └→ 同期結果を通知表示（「3件追加、1件更新、1件削除」）
```

### 6.3 状態表示
| 状態 | 表示 |
|------|------|
| 未同期 | 「まだ同期していません」 |
| 同期済み | 「最終同期: 2026/03/16 15:30（5件）」 |
| 同期中 | ローディングスピナー + 「同期中...」 |
| エラー | エラーメッセージ + リトライボタン |

## 8. スコープ外

以下は本機能のスコープに含めない。

| 項目 | 理由 |
|------|------|
| Google Calendar → アプリの同期（双方向） | 競合解決の複雑さ、要件過多 |
| 対戦組み合わせ情報のカレンダー反映 | 初回リリースではシンプルに保つ |
| リフレッシュトークンのDB保存 | セキュリティリスクを低く保つため毎回認証する方式を採用 |
| 自動同期（Webhook / 定期ポーリング） | 手動同期で十分な利用頻度と判断 |
| 共有カレンダーへの管理者一括登録 | 個人カレンダーへの同期のみとする |
| 専用カレンダーの自動作成 | メインカレンダーへの追加のみとする |

## 9. リスク・注意点

| リスク | 影響 | 対策 |
|--------|------|------|
| OAuth同意画面の審査 | 本番公開時にGoogleの審査が必要（数週間かかる場合あり） | テストモード（100ユーザーまで審査不要）で先行リリース |
| アクセストークンの有効期限 | 約1時間で失効、同期途中で切れる可能性 | 同期開始前にトークン有効性を確認、失効時は再認証を促す |
| Google API障害 | 同期が一時的に利用不可 | エラーメッセージを表示し、アプリ本体の機能には影響させない |
| 現行認証基盤との整合性 | アプリ認証（簡易）とGoogle OAuth2が別系統で動作 | 将来的な認証統合は別途検討。今回はGoogle OAuthをカレンダー同期専用として独立運用 |
| トークンを毎回取得する方式のUX | 同期のたびにGoogleログインが必要になる場合がある | GISのサイレント認証（同一セッション内ならポップアップ不要）を活用 |
