# 対戦組み合わせ ロック表示まわりの改善（pairing-lock-display-fixes）要件定義書

> 改修（delta）。対象は `/pairings`（`PairingGenerator.jsx`）のみ。既存のロック仕様は
> [[pairing-manual-lock]]（`docs/features/pairing-manual-lock/requirements.md`）と
> `pairing-lock-explicit-save-and-help`（明示保存化）が正典。本書はその表示・文言まわりの
> 差分のみを扱い、ロックの機能ロジック（保護判定・保存・二重ブッキング等）は一切変更しない。

## 1. 概要

`/pairings`（組み合わせ作成画面）のロック表示・ボタン文言に関する3つの小さな UX 改善。バックエンド・DB・API・ロックの機能ロジックには手を入れない、純フロントの表示変更。

- **変更A（ロック通知の削除）**: 対戦編集／再シャッフル実行時に出る「（結果入力済みの N 組・）手動ロックの M 組はロックされています。」通知は不要。ロックされているかどうかは編集時に各組の表示で分かれば十分なため、この通知自体を削除する。
- **変更B（ロック済み組のアイコン重複解消）**: 手動ロック済みの組は現在「ロック」バッジ（鍵アイコン＋「ロック」）と「解除」ボタン（開いた鍵アイコン＋「解除」）が横に並んで冗長。編集画面では「解除」だけを残す。
- **変更C（ボタン＋確認ダイアログの文言変更）**: 既存の組み合わせがある編集状態で表示される「全削除」ボタンは意味が伝わりにくいため「対戦をリセット」に変更し、押下時の確認ダイアログも「削除しますか？」から「対戦組み合わせをリセットしますか？」の文言に合わせる。

## 2. 変更の動機と内容

### 変更A — ロック通知を削除する
- **現状**: `runAutoMatch`（対戦編集・再シャッフルの共通処理）が、応答にロック組（結果入力済み or 手動ロック）が含まれると `notice` state に「（結果入力済みの N 組・）手動ロックの M 組はロックされています。」を設定する（`PairingGenerator.jsx` の該当ブロック）。この通知は試合番号タブ切替でクリアされず他タブにも残るうえ、そもそもロック状態は各組の行表示で分かる。
- **なぜ問題**: ロック中かどうかは編集操作のとき各組の表示（結果入力済／解除ボタン）で分かれば十分で、通知は冗長。実在しない試合番号タブにまで残って誤認を招く。
- **変更内容**: この「…はロックされています。」通知を**設定するコード（該当ブロック）ごと削除する**。ロック組の有無にかかわらず当該通知は一切表示しない。
- **注**: 同じ `runAutoMatch` にある「参加者数が異なるため最新の組み合わせ対象で生成しました。」通知は別目的（当該アクションの警告）のため対象外・不変。

### 変更B — ロック済み組に「ロック」と「解除」が二重表示される
- **現状**: 手動ロック済み（`locked && !hasResult` かつキャンセルなし）の組は、行の右側に静的な「ロック」バッジ（`Lock` アイコン＋「ロック」）と、操作用の「解除」ボタン（`Unlock` アイコン＋「解除」）の両方が並ぶ。
- **なぜ問題**: 鍵アイコン＋文言が2つ並び冗長・紛らわしい。開いた鍵（`Unlock`）＋「解除」だけでロック中であることも解除操作も伝わる。
- **変更内容**: 編集モード（＝「解除」ボタンが出る状態）では静的「ロック」バッジを出さず、「解除」ボタンのみにする。閲覧モード／読み取り専用モードは「解除」ボタンが出ない（現行仕様）ため、そこでは「ロック」バッジを従来どおり残す（後述 §3 の変わらないもの）。

### 変更C — 「全削除」ボタン＋確認ダイアログの文言変更
- **現状**: 既存組み合わせを持つ試合の編集状態で「全削除」ボタン（`handleDeleteExisting`）を表示。押下すると確認ダイアログ（ロック有無で2分岐）「…組み合わせを削除しますか？」を出し、OK でロック済み以外を削除する。
- **なぜ問題**: 「全削除」だと何が消えるか伝わりにくい。ボタンとダイアログの語（削除 / リセット）も揃っていないと混乱する。
- **変更内容**: ボタン文言を「対戦をリセット」に変更し、確認ダイアログも「対戦組み合わせをリセットしますか？」に合わせる（挙動は不変：ロック済み以外の組を削除する現行処理のまま）。
  - ロックなし分岐: 「既存の対戦組み合わせをリセットしますか？」
  - ロックあり分岐: 「ロック済みの N 組を除く対戦組み合わせをリセットしますか？」（保持される旨の補足行はそのまま）

## 3. 変更後の挙動と、変わらないもの（回帰の明示）

### 変更後の挙動
- **A**: 対戦編集／再シャッフルを実行しても「…はロックされています。」通知は出ない（コードごと削除）。
- **B（編集モード）**: 手動ロック済み組の行は「解除」ボタン（`Unlock` アイコン＋「解除」）のみを表示し、「ロック」バッジは表示しない。
- **C**: ボタン文言が「対戦をリセット」、確認ダイアログが「（ロック済みの N 組を除く／既存の）対戦組み合わせをリセットしますか？」になる。

### 変わらないもの（維持すべき既存挙動＝回帰ACの対象）
- **B（閲覧／読み取り専用モード）**: 手動ロック済み組には「ロック」バッジを従来どおり表示する（このモードでは「解除」ボタンが無いため、バッジがロック中の唯一の視覚指標）。**削ってはならない。**
- **結果入力済み組**の表示（「結果入力済」バッジ＋ ADMIN+ 「リセット」ボタン）は一切変更しない。
- ロックの**機能ロジック**（保護判定 `hasResult OR locked`・`buildSaveRequests`・`computeLockedPairsInput`・二重ブッキング・自動組み合わせ／回戦削除からの保護・確定して保存での永続化）は不変。
- **C** の削除処理そのもの（`handleDeleteExisting`：ロック済み以外を削除）は不変（文言のみ変更）。
- 「参加者数が異なるため最新の組み合わせ対象で生成しました。」通知（変更A対象外）は不変。
- 「対戦をリセット」への改名により、組ごとの結果リセットボタン「リセット」と語が一部重なるが、これはユーザー指定の文言のため許容する。
- バックエンド・DB・API・`/pairings/summary`（`PairingSummary`）は対象外。

## 4. Acceptance Criteria

| ID | 条件（客観的に判定できる文） | 検証手段 |
|----|------|------|
| AC-1 | 対戦編集／再シャッフルを実行したとき、ロック組の有無にかかわらず「…はロックされています。」通知が一切表示されない（当該通知を設定するコードを削除） | verify |
| AC-2 | 編集モードでは、手動ロック済み組（`locked && !hasResult`・キャンセルなし）の行に「ロック」バッジを表示せず、「解除」ボタン（`Unlock`＋「解除」）のみを表示する（純関数 `shouldShowManualLockBadge` が編集モードで false を返す） | auto-test |
| AC-3 | 閲覧モード／読み取り専用モードでは、手動ロック済み組に「ロック」バッジを従来どおり表示する（`shouldShowManualLockBadge` が該当条件で true を返す） | auto-test |
| AC-4 | 結果入力済み組の表示（「結果入力済」バッジ＋「リセット」ボタン）は従来どおりで、`shouldShowManualLockBadge` は結果入力済み組に対して false を返す（ロックバッジは出さない） | auto-test |
| AC-5 | 既存組み合わせを持つ試合の編集状態で表示されるボタンの文言が「対戦をリセット」であり、押下時の確認ダイアログが「（ロック済みの N 組を除く／既存の）対戦組み合わせをリセットしますか？」の文言である。削除処理そのものの挙動は不変 | verify |
| AC-6 | 既存の pairing 関連ユニットテスト・lint がすべて green（回帰なし） | auto-test |

- 検証手段内訳: **auto-test 3件 / verify 2件 / manual 0件**（AC-6 の回帰も含めると auto-test 4件）
- AC-1・AC-5 が `verify` なのは、いずれもコンポーネントの表示・文言（純関数化しても意味の薄い定数・副作用）であり、当画面に API モック込みのフルマウント回帰ハーネスが無く既存の純関数ユニットパターンに乗らないため。実装後に `/verify` または手動で確認する。

## 5. Non-goals（今回やらないこと）

- ロックの機能ロジック・保護判定・保存・二重ブッキング・API・DB・entity の変更（表示と文言のみ）。
- 「参加者数が異なる…」通知の挙動変更（変更A対象外・不変）。
- 組ごと「リセット」ボタンとの語重複の解消（ユーザー指定文言を優先）。
- `PairingSummary`（`/pairings/summary`）その他画面への波及。

## 6. 技術的制約・契約

- **純フロント変更**（`karuta-tracker-ui/src/pages/pairings/` 内）。バックエンド・DB・API は不変＝FE⇔BE 契約に影響なし。
- コードベースの規律に従い、変更Bの表示判定は純関数として `pairingLockLogic.js` に `shouldShowManualLockBadge` を追加し、本番 JSX と `PairingGenerator.integration.test.jsx` の `ViewRow` 検証ハーネスの双方が同じ関数を import する（条件式のコピー禁止＝退行を確実に検知）。
- ユーザー向け文言は日本語。
- ドキュメント更新: `docs/SCREEN_LIST.md` の `/pairings`（#19）行のロック表示・「全削除」記述を実装と同じコミットで in-place 更新する。

## 7. 設計判断の根拠

| 判断 | 理由 |
|------|------|
| 変更A は通知を「クリア」ではなく「削除」 | ロック状態は各組の行表示（結果入力済／解除ボタン・閲覧時のロックバッジ）で常に分かるため通知は冗長。タブ間リークもコードごと消せば根絶でき、最小・確実。 |
| 変更B は「解除ボタンが出る時はバッジを出さない」= 閲覧/読み取り専用のみバッジ | 編集モードでは `locked && !hasResult` の組で `canShowUnlock` が常に true になるため、バッジ非表示／ボタン表示が完全に排他となり、閲覧モードのロック指標（バッジ）を失わずに冗長表示だけ解消できる。 |
| 変更B の判定を純関数化 | 本プロジェクトの pairing 表示ロジックは全て純関数化して本番・テストで共有する規律に合わせ、退行検知を担保するため。 |
| 変更C はボタン＋確認ダイアログを揃えて「リセット」文言に | ラベルの意味明確化に加え、確認ダイアログの語も揃えて混乱を避ける。削除処理の挙動は変えず影響範囲を最小化。 |
