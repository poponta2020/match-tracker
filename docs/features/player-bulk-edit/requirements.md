# 選手情報の一括編集 (player-bulk-edit) 要件定義書

## 1. 概要

### 目的
管理者が、選手一覧から複数の選手を選択し、専用画面で **性別・級・所属かるた会・所属練習会** をまとめて編集できるようにする。

### 背景・動機
新入生が大量に入ってくる時期に、一人ずつ選手編集画面を開いて E級 に設定する作業が非常に手間。1画面で複数選手をまとめて設定したい。特に「全員を E級 に」「全員に 北大 / わすら を追加」が頻出操作。

---

## 2. ユーザーストーリー

- **対象ユーザー:** 管理者（ADMIN / SUPER_ADMIN）
- **目的:** 複数の新入生選手の属性を一度の操作でまとめて設定し、登録作業の手間を減らす
- **利用シナリオ:**
  1. 新入生が多数登録された後、選手一覧を開く
  2. 対象の新入生にチェックを入れて「一括編集」へ進む
  3. 一括編集画面で「全員を E級 に」「全員に 北大 を追加」などのボタンを押す
  4. 必要に応じて行（選手）ごとに個別調整する
  5. 保存して確認ダイアログでまとめて反映する

---

## 3. 機能要件

### 3.1 画面仕様

#### A. 選手一覧（PlayerList）への選択UI・団体フィルタ追加
- 各選手行に **選択チェックボックス** を追加する
- 「全選択 / 全解除」操作を用意する（現在表示中＝フィルタ後の選手が対象）
- 1人以上選択すると **「一括編集」ボタン**（＋選択人数）が表示され、押すと一括編集画面へ遷移する
- **団体フィルタの追加:** 一覧を団体所属で絞り込めるセレクトを追加する
  - 選択肢: **すべて / 各団体（団体別） / 無所属（どの団体にも所属していない）**
  - 判定は `PlayerDto.organizationIds` を用いる（空配列＝無所属）
  - **初期表示:** ADMIN は自分の管轄団体（`adminOrganizationId`）、SUPER_ADMIN は「すべて」。ADMIN は「無所属」に切り替えて未所属の新入生を素早く探せる
  - 名前検索（既存）と併用可能
- **選択スコープ:** ADMIN / SUPER_ADMIN ともに表示中の選手を制限なく選択・編集できる（団体フィルタは絞り込みの利便性であり、編集可否の制限ではない）
- **既存の団体セレクト（招待リンク用）との関係:** 現状 PlayerList にある団体セレクト（`selectedOrgId`、招待トークン生成用）と本フィルタを **1つの団体セレクトに統合** する。具体的な団体を選んでいるときのみ招待ボタンを有効化し、「すべて / 無所属」選択時は招待ボタンを無効化する（招待は具体的な団体が必須のため）

#### B. 一括編集画面（新規 `/players/bulk-edit`）
選択した選手を **行で並べた編集テーブル**。

**上部：全員一括設定エリア**
- 性別：一括設定セレクト（男性／女性／その他）→「全員に適用」
- 級：一括設定セレクト（E〜A級）＋ **「全員を E級 に」ショートカットボタン**
- 所属かるた会：一括設定テキスト入力 →「全員に適用」
- 所属練習会：**「全員に 北大 を追加」「全員に わすら を追加」** ボタン（追加のみ）
- 一括設定ボタンは、テーブル全行の該当項目を上書き／追加する（保存はまだ行わない）

**テーブル本体：1行＝1選手**

| 列 | 内容 | 編集 |
|----|------|------|
| 選手名 | 氏名（＋読み等） | 読み取り専用 |
| 性別 | セレクト | 行ごと編集可。初期値＝現在値 |
| 級 | セレクト（E〜A級）。選択で段位を自動連動 | 行ごと編集可。初期値＝現在値 |
| 段位 | セレクト。A級のときのみ手動指定（四段〜八段）、それ以外は級に連動し非活性 | 単体編集と同一挙動 |
| 所属かるた会 | テキスト入力 | 行ごと編集可。初期値＝現在値 |
| 所属練習会 | 現在の所属団体を表示＋「北大を追加」「わすらを追加」（追加分は取り消し可） | 行ごと追加可（追加のみ） |

- **保存ボタン:** 押すと **確認ダイアログ**（「N人に以下の変更を適用します」＋変更概要）を表示。OKで一括反映
- 保存成功後は選手一覧へ戻り、最新状態を再取得する
- キャンセルで一覧へ戻る

### 3.2 ビジネスルール

- **未変更項目の扱い:** 各行・各項目の初期値は当該選手の現在値。触らなければ現在値のまま保存される（実質変更なし）。一括設定ボタンを使うか行ごとに変更した項目だけが実際に変わる
- **級と段位の連動:** 単体編集と同一。E級→無段、D級→初段、C級→弐段、B級→参段、A級→段位を手動指定（四段〜八段）。級を変えると段位は自動更新（A級以外は段位入力を非活性）
- **所属練習会は追加のみ:** 「北大を追加」「わすらを追加」は既存の所属を維持したまま団体を追加する（idempotent。既に所属していれば二重登録しない）。**既存の所属を削除する操作は提供しない**。行ごとの「取り消し」は、まだ保存していない“追加予定”の取り消しのみ（既存所属は消えない）
- **性別は必須:** 空にはできない（初期値が常に入っているため通常問題にならない）
- **権限:** ADMIN / SUPER_ADMIN のみ利用可。**ADMIN も全選手を編集可能**（「管理者は悪用しない」前提のトラストベース）。バックエンドは `@RequireRole(ADMIN)` のみを要求し、対象選手の団体スコープ検証は行わない。団体フィルタは一覧の絞り込み利便性のためのものであり、編集可否を制限しない
- **PlayerProfile（履歴テーブル）:** 単体編集と同様、本機能でも `player_profiles`（temporal な grade/dan/karutaClub 履歴）は更新しない（`players` テーブルの列のみ更新）。※ `player_profiles` は現状どの主要フローからも呼ばれていない休眠テーブルであり、級の時系列履歴を残す機能は本件のスコープ外

### 3.3 エラーケース・境界条件
- 選択0人で一括編集画面に来た場合（直接URL/リロード等）：一覧へリダイレクト
- 保存はトランザクションで all-or-nothing。一部失敗時は全件ロールバックしエラー表示
- 北大／わすら団体がDBに存在しない環境では該当ボタンを非活性または非表示（団体名で解決）

## 4. 技術設計

### 4.1 API設計

**新規エンドポイント（一括更新）**
```
PUT /api/players/bulk
@RequireRole(ADMIN)   // ADMIN 以上。対象選手の団体スコープ検証は行わない
```
リクエスト:
```json
{
  "updates": [
    {
      "playerId": 12,
      "gender": "男性",
      "kyuRank": "E級",
      "danRank": "無段",
      "karutaClub": "北海道大学かるた会",
      "addOrganizationIds": [3]
    }
  ]
}
```
- `gender/kyuRank/danRank/karutaClub` … `players` 列を更新（単体更新と同じ値域）
- `addOrganizationIds` … 既存に無いものだけ `player_organizations` に追加（追加のみ）
- レスポンス: 更新後の `PlayerDto` リスト（または 200 + 件数）

**既存エンドポイント（再利用）**
- `GET /api/players` … 一覧取得（`PlayerDto` に `organizationIds`/`adminOrganizationId` あり）
- `GET /api/organizations` … 北大/わすらの団体ID解決

> 代替案として「フロントから単体API（`PUT /players/{id}` + `PUT /organizations/players/{id}`）をループ呼び出し」も可能だが、件数分のリクエスト・部分失敗時の不整合・スコープ検証の分散を避けるため、**専用バルクエンドポイント（トランザクション）** を採用する。

### 4.2 DB設計
- **スキーマ変更なし（マイグレーション不要）**
  - 性別・級・段位・かるた会 … `players` の既存列
  - 所属練習会 … `player_organizations`（既存。`(player_id, organization_id)` のユニーク制約あり＝重複追加を防止）

### 4.3 フロントエンド設計
- 新規ページ: `karuta-tracker-ui/src/pages/players/PlayerBulkEdit.jsx`
- ルーティング: `App.jsx` に `/players/bulk-edit` を追加
- `PlayerList.jsx`:
  - 選択状態 `selectedIds`（Set）を管理。チェックボックス／全選択／選択人数バッジ／「一括編集」ボタンを追加
  - **団体フィルタ**: 「すべて / 団体別 / 無所属」の表示フィルタを追加。`organizationIds`（空＝無所属）で絞り込む。初期値は ADMIN=`currentPlayer.adminOrganizationId`、SUPER_ADMIN=すべて（`currentPlayer`・`isAdmin`/`isSuperAdmin` は `useAuth()` から取得。`adminOrganizationId` は `LoginResponse` に含まれる）
  - 既存の招待用団体セレクト（`selectedOrgId`）と統合し、具体的な団体選択時のみ招待ボタンを有効化
  - 全選手を選択・編集可（スコープによる無効化はしない）
  - 「一括編集」押下時、選択選手を router state（`{ players: [...] }`）で `/players/bulk-edit` に渡す
- `api/players.js`: `bulkUpdate(updates)` → `PUT /players/bulk` を追加
- 一括編集画面の状態管理:
  - 行ごとの編集中値（性別/級/段位/かるた会/追加予定orgIds）を `useState` で保持
  - 一括設定ボタンは全行へ反映、行操作は当該行のみ反映
  - 級変更時の段位自動連動ロジックは `PlayerEdit.jsx` の `handleKyuRankChange` を踏襲（共通化を検討）
- 確認ダイアログ（既存のダイアログコンポーネント or 簡易モーダル）

### 4.4 バックエンド設計
- `controller/PlayerController.java`: `PUT /bulk`（`@RequireRole(Role.ADMIN)`）を追加
- `dto/PlayerBulkUpdateRequest.java`（新規）: `List<Item> updates`、`Item{ playerId, gender, kyuRank, danRank, karutaClub, addOrganizationIds }`
- `service/PlayerService.java`: `bulkUpdate(request)`（`@Transactional`）を追加
  - 団体スコープ検証は行わない（`@RequireRole(ADMIN)` のみ）
  - 各 player の `players` 列を更新
  - `addOrganizationIds` を既存所属とマージして不足分のみ `playerOrganizationRepository.save`
- `repository/PlayerOrganizationRepository`: 既存 `findByPlayerIdIn` 等を活用（必要なら追加メソッド）
- 級↔段位の整合チェックは単体更新と同じバリデーションを適用

## 5. 影響範囲

### 変更が必要な既存ファイル
- フロント
  - `karuta-tracker-ui/src/pages/players/PlayerList.jsx` … 選択UI・団体フィルタ・一括編集導線。既存の名前検索を維持。**招待用団体セレクトを表示フィルタと統合**するため、招待ボタンの有効/無効条件が変わる（具体的な団体選択時のみ有効）→ 招待機能のデグレに注意
  - `karuta-tracker-ui/src/api/players.js` … `bulkUpdate` 追加
  - `App.jsx`（ルーティング）… ルート追加
  - 新規 `PlayerBulkEdit.jsx`
- バックエンド
  - `controller/PlayerController.java` … エンドポイント追加（既存メソッドは不変）
  - `service/PlayerService.java` … `bulkUpdate` 追加
  - 新規 `dto/PlayerBulkUpdateRequest.java`

### 既存機能への影響
- 単体編集（`PlayerEdit` / `PUT /players/{id}`）には変更を加えない（挙動互換）
- `player_profiles` には触れない（既存の履歴管理に影響なし）
- DBスキーマ変更なし → 本番DBマイグレーション不要

### 互換性
- 新規エンドポイント追加のみ。既存API・レスポンス形式は不変

## 6. 設計判断の根拠

- **専用バルクエンドポイント採用:** 件数分の単体API呼び出しは部分失敗時の不整合リスクとスコープ検証の分散を招くため、トランザクション保証できる単一エンドポイントにした
- **編集テーブル（行＝選手）+ 一括設定ボタンの二段構え:** 「全員を E級／北大・わすら追加」という頻出一括操作と、「行ごとの個別調整」の両方を1画面で満たすユーザー要望に合致
- **追加のみ（削除なし）:** 新入生の所属付与が主目的で、誤操作による所属剥奪を避けるため
- **初期値＝現在値:** 「変更しない」をいちいち選ぶ冗長さを避けつつ、触らなければ現状維持で意図しない上書きを防ぐ
- **ADMINは全選手編集可（トラストベース）+ 団体フィルタ:** 新入生は登録直後に団体未所属のことがあり、ADMINを自団体所属者のみに厳格制限すると新入生を取り込めない。運用上「管理者は悪用しない」前提で全選手編集可とし、代わりに一覧の団体フィルタ（自会／無所属など）で対象選手を素早く見つけられるようにして実用性を確保する
- **DB変更なし:** 必要な属性は既存スキーマで充足。新カラム追加・本番適用の事故リスクを持ち込まない
