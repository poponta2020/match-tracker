# 対戦履歴への読み・一人取り表示 要件定義書

## 1. 概要

### 目的
選手別の月別対戦履歴一覧（[MatchList.jsx](../../../karuta-tracker-ui/src/pages/matches/MatchList.jsx)）に、その選手が「読み（読み手）」「一人取り」を行った回を、試合と同じタイムライン上の**該当試合番号の位置**に表示する。試合に出ていない回も含め、その日の練習で何をしていたかを一目で把握できるようにする。

### 背景・動機
- 抜け番の活動（読み・一人取り等）は `bye_activities` テーブル（[ByeActivity.java](../../../karuta-tracker/src/main/java/com/karuta/matchtracker/entity/ByeActivity.java)）に記録できるが、対戦履歴一覧には**試合（matches）しか表示されず**、読み・一人取りを行った回は履歴に現れない。
- そのため、例えば 6/20 クラ館で 3〜7試合目だけが並び、1・2試合目に読みをしていても履歴は空白に見え、その日の活動全体が分からない。

## 2. ユーザーストーリー
- **対象ユーザー**: アプリ利用者（選手本人 / 他選手の履歴を閲覧する人）。
- **目的**: 月別の対戦履歴で、試合だけでなく自分（または対象選手）が読み・一人取りをした回も時系列で確認したい。
- **利用シナリオ**:
  - 自分の月別履歴を開くと、試合行に交じって「クラ館 2試合目 読み」のような行が該当位置に表示される。
  - 統計エリアに「読み 3回 ・ 一人取り 2回」と表示され、その期間に何回読み・一人取りをしたかが分かる。
  - 他選手の履歴を開いても同様に表示される。

## 3. 機能要件

### 3.1 画面仕様（対象: MatchList.jsx）

**表示対象活動**
- 読み（`READING`）・一人取り（`SOLO_PICK`）の2種のみ。見学・見学対応・その他・休みは**表示しない**。

**行の表示内容**
- 形式: 「`<会場名> <N>試合目 <活動名>`」。例: 「クラ館 1試合目 読み」「クラ館 2試合目 一人取り」。
- 会場名はその選手の**同日の試合（matches）から取得**。会場が特定できない日（その日に試合が0件）は**会場名を省略**し「`<N>試合目 <活動名>`」と表示。
- 対戦相手・勝敗・枚数差・動画・メモ・お手付きは表示しない（該当データがないため）。
- 活動アイコン（読み=BookOpen / 一人取り=User、[MatchResultsView.jsx](../../../karuta-tracker-ui/src/pages/matches/MatchResultsView.jsx) の `ACTIVITY_ICONS` と同様）を付与してもよい（任意）。

**並び順・差し込み位置**
- リスト全体を「**日付の新しい順 → 同じ日は試合番号の降順**」で並べ、読み・一人取り行を該当試合番号の位置に差し込む。
- 既存の試合並び順（バックエンド `MatchRepository.findByPlayerId` の `ORDER BY m.matchDate DESC, m.matchNumber DESC`）と一致。
- 例: 6/20 に 7,6,5,4,3試合目の試合 + 2,1試合目の読みがある場合 → 上から `7 → 6 → 5 → 4 → 3 → 2(読み) → 1(読み)`。

**統計エリアの回数表示**
- 総計カード付近に、対象期間内の「**読み `<n>`回 ・ 一人取り `<m>`回**」を活動ごとに分けて表示。期間（年/月）フィルタに連動。
- 回数が0の活動は非表示（両方0なら回数表示自体を出さない）。
- 上部の勝敗統計（総計・級別）には読み・一人取りを**含めない**（試合ではないため）。

**フィルタとの関係**
- 期間（年/月）フィルタ: 読み・一人取り行と回数表示の**両方に常に連動**。
- 結果フィルタ（勝ち/負け）・対戦相手名検索・級/性別/利き手フィルタ: いずれかが有効なときは読み・一人取り行を**非表示**（対戦相手前提のフィルタで、読み・一人取りには相手がいないため）。回数表示は影響を受けない（期間のみ連動）。

**タップ挙動**
- 読み・一人取り行はタップ無効（表示のみ、遷移なし）。

**対象範囲**
- 自分の履歴・他選手の履歴のどちらでも表示。

### 3.2 ビジネスルール
- 1選手について同一(日付, 試合番号)に試合と抜け番が両立することはない（出ていれば抜け番ではない）ため、差し込み時の重複は発生しない想定。
- 論理削除（`deleted_at`）された抜け番は表示しない（既存リポジトリのクエリで担保済み）。
- データ取得失敗時は、読み・一人取り表示なしでフォールバック（試合履歴は従来どおり表示、コンソールエラーのみ）。

## 4. 技術設計

**方針: フロントエンドのみで完結。バックエンド・DBの変更なし。**

### 4.1 API設計
- 新規エンドポイントなし。既存 `GET /api/bye-activities/player/{playerId}`（[ByeActivityController.java](../../../karuta-tracker/src/main/java/com/karuta/matchtracker/controller/ByeActivityController.java) L58）を使用。
  - `type` パラメータ**なし**で全活動を取得 → フロントで `READING` / `SOLO_PICK` のみにフィルタ（1回の呼び出しで両方取得）。
  - 返却: `ByeActivityDto[]`（`sessionDate`, `matchNumber`, `playerId`, `playerName`, `activityType`, `activityTypeDisplay`, `freeText` 等）。会場情報は含まない。
- 既存 `GET /api/matches/player/{playerId}` ・ `/statistics-by-rank` は従来どおり利用。

### 4.2 DB設計
- 変更なし。

### 4.3 フロントエンド設計（[MatchList.jsx](../../../karuta-tracker-ui/src/pages/matches/MatchList.jsx)）
- **取得**: 既存の matches / stats 取得に加え、`byeActivityAPI.getByPlayer(targetPlayerId)` を取得（`targetPlayerId` 変更時に1回。フィルタ変更では再取得不要）。
- **フィルタ**: 取得した抜け番を `READING` / `SOLO_PICK` に限定し、期間（`selectedYear` / `selectedMonth`）で client-side フィルタ。
- **会場マップ**: matches から `dateString -> venueName` の Map を構築（同日の任意の試合の `venueName`）。抜け番行はこの Map で会場を解決し、なければ会場省略。
- **マージ**: matches と 抜け番(READING/SOLO_PICK) を統合した表示用配列を作り、`(matchDate desc, matchNumber desc)` でソートしてレンダリング。各行は種別（`'match'` | `'bye'`）で描画分岐。
- **回数**: 期間フィルタ後の抜け番から `READING` / `SOLO_PICK` を集計し統計エリアに表示。
- **表示制御**: 結果 / 相手名 / 属性フィルタが有効なときは `bye` 行をレンダリング対象から除外。

### 4.4 バックエンド設計
- 変更なし。

## 5. 影響範囲
- **変更ファイル**: `karuta-tracker-ui/src/pages/matches/MatchList.jsx`（主たる変更）。
- **デグレ確認ポイント**: 既存の試合並びはバックエンド順（date desc, matchNumber desc）と一致するため、merge 後に `(date desc, matchNumber desc)` で再ソートしても**同日内の表示順は不変**であること（現状フロントは date のみで安定ソート）。
- **影響を受けない**: バックエンド、DB、他画面（MatchResultsView 等）。既存APIはレスポンス形式変更なし。
- **既存テスト**: `MatchList.test.jsx` に表示・差し込み順・フィルタ連動・回数表示のケース追加が望ましい。
- **ドキュメント**: `docs/SPECIFICATION.md` / `docs/SCREEN_LIST.md` の対戦履歴一覧の表示仕様を更新。

## 6. 設計判断の根拠
- **フロントのみで完結**: 既存 `/api/bye-activities/player/{playerId}` がそのまま使え、会場は同日試合から借用できるため、バックエンド/DB改修を避けて最小工数で実現（会場不明日は省略で許容）。
- **対象を読み・一人取りに限定**: ユーザー選択。休み/見学等は履歴の主旨（取り・読みなど練習活動の記録）から外れるため除外。
- **相手前提フィルタ時は非表示**: これらのフィルタは対戦相手の属性・勝敗で絞るものであり、相手のいない読み・一人取りを混在させると絞り込み結果として不自然なため。
- **タップ無効**: 読み・一人取り行に紐づく詳細データ（試合詳細・メモ等）がないため、v1では表示のみ。
