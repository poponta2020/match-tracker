# 戦績確認画面の2タブ化（戦績確認／カレンダー）要件定義書

## 1. 概要

- **目的:** 現在の「戦績確認」画面（`/matches` ＝ `MatchList`）を、緑トップバー直下のタブ帯で **「戦績確認」／「カレンダー」** の2タブ構成にする。カレンダータブは自分の試合日を月カレンダーで俯瞰し、日を選ぶとその日の試合（勝敗・相手・お手つき・個人メモ）を確認して試合詳細へ遷移できる。
- **背景・動機:** 戦績確認は級別統計＋試合リストの縦長ビューで、「いつ試合したか」「特定の日の記録・メモを振り返る」という時間軸の閲覧に弱い。カレンダー形式のタブを追加して時間軸の振り返り導線を用意する。既存の戦績確認機能はそのまま1タブ目として維持する。
- **用語補足:** 本アプリで言う「統計画面」は実体として `/matches`（`MatchList`）を指す。`/statistics` は `App.jsx` にルート登録されていない死んだスタブであり本機能とは無関係（放置する）。

## 2. ユーザーストーリー

- **対象ユーザー:** 全ロール（ALL）の選手本人。
- **目的:** 自分の試合を「日付」の軸で振り返り、特定の日の対戦結果・個人メモを素早く確認し、必要なら試合詳細へ進む。
- **利用シナリオ:**
  - 他選手の戦績を見ていて（`?playerId=`）、ふと自分の直近の試合を日付で振り返りたくなり、カレンダータブに切り替える → 自分のカレンダーが出る。戻ると他選手の戦績確認がそのまま残っている。
  - カレンダーで試合数バッジのある日をタップ → その日の試合一覧（勝敗・相手・お手つき・メモ）を見る → 気になる試合をタップして詳細へ。

## 3. 機能要件

### 3.1 画面と遷移（画面インベントリ＋ナビゲーション地図。見た目は design-spec 参照）

- **対象ルート:** `/matches`（変更なし。コンポーネント構成のみ2タブ化）。
- **タブ:** 緑トップバー直下に固定タブ帯として「カレンダー」（左）／「戦績確認」（右）を等幅・下線アクティブ式で配置。既定選択は「戦績確認」。
- **URL とタブ状態:**
  - カレンダータブ ＝ `/matches?view=calendar`、`view` 無指定 ＝ 戦績確認タブ。
  - `playerId` は常に URL に保持する（例: 他選手戦績中は `/matches?playerId=5`、そこからカレンダーへは `/matches?playerId=5&view=calendar`）。
  - タブ切替は `navigate(..., { replace: true })`。戻るボタンでタブがトグルせず、試合詳細から戻ったときは居たタブに復帰する。
- **戦績確認タブ:** 現行 `MatchList` の挙動を完全維持（`?playerId=` で他選手閲覧、「自分に戻す」、級別統計、期間フィルタ、選手検索、試合＋抜け番リスト）。
- **カレンダータブ:** `playerId` を無視し**常に自分（`currentPlayer.id`）**を表示。上部トップバーは自分の名前＋級を表示（期間フィルタ・選手検索ボタンは非表示）。
- **遷移:**
  - カレンダーの選択日リストの各試合タップ → `/matches/:id`（自分の試合なので `playerId` クエリは付けない）。
  - 戦績確認タブ内の既存導線（相手名タップ `/matches?playerId=<opponentId>` 等）は不変。

### 3.2 ビジネスルール（処理・制約・エラー・境界）

- **状態の復元範囲:** カレンダー→戦績確認で復元するのは「どの選手の戦績か（`playerId`）」のみ。期間フィルタ・スクロール位置・選手検索の状態はリセットされてよい（URL の `playerId` 保持だけで満たす）。
- **データ取得:** カレンダーは自分の全試合 `GET /matches/player/{currentPlayer.id}` を1回取得し、クライアント側で日付ごとに集計してカレンダーを構築する（`MatchList` が自分表示時に呼ぶのと同一エンドポイント。enrichment 済みで必要項目が全て揃う）。
- **カレンダー本文:**
  - 月送りナビ（‹ 年月 ›）を本文内に配置。既定表示は当月（JST基準）。初期選択日は今日。月送りは過去・未来とも制限なし（未来月はバッジ無し）。
  - **年月ラベル（`YYYY年M月`）押下で年月ピッカー**（既存 `YearMonthPicker` 流用）を開き、任意の年月へジャンプできる。
  - **年月ラベル直下に当月の総試合数バッジ**（例: `9試合`）を表示し、月移動で連動更新する。
  - 日セルにその日の自分の試合数を緑バッジ（緑丸＋白数字）で表示。**今日＝数字に緑リング／選択日＝セルを角丸ボックスで囲い＋薄緑背景**でハイライト。曜日は日曜=赤／土曜=青。前月・翌月のはみ出しセルはタップ不可のプレースホルダー（薄色）。
  - **カレンダーの集計・リスト対象:** 自分の**試合（`matches`）のみ**。抜け番（読み/一人取り）はバッジ・リストに含めない（「試合の日」に限定）。指導試合は含め、「指導」表示にする。ゲスト/相手不明の自分の試合も含める。
- **選択日リスト:**
  - 見出し帯は `M/D　会場名`（M/D は十の位が0なら省略。**会場は日付に一意の前提**で1回だけ表示。会場不明時は `M/D` のみ。「試合記録」表記は使わない）。
  - 試合番号昇順で表示。各項目の1行目は `N試合目： {勝敗記号}{枚数差} {相手名}（{級}） お手{お手つき回数}` の形式（相手名（級）＝級は単字を全角括弧で。お手つきは半角空白区切りで末尾に付す。級なし＝括弧なし、お手つき不明＝`お手` を出さない）。指導試合は勝敗記号＋枚数差の代わりに「指導」。2行目以降に個人メモ本文（`myPersonalNotes`）。
  - 各項目タップで `/matches/:id` へ遷移。
  - 選択日に試合が無ければ「記録がありません」を表示。**「記録を登録」ボタンは設けない（読み取り専用）。**
- **「コメント」の定義:** 選択日リストで見せる「コメント」は個人メモ（`myPersonalNotes`）を指す。メンター⇔メンティーのコメントスレッド（`MatchCommentThread`）は取得・表示しない。
- **エラー/境界:** 自分の試合が0件でもカレンダーは当月・今日選択で正常表示（全日「記録がありません」）。試合取得失敗時は戦績確認タブの既存挙動と整合するエラーハンドリング（コンソールエラー＋空表示）。

## 5. Non-goals（今回やらないこと）

- カレンダータブからの試合記録の新規登録・編集（「記録を登録」ボタンは設けない。読み取り専用）。
- 他選手のカレンダー閲覧（カレンダーは常に自分のみ）。
- 抜け番（読み/一人取り）のカレンダー表示。
- メンター⇔メンティーのコメントスレッドのカレンダー表示。
- バックエンド改修（新規API・DBスキーマ変更・集計エンドポイント新設）。カレンダーは既存 `GET /matches/player/{id}` レスポンスから構築する。
- `/statistics` スタブの実装・撤去。
- 期間フィルタ・スクロール位置のタブ往復での復元（`playerId` のみ復元）。

## 6. 技術的制約・契約

- **純フロントエンド・1PR:** 変更は `karuta-tracker-ui/src/**` に閉じる。バックエンド・`database/`・`@RequireRole` を触らない。
- **既存契約の維持:** `GET /matches/player/{id}` のレスポンス形状に依存するのみ（`matchDate, matchNumber, result, scoreDifference, player1KyuRank, player2KyuRank, myPersonalNotes, myOtetsukiCount, isLesson, venueName, opponentName, player1Id, player2Id, video, id`）。契約変更なし。
- **壊してはいけない既存挙動:** `MatchList`（戦績確認タブ）の全機能。特に `?playerId=` の他選手閲覧と「自分に戻す」導線。
- **権限:** カレンダーは自分のデータのみを自分が見るため、権限追加は不要（ALL）。
- **既存パターン準拠:** カレンダー描画は `PracticeList.jsx` の `generateCalendar`（前月空白埋め・週配列生成）パターンを流用する。API 呼び出しは既存 `api/matches.js` の `matchAPI.getByPlayerId`。フロントは `pages/` 構成・既存デザイントークン（緑 `#4a6b5a`／クリーム `#f9f6f2` 等）に従う。
- **未解決の技術論点（技術計画フェーズへ申し送り）:**
  - タブ帯とタブ切替時の緑トップバー内容切替を、`MatchList` 内に取り込むか、`/matches` 用のラッパーコンポーネントを新設して `MatchList`（戦績確認ビュー本体）と `MatchCalendar`（新規）を出し分けるか。既存 `MatchList` の独自トップバー（`fixed top-0`）との共存・リファクタ範囲を技術計画で確定する。
  - 相手の級の算出（`player1KyuRank`/`player2KyuRank` のうち相手側）と、相手ID→相手名（`opponentName`）のカレンダー日リストでの利用可否確認。

## 7. 設計判断の根拠

- **URL に `playerId` を常時保持し、カレンダーは無視する**設計により、「他選手戦績→カレンダー（自分）→戻ると他選手戦績」という要件を追加の状態管理（sessionStorage 等）なしに満たす。
- **タブ切替を `replace` にする**ことで、戻るボタンでのタブtrグル（履歴汚染）を避けつつ、試合詳細から戻ったとき居たタブに復帰できる（履歴の末尾が現在タブの URL になる）。
- **BE改修ゼロ**は、必要データが既存レスポンスに揃っておりクライアント集計で完結するため。1PR＝1機能・migration不要でリスクとレビュー負荷を最小化する。
- **抜け番を除外**するのは「試合の日」というユーザー表現に忠実であるため（戦績確認タブでは従来どおり抜け番も表示）。

## デザインへの宿題（→ /design-screen match-record-calendar-tabs）

以下は視覚設計が要る論点。実画面のライブプロトタイピングで詰める（配色はアプリ既存トークンに合わせる。ユーザー提供のモックアップ写真＝Karuta Log のカレンダーに準拠しつつ本アプリの緑基調へ）：

1. **タブ帯**の質感（セグメント/下線アクティブ、緑トップバーとの接続、固定表示時の高さと本文パディング）。既存の `/pairings` 試合番号タブ（cream ハイライト）や他のタブUIとのトーン整合。
2. **カレンダーセル**: 試合数バッジ（緑・数字）、今日ハイライト、選択日ハイライト（モックの青丸をアプリ緑基調へ翻訳）、曜日ヘッダの日=赤/土=青、はみ出しセルの薄色。
3. **選択日リスト項目**のレイアウト（勝敗記号＋枚数差・相手級・相手名・お手つき・メモ本文の階層。メモが長い場合の折返し/truncate）。モックの「試合記録 M月D日」見出し帯の扱い。
4. **カレンダータブのトップバー**（自分の名前＋級）と戦績確認タブのトップバーの見た目整合。
5. 空状態「記録がありません」の見せ方。
