# UXライティング導入 要件定義書

> 本書は「UXライティング導入」という取り組み全体（3層ロードマップ）の文脈を示しつつ、**第1弾（B-1＋B-2）の実装可能な要件**を確定したもの。

## 1. 概要

### 目的
UXライティングの観点から、アプリ内の **操作導線**（ユーザーを次の行動に導く案内・ラベル・ヒント・凡例・空状態メッセージ・操作フィードバック）を強化し、特に **新しく入ったメンバーが迷わず操作できる** ようにする。

### 背景・動機
- アプリを一部公開したところ、ユーザー（特に新メンバー）が操作方法にてこずっている。
- 外部マニュアル（説明書）を作るより前に、そもそも **アプリ内の導線が少なすぎる** ことが根本課題ではないか、という仮説。
- UXライティング = プロダクト内でユーザーの行動を導く「言葉」の設計。記号・色だけに頼らず、言葉で「ここで何ができるか」「次に何をすればよいか」を示す。

### 今回のスコープ方針
- ユーザー選択: **「全体ロードマップ＋第1弾を定義」**、かつ第1弾は **B-1＋B-2 に限定**。
- UXライティング導入を **3層（ガイドライン × 仕組み × 運用）** の段階導入として捉え、第1弾を最小・低リスクで実装する。

## 2. ユーザーストーリー

### 対象ユーザー
- **新しく入ったメンバー**（招待されたばかり・初参加）。スマホ操作に必ずしも慣れていない層も含む。
- （副次的に）すべてのプレイヤー。導線が改善されれば全員が恩恵を受ける。

### ユーザーの目的
- 練習の **出欠・参加を、迷わず・不安なく登録** できる。
- カレンダー上の空き状況（○△×）を **正しく読み取り**、参加判断ができる。

### 利用シナリオ（現状の痛み = ヒアリングで確認済み）
1. 新メンバーがカレンダーの ○△× を見ても意味が分からない（○=空きあり / △=残りわずか / ×=満員 だが **凡例が画面のどこにもない**）。← **第1弾の主訴**
2. セルの背景色（緑/黄）の意味も分からない。← **第1弾で同時に解決**
3. （第2弾以降）出欠登録モーダルで「参加登録／キャンセル登録」のどちらを選ぶべきか分からない 等。

## 3. UXライティング 調査サマリ（設計の拠り所）

### 中核5原則
| 原則 | 内容 |
|------|------|
| 明確 (Clarity) | ボタンを押すと何が起きるか推測させない |
| 簡潔 (Concise) | 不要な語を削る。スマホでは「読まずにスキャンされる」前提 |
| 一貫 (Consistent) | 同じ概念は同じ言葉で。表記・トーンの統一が信頼を生む |
| 有用 (Useful) | 特にエラー/空状態は「何が起きたか＋次に何をすべきか」を示す |
| 人間味 (Human) | 機械的でなく状況に寄り添う。日本語UXライティング最大の課題 |

### 日本語特有の論点（Ubie / OPTiM 等の実プロダクト指針）
- 敬体/常体: UI要素（ボタン・見出し）は敬語を避ける。ユーザーが本来やりたくない操作は敬体で寄り添う。
- 用語の統一: 同義語を1つに集約（例: 試合/対戦/マッチ → 統一）。
- 漢字/ひらがな比 約3:7、横書きは算用数字、カタカナ長音符号は省略しない（「ユーザー」）。
- 1文80字以内・1文1事項・感嘆符（！）は使わない。
- アクセシビリティ: 「下のボタン」でなく「『次へ』を選んでください」（視覚位置に依存しない／**色だけに頼らない**）。← 第1弾の凡例はこの原則の実践。

### 導入の3層
- 第1層 ガイドライン（土台）: ボイス&トーン定義・用語集・表記ルール・メッセージ型テンプレート。
- 第2層 仕組み（基盤）: 文言の一元管理（定数 or i18n）・共通UIコンポーネント（confirm/alert→統一トースト/ダイアログ）。
- 第3層 運用（適用と維持）: 既存画面への適用・表記ゆれ修正・レビュープロセス（PRチェックリスト/lint）。

### 主な参照ソース
- Ubie Vitals UXライティングガイドライン / OPTiM「製品共通のUXライティングのルールを作ってみた」
- Nielsen Norman Group「Error-Message Guidelines」/ Atlassian Design System（Voice and tone, Empty state）
- UX Content Collective（style guide, error messages）

## 5. 段階導入ロードマップ（3層）

```
第1弾（今回）  練習カレンダーの「記号・色の凡例」整備（B-1＋B-2）
第2弾          C層（出欠フローの言葉）／A層（ナビ導線）／オンボーディング・初回ガイド
```

## 6. 第1弾スコープ（確定）

ユーザー確定: **B-1 ＋ B-2 のみ**。練習カレンダー画面（`PracticeList.jsx`）の「記号・色の意味」を **凡例（ⓘ 記号の見方）** で言語化する。

- **B-1** 試合別ステータスグリッドの ○△× の凡例（○=空きあり / △=残りわずか / ×=満員）
- **B-2** セル背景色の凡例（緑=参加確定 / 黄=キャンセル待ち）

### 確定した設計方針（ヒアリング結果）
| 論点 | 確定 |
|------|------|
| 形式 | **ヘルプアイコン（タップで凡例パネルを開く）** |
| アフォーダンス | ラベル付き「**ⓘ 記号の見方**」（bare iconにしない＝新人の発見性向上） |
| 初回表示 | **初回訪問時のみ自動で開く**（`localStorage`で既読管理）。以降は手動 |
| 文言 | **自然・アプリ用語準拠**（○空きあり/△残りわずか/×満員 ｜ 緑=参加確定/黄=キャンセル待ち） |
| 構成 | **2グループに分離**（「試合の空き状況」「あなたの参加状況」）＋「※試合ごとの空き状況です」 |
| パネル方式 | 既存 `YearMonthPicker`（L615）と同じ**アンカー型ドロップダウン**（画面外タップで閉じる） |
| 対象画面 | 練習カレンダー（`PracticeList.jsx`）のみ |
| 技術前提 | **フロントのみ・DB/バックエンド変更なし・マイグレーションなし** |

### 重要な設計上の注意
カレンダーセルには **2種類の異なる情報** が重なっている。凡例はこれらを明確に分離する:
1. **試合の空き状況**（○△×, L713-741）= セッション共通の情報（誰が見ても同じ）
2. **あなたの参加状況**（セル背景色, L664-684）= 個人ごとの情報（緑=参加確定 / 黄=キャンセル待ち。`cancelled`/未参加は通常色）

## 7. 機能要件

### 7.1 画面仕様

#### 凡例アフォーダンス（「ⓘ 記号の見方」ボタン）
- **配置**: カレンダーテーブルの直上、右寄せ（エラー表示とカレンダー本体の間。L645付近）。
- **表示**: `lucide-react` の `Info`（または `HelpCircle`）アイコン＋テキスト「記号の見方」。常時表示。
- **操作**: タップで凡例パネルを開閉。
- **アクセシビリティ**: `aria-label="記号の見方を開く"`。

#### 凡例パネル
- **形式**: ボタンにアンカーしたドロップダウン（既存 `YearMonthPicker` と同方式）。ボタン直下に表示。
- **閉じる**: パネル外タップ、再タップ、または閉じる操作。
- **内容**（2グループ＋補足）:

  ```
  記号の見方

  ▎試合の空き状況（試合ごと）
    ○  空きあり        ← 緑
    △  残りわずか      ← オレンジ
    ×  満員            ← 赤

  ▎あなたの参加状況
    ■  参加確定        ← 緑系（セル背景と同色）
    ■  キャンセル待ち  ← 黄系（セル背景と同色）
  ```
  - ○△× は実際の記号を**カレンダー本体と同じ配色**で表示。
  - 参加状況は実際の**セル背景色のスウォッチ（小さな色付き四角）**で表示。
  - 「（試合ごと）」「※試合ごとの空き状況です」の補足を試合グループに添える。

#### 初回自動表示
- この画面を初めて開いた端末では、表示時に凡例パネルを**自動で開く**。
- `localStorage` キー `practiceCalendarLegendSeen` で既読管理。値があれば自動表示しない。
- 自動表示のタイミングで既読フラグを保存し、以降は手動のみ。
- **端末単位**（ユーザー単位ではない）。`localStorage` が使えない環境では try/catch でガードし、最悪「毎回開く」挙動になっても機能は阻害しない。

### 7.2 ビジネスルール
- 凡例は**表示専用**。データ変更・API呼び出しは一切ない。
- 月の切り替えやセッション有無に関わらず、凡例ボタンは**常に表示**。
- ○△× の意味は既存の判定ロジック（`matchCapacityStatuses`）に準拠。凡例はその意味を文章化するのみで、**判定ロジックは変更しない**。

### 7.3 エラーケース・例外処理
- `localStorage` 例外（プライベートモード等）→ try/catch で握りつぶし、カレンダー表示を阻害しない。
- グリッド非表示日（複数セッション/capacity未設定 等）でも、凡例ボタン・パネルは表示する（一般的な見方の説明のため）。

## 8. 技術設計

### 8.1 API設計
- **変更なし**。

### 8.2 DB設計
- **変更なし**（マイグレーションSQLなし。CLAUDE.md のDB適用ルールは**非該当**）。

### 8.3 フロントエンド設計
#### 変更ファイル
| ファイル | 変更内容 |
|---------|----------|
| `pages/practice/PracticeList.jsx` | 凡例ボタン＋凡例パネル＋初回自動表示ロジックを追加（**追加のみ**、既存セル描画は不変） |

#### 実装方針
- **state追加**: `showLegend`（パネル開閉）。初期値は初回判定で決定。
  ```js
  const [showLegend, setShowLegend] = useState(() => {
    try { return !localStorage.getItem('practiceCalendarLegendSeen'); }
    catch { return false; }
  });
  useEffect(() => {
    if (showLegend) {
      try { localStorage.setItem('practiceCalendarLegendSeen', '1'); } catch {}
    }
  // 初回マウント時のみ既読フラグを保存する意図（依存配列は実装時に確定）
  }, []);
  ```
- **凡例ボタン＋パネル**: 同ファイル内のインライン JSX（または同ファイル内サブコンポーネント `CalendarLegend`）。**汎用化はしない**（既存の同画面方針＝グリッド等もインライン実装に合わせる。汎用 Legend 化は第3弾の横展開時に検討）。
- **配色**: `symbolFor`（L721-725）と同じ色クラスを使う（○ 緑 / △ オレンジ / × 赤）。参加状況スウォッチはセル背景色（confirmed: `bg-[#e8efea]`/`border-[#a3c4ad]`、waitlisted: `bg-[#fefcf5]`/`border-[#e8d48b]`）に揃える。視認性は実装時に微調整可。
- **画面外タップで閉じる**: 既存 `YearMonthPicker` のクローズ方式に合わせる（実装一貫性）。
- **アイコン**: `lucide-react`（`Info` / `HelpCircle`）。

### 8.4 状態管理
- 既存ステート（`sessions` 等）への追加はなし。`showLegend` のみ追加。

- カレンダー直上に凡例ボタン1行分の高さが加わる。**セル描画・グリッド・背景色ロジックは不変**（B-1/B-2 は「説明を足す」だけで表示自体は変えない）。

  - ボタンタップでパネルが開閉する
  - 初回自動表示: `localStorage` 未設定→自動で開く / 設定済み→開かない

### ドキュメント更新（CLAUDE.md ルールに従い同コミットで実施）

- `docs/SCREEN_LIST.md`: 該当画面の説明に凡例を追記。
- `docs/DESIGN.md`: API/DB変更なしのため原則更新不要（必要なら凡例UIの補足のみ）。

### 本番DB
- **マイグレーションなし**。Render本番DBへの適用作業は**不要**。

## 10. 設計判断の根拠

| 判断 | 理由 |
|------|------|
| 凡例で対応（セル内にテキストを増やさない） | グリッドは省スペース設計（9px）。意味の説明は凡例に外出しするのが現実的。表示自体を変えないため低リスク。 |
| ヘルプアイコン（タップ式） | ユーザー選択。省スペース優先。ただし新人到達のため「**ラベル付き＋初回自動表示**」で発見性を補強。 |
| 2グループに分離 | セルには「試合の空き（共通）」と「自分の参加状況（個人）」の2種類の情報が重なるため、混同を防ぐ。 |
| 初回自動表示を localStorage で1回だけ | 毎回出すと既存ユーザーに邪魔。新人に1度確実に見せる目的を最小コストで達成。 |
| `PracticeList.jsx` インライン実装 | 既存の同画面方針（グリッド等もインライン、汎用化しない）に合わせ、第1弾を小さく保つ。汎用化は第3弾で検討。 |
| 第1弾を B-1＋B-2 に限定 | 同一画面・同一テーマ（記号/色の言語化）で完結し、フロントのみ・DB変更なしで低リスク。効果を見てから横展開できる。 |
