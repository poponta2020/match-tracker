# 抽選機能 連鎖落選ロジック改修 要件定義書

## 1. 改修概要

### 対象機能
抽選機能（Lottery） — `LotteryService.processMatch` の連鎖落選（cascade loser）ロジック

### 改修の背景
連番試合で前試合の落選者が次試合でも自動落選になる仕様により、定員に空きがあるにも関わらず落選するケースが発生している。

**ユーザーの意図する連鎖落選:**
- 定員超過時は、同じ人がまとめて落選するほうが望ましい（会場までの移動コストを考慮）
- ただし、空きがあるなら前試合の落選者も当選させるべき

### 改修スコープ

| # | 項目 | 分類 |
|---|------|------|
| 1 | processMatch の連鎖落選ロジック修正 | バグ修正 |

---

## 2. 改修内容

### 現状の問題

`LotteryService.processMatch`（L259-282）で、前試合の落選者（`cascadeLosers`）を抽選対象から完全に除外し、`remaining`（非落選者）のみで当選者を決定している。`remaining` が定員以下でも余った枠に `cascadeLosers` を入れるロジックがない。

**再現例:**
- 定員6、試合2の申込者7名（うち前試合落選者 G, H の2名）
- remaining = A, B, C, D, E（5名）→ 全員当選
- 空き1枠があるのに G, H は両方落選

### 修正方針

`processMatch` の Step 1〜2 を以下に変更:

1. `remaining`（前試合で落選していない人）を既存の優先当選/一般抽選ロジックで処理し、当選者を決定
2. **余り枠 = `capacity - winners.size()`** を計算
3. 余り枠 > 0 の場合:
   - `cascadeLosers` の人数 <= 余り枠 → 全員当選
   - `cascadeLosers` の人数 > 余り枠 → ランダム抽選で余り枠分を当選、残りは落選
4. 余り枠 = 0 の場合: `cascadeLosers` は全員落選（現行と同じ）

### 修正後のあるべき姿

- 空きがあれば前試合落選者も当選する
- 定員超過時は前試合落選者が優先的に落選する
- 同じ人がまとめて落選するという基本方針は維持される

## 4. 影響範囲

### 影響を受ける既存機能
- `processMatch` は `processSession`（通常抽選）と `reExecuteLottery`（再抽選）の両方から呼ばれるため、両方に自動的に反映される
- プレビュー機能も同じ `processSession` を使用するため、プレビュー結果にも反映される

なし（API・DB・フロントエンドに変更なし）

## 5. 設計判断の根拠

| 判断 | 理由 |
|------|------|
| cascadeLosers を「自動落選」から「低優先度」に変更 | 空きがあるのに落選させるのはユーザーの意図に反する |
| remaining の抽選ロジック（優先当選/一般枠）はそのまま維持 | 連鎖落選の扱いだけが問題であり、他のロジックは正常に機能している |
| cascadeLosers 内の余り枠抽選はランダム（優先当選対象外） | cascadeLosers は既にセッション内で落選した人であり、月次優先当選とは別の概念 |
