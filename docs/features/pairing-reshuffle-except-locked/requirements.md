# ロック以外を再シャッフル（pairing-reshuffle-except-locked）要件定義書（ドラフト）

## 1. 概要

- **目的:** `/pairings`（組み合わせ作成）で、一部の組をロックした状態のまま、**ロックしていない組だけを既存の最適化アルゴリズムで再シャッフル**できるようにする。
- **背景・動機:** 現在「対戦編集」（＝自動マッチング）ボタンは組み合わせが1つも無い時（`pairings.length === 0`）だけ表示され、一度組み合わせを作ると消える。そのため「良い組はロックで固定し、残りだけ組み直す」という運用が UI 上できない。バックエンドの自動マッチング（`autoMatch`）は既にロック済みの組を除外して残りを最適化する能力を持つが、その導線が塞がっている。

## 2. 変更の動機と内容（delta）

| 観点 | 現行 | 変更後 |
|---|---|---|
| 再シャッフル導線 | 組み合わせ未作成時のみ「対戦編集」ボタン。作成後は消える | 組み合わせがある編集可能状態でも**再シャッフルボタンを常設** |
| ロック済みの組 | 全削除では保持されるが、再最適化する導線が無い | ロック済みの組を保持したまま**残りを再最適化** |
| 保存前ロック | ローカル下書き。再シャッフル導線が無いため関与しない | **未保存の手動ロックも尊重**して再シャッフル |

## 3. 機能要件

### 3.1 画面と導線（見た目の詳細は design-spec / SCREEN_LIST に委譲）

- 対象画面: `/pairings`（`PairingGenerator.jsx`）。**1つの再シャッフルボタン**を追加する。
- **表示条件:** 組み合わせが1つ以上あり（`pairings.length > 0`）、編集可能状態（自動マッチング直後、または「編集」で編集モードに入った後。`isReadOnly` / 閲覧モードでは非表示）。**保存済みの閲覧モードからは「編集」を1タップしてから使う**（＝ボタンは編集可能状態に出す。確定済み）。
- **ボタン文言と挙動はロック済みの組の有無で切り替わる（動的）:**
  - **ロック済みの組が1つ以上ある場合**（`hasResult` または `locked` の組が1件以上）: 文言「**ロックされた組以外をシャッフル**」。挙動＝ロック済みの組を保持し、それ以外を再シャッフル。
  - **ロック済みの組が0件の場合**: 文言「**再シャッフル**」（確定）。挙動＝全参加者を再シャッフル。
- **確認ダイアログ:** 実行前に `window.confirm` で確認する（既存「全削除」と同様）。キャンセル時は何も変えない。
- 既存の新規作成用「対戦編集」ボタン（`pairings.length === 0` 時のみ表示）は**そのまま残す**（文言・挙動不変）。

### 3.2 ビジネスルール

- **保持対象（ロック済み）:** 結果入力済み（`hasResult`）の組 ＋ 手動ロック（`locked`）の組。**未保存の手動ロックも含める**（＝画面の現在の状態を基準にする）。保持組の選手はそのまま同じ組で維持。
- **再シャッフル対象:** 非ロック組の選手 ＋ 現在の待機者。これらを1つのプールに戻し、既存の最適化アルゴリズム（過去30日のペアリング/対戦履歴・スコア = -(100/最終対戦からの日数)・貪欲法・同日既存ペア除外）で組み直す。
- **アクティブ参加者の真:** 誰が参加者か（当日キャンセル等）は従来どおり DB（`loadActiveParticipantIdsForMatch`）を真とする。ロックの除外だけを画面の現在状態から受け取る。表示中参加者数と対象参加者数が食い違う場合は既存の注意メッセージ挙動を踏襲。
- **奇数人数:** 再シャッフル対象が奇数なら1名が待機者になる（既存挙動）。
- **実行後の状態:** 編集可能状態（未保存変更あり）になる。手動で微調整し「確定して保存」で永続化する（保存の契約は従来どおり）。
- **権限:** 既存 `auto-match` と同一（PLAYER+。ADMIN/PLAYER は自/所属団体のみ）。

## 5. Non-goals（今回やらないこと）

- ロックの明示保存モデル自体は変えない（永続化は従来どおり「確定して保存」。再シャッフルは現在の下書き状態を尊重するだけ）。
- 自動マッチングのスコアリングアルゴリズム自体は変えない（既存を再利用）。
- `PairingSummary`（札ルール一覧）・LINE送信用テキスト生成には手を入れない。
- 二重ブッキング検証（旧 `PATCH /lock` のサーバー検証）の復活はしない。
- 「全削除」ボタンの廃止・統合はしない（再シャッフルとは別機能として残す）。
- DB スキーマ変更はしない。

## 6. 技術的制約・契約

- **公開契約の拡張（後方互換）:** `POST /api/match-pairings/auto-match` は現在 `{sessionDate, matchNumber}` だけを受け取り、ロック状態を DB から再導出する。未保存ロックを尊重するため、**「現在ロック中の組」をクライアントから受け取れるよう後方互換で拡張する**必要がある（既存の新規作成フローはロック無しで呼ぶため従来どおり動く）。具体的な request 形状は技術計画（Step 4）で確定。
- **権限:** 拡張後も `@RequireRole` は既存 auto-match と同一（PLAYER+、団体スコープ）。
- **DB:** スキーマ変更なし。保持判定は既存の `match_pairings.locked` / `matches` 存在で行う。
- **アクティブ参加者:** DB（`loadActiveParticipantIdsForMatch`）を真とする既存挙動を維持。
- **未解決の技術論点（Step 4 で解決）:** ①auto-match 契約拡張 or 別エンドポイントか ②「保持する組」をどの形でクライアントから渡すか。（③ボタンの閲覧モード表示は「編集モードで出す」で確定済み）

## 7. 設計判断の根拠

- **アルゴリズムは再利用:** ロック除外＋残り最適化はバックエンドに既存。新規に組み直さず既存 `autoMatch` を活かす（挙動の一貫性・実装最小）。
- **画面の現在状態を基準（未保存ロック尊重）:** ユーザーの直感（「ロックをかけた後、すぐ残りをシャッフル」）に沿う。保存を挟む運用より一手間少ない。代償として auto-match の契約を後方互換で拡張する。
- **1ボタン・動的文言:** ロック有無で意味が変わることを文言で明示し、ボタン数を増やさない（ユーザー要望）。
