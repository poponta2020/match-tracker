# 自動シャッフルで前回練習日の対戦相手も避ける 要件定義書（改修）

## 1. 概要

- **目的:** 対戦組み合わせの自動シャッフル（自動マッチング）で、これまで避けていた「同日（前の試合番号）」に加えて、**同一団体の前回練習日**に組んだペアも強く避けるようにし、練習ごとの対戦相手の被りを減らす。
- **背景・動機:** 現状のスコアは「最後の対戦からの日数」に応じた連続的なペナルティ（`-(100 / 日数)`）で、前回練習日の相手も一応スコアは下がるが「強く避ける」ほどではない。特にわすらもち会のような少人数常連クラブでは、毎回同じ顔ぶれで前回と同じ相手に当たりやすい。前回の相手を優先的に避けたい。

## 2. ユーザーストーリー（改修のため要点のみ）

- 対象ユーザー: 組み合わせを作成する ADMIN 以上（自動シャッフルを実行する運用者）。
- ねらい: 「前回この人とやったから、今回は別の人と」を、手動で組み替えなくても自動シャッフルが最大限やってくれる状態。

## 3. 機能要件（変更差分）

### 3.1 画面と遷移

- 画面・UI・API の形は**変更なし**。既存の「対戦編集 / 再シャッフル」ボタン（`POST /api/match-pairings/auto-match`）の内部スコアリングのみを変更する。
- リクエスト／レスポンスの契約（`AutoMatchingRequest` / `AutoMatchingResult`）は不変。

### 3.2 ビジネスルール（変更後の挙動）

**A. 前回練習日ペアの強い回避（新規）**

1. 自動シャッフル実行時、対象セッションの**団体の「前回練習日」**を特定し、その日に組まれた全ペア（全試合番号）を「前回練習日ペア集合」として求める。
2. スコア計算で、候補ペアが「前回練習日ペア集合」に含まれる場合、通常の日数ベーススコアに加えて**十分に大きい固定ペナルティ**を加算する（非・前回練習日の候補より必ず下位になり、かつ有限＝最後の手段としては選択可能な値）。
3. これにより、**他に組める相手がいる限り前回練習日のペアは再形成されない**。一方、それ以外に組みようがない場合（下記「変わらないもの」の少人数ケース）は、待機者を増やさず前回練習日ペアを許容する（**グレースフルに劣化**）。

**B. 「前回練習日」の定義**

- **同一団体スコープ**で特定する。北大かるた会のセッションの再シャッフルは北大の練習日のみ、わすらもち会は わすらもち会 の練習日のみを見る。
- 「前回練習日」= 今回のセッション日より前で、**実際に対戦（組み合わせ／試合結果）がある**直近の練習日 1 日分。
  - 直前の練習日が対戦なし（中止・記録なし）の場合は、対戦がある日まで**さらに遡る**。
- 前回練習日ペアは `match_pairings`（組み合わせ）と `matches`（試合結果）の**両方**から収集する（既存の履歴マージと同じ扱い）。
- **団体非限定（SUPER_ADMIN）実行時**は、対象セッションから解決した実効団体ID（現行 `autoMatch` が参加者判定で使うのと同じ、`session.getOrganizationId()`）で前回練習日を特定する。同日複数団体は現行実装が既に単一セッション取得の縮退挙動であり、本改修でも新たな横断処理は追加しない。

**C. スコアの優先順位（変更後）**

| 状況 | 扱い |
|---|---|
| 同日（前の試合番号）で対戦済み | 候補から**完全除外**（従来どおり・不変） |
| **前回練習日で対戦済み（同一団体）** | **強い固定ペナルティで強く回避**（新規。最後の手段としてのみ選択） |
| 過去（前回練習日でない日）に対戦 | 従来どおり `-(100 / 日数)` の連続ペナルティ（不変） |
| 過去30日で一度も対戦なし（初対戦扱い） | スコア 0・最優先（不変） |

## 4. Acceptance Criteria

| ID | 条件（客観的に判定できる文） | 検証手段 |
|----|------|------|
| AC-1 | 前回練習日に組んだペアは、他に組める相手がいる限り自動シャッフルで再形成されない（例: 参加者3名で前回1-2が対戦した場合、結果ペアは 1-3 または 2-3 になり 1-2 にならない） | auto-test |
| AC-2 | 前回練習日ペアしか残らない場合でも、待機者を増やさず前回練習日ペアで組む（例: 4名で前回3試合の全ペアを消化済みでも、2組が形成され待機者は出ない＝グレースフル劣化） | auto-test |
| AC-3 | 前回練習日の特定は同一団体スコープ。ある団体のセッションの再シャッフルは、別団体の同日以外の練習ペアをペナルティ対象にしない | auto-test |
| AC-4 | 直前の練習日に対戦がない場合、さらに遡って直近の対戦がある練習日を「前回練習日」として扱う | auto-test |
| AC-5 | 同日（前の試合番号）ペアの完全除外という既存挙動が維持される | auto-test |
| AC-6 | ロック済み・結果入力済みペアの保護、待機者算出、団体スコープ（他団体ペア混入防止）など既存の自動マッチング挙動が維持される | auto-test |
| AC-7 | 初対戦（過去に一度も対戦がない）ペアが最優先される既存挙動が維持される | auto-test |
| AC-8 | 参加者が多くペアの選択肢が十分ある通常時は、前回練習日ペアが回避されつつ全員が組まれる（少人数以外で待機者が新たに増えない） | auto-test |
| AC-9 | 既存テスト・lint・typecheck がすべて成功する | auto-test |

## 5. Non-goals（今回やらないこと）

- **手動編集時のリアルタイム履歴表示への警告追加**（現状の `⚠今日` 警告に「前回練習日」警告バッジを足すこと）。今回は自動シャッフルのスコアリングのみ。
- **前々回以前の練習日の回避**。対象は「前回（直近の対戦がある練習日）」1 日分のみ。
- **既存スコアパラメータの変更**（30日窓 `MATCH_HISTORY_DAYS`、同日ペナルティ、`-(100/日数)` 式）。前回練習日ペナルティを**加算する**のみで既存式は据え置き。
- **ペナルティ強度やON/OFFのユーザー設定UI**。固定挙動とする。
- **同日除外を「強い回避」に緩和すること**。同日は従来どおり完全除外を維持。

## 6. 技術的制約・契約

- **公開契約は不変**: `POST /api/match-pairings/auto-match` のリクエスト（`AutoMatchingRequest`）・レスポンス（`AutoMatchingResult`）は変更しない。フロント変更なし。
- **DBマイグレーション不要（N/A）**: スキーマ変更・`@Column` 変更なし。追加するのは読み取り専用の finder（`practice_sessions` を団体・日付降順で引く）1 本のみ。DoD のマイグレーション項目は N/A。
- **スコープ規律（1PR=1機能）**: `getPairingHistory` / `getMatchHistory` が選手ID＋期間ベースで団体スコープを持たない既存のニュアンスは**本PRでは是正しない**。新規の「前回練習日ペア」検出のみ明示的に団体スコープを適用する。
- **前回練習日検出は30日窓に依存しない**: スコアリング用の履歴取得（過去30日）とは別に、団体の過去セッションを遡って前回練習日を特定する。前回練習日が30日より前でも回避対象にする（意図的な設計判断）。
- **テスト可能性**: `autoMatch` は `Collections.shuffle` により非決定的。ACは shuffle 不変な性質（AC-1 は「1ラウンドで全体最高スコアを選ぶため -大ペナルティのペアは選ばれない」／AC-2 は「ソフトペナルティゆえ組数・待機者数が変わらない」）で検証する。既存 `MatchPairingServiceTest` の Mockito スタブ方式（各 repository finder をモック）に従う。

## 7. 設計判断の根拠

- **ハード除外でなくソフト強ペナルティ（グレースフル劣化）**: 少人数常連クラブ（例: わすらもち会の常連4名・3試合）は、同日除外だけで1日に「4名の全3ペア」を消化するため、前回も完全除外すると組める相手が一組も無くなり、来た人が試合できず待機になる。全員をなるべく試合させることを優先し、前回ペアは「最後の手段」としてのみ許容する（ユーザー選択）。
- **既存の貪欲法を維持（最適マッチングに置き換えない）**: 「最後に残った2人が最近対戦」型の貪欲法の末端非最適はユーザーに開示済みで、今回の受容範囲。全体最適化（最大重みマッチング）への置き換えはスコープ外。
- **前回練習日ペナルティは加算方式**: `calculatePairScore` に前回練習日ペア集合を渡し、日数ベーススコアに固定ペナルティを加算する。呼び出し元は `autoMatch` のみ（確認済み）でシグネチャ変更は局所。
- **効果が出るのは制約の強い（少人数）場面に集中**: 参加者が多い通常時は既存スコアで十分に最近ペアを避けられており、本改修の可視効果はわすらもち会のような少人数常連ケースに集中する。これは意図した改善であり、従来挙動が壊れていたわけではない。
