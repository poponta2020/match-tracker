# キャンセル待ちオファー通知の統合 改修要件定義書

## 1. 改修概要

### 対象機能
キャンセル待ち繰り上げオファーのLINE通知（プレイヤー向け）

### 改修の背景
現在、同一セッション内で複数試合のオファーが発生した場合、試合ごとに個別のLINEメッセージが送信される。1人のプレイヤーが同一セッション内で複数試合のオファーを受けると、試合数分のメッセージが届き煩雑になる。管理者向け通知は既にバッチ化されているが、プレイヤー向けオファー通知は未対応。

### 改修スコープ
1. 同一セッション×同一プレイヤーのオファー通知を1通のFlexメッセージに統合
2. 個別参加/一括参加/一括辞退のボタンを提供
3. 部分参加後の残りオファー通知フローを追加
4. Web版オファー応答画面（OfferResponse）の統合対応

## 2. 改修内容

### 2.1 オファー通知の統合

**現状の問題:**
`promoteNextWaitlisted()` 内で `lineNotificationService.sendWaitlistOfferNotification()` が試合ごとに即時送信される。

**修正方針:**
管理者通知と同じSuppressedパターンを適用する。`promoteNextWaitlisted()` からLINE通知送信を分離し、呼び出し元（DensukeImportService / LotteryController / LotteryService）で同一セッション×同一プレイヤーのオファーをグルーピングして統合通知を送信する。

**修正後のあるべき姿:**
同一セッション内で複数試合のオファーを受けたプレイヤーには、1通の統合Flexメッセージが届く。

### 2.3 新しいアクション種別

| アクション名 | パラメータ | 動作 |
|------------|-----------|------|
| `waitlist_accept` (既存) | `participantId` | 個別試合の参加承諾 |
| `waitlist_decline` (既存) | `participantId` | 個別試合の辞退 → **廃止対象**（一括辞退に統合） |
| `waitlist_accept_all` (新規) | `sessionId` + `playerId` | 同一セッション内の全OFFEREDを一括承諾 |
| `waitlist_decline_all` (新規) | `sessionId` + `playerId` | 同一セッション内の全OFFEREDを一括辞退 |

- 個別の「辞退する」ボタンは廃止し、「辞退する」ボタンは常に一括辞退（`waitlist_decline_all`）として動作する
- 個別の「〇試合目に参加」ボタンは既存の `waitlist_accept` をそのまま使用

### 2.4 部分参加後の残りオファー通知

**フロー:**
1. プレイヤーが「1試合目に参加」を押す
2. `respondToOffer()` で1試合目を WON に変更
3. 同一セッション×同一プレイヤーで残りのOFFEREDを検索
4. 残りがあれば「残りの試合のオファー」Flexメッセージを送信
5. 残り1試合 → パターン3、残り2試合以上 → パターン4
6. プレイヤーがさらに個別参加 → 残りがあれば再度通知（繰り返し）

### 2.5 応答期限の統一

同一セッション内の複数オファーで応答期限が異なる場合、**最も遅い期限に統一**する。
- 統合メッセージ送信時に、対象オファー群の中で最も遅い `offerDeadline` を表示
- 各オファーレコードの `offerDeadline` 自体は個別のまま保持（DB変更なし）

### 2.6 トリガー情報の表示

**「〇〇がキャンセル」の表示ルール:**
- キャンセルが原因の場合: 「山田太郎がキャンセル」
- オファー辞退が原因の場合: 「山田太郎がオファー辞退」
- 複数人のキャンセル/辞退で発生した場合: トリガー情報は `AdminWaitlistNotificationData` から取得。同一トリガーであれば1名表示、異なるトリガーの場合は最初のトリガー情報を表示

## 3. 技術設計

### 3.1 API変更

**新規エンドポイント:**

| メソッド | パス | 説明 |
|---------|------|------|
| POST | `/api/lottery/respond-offer-all` | 同一セッション内の全OFFEREDを一括承諾/辞退 |

**リクエストBody:**
```json
{
  "sessionId": 123,
  "accept": true
}
```
- `playerId` はサーバー側で認証情報から取得

**既存エンドポイント（変更なし）:**
- `POST /api/lottery/respond-offer` — 個別応答はそのまま維持
- `GET /api/lottery/offer-detail/{participantId}` — 個別オファー詳細はそのまま維持

### 3.2 DB変更

なし。既存のテーブル構造で対応可能。

### 3.3 フロントエンド変更

| ファイル | 変更内容 |
|---------|---------|
| `OfferResponse.jsx` | 同一セッション内の複数OFFEREDに対応。個別参加/一括参加/一括辞退のUI。部分参加後の画面更新 |
| `lottery.js` | `respondOfferAll(sessionId, accept)` API呼び出し追加 |

### 3.4 バックエンド変更

| ファイル | 変更内容 |
|---------|---------|
| `WaitlistPromotionService` | `promoteNextWaitlisted()` からLINE通知を分離（アプリ内通知 `createOfferNotification` は残す）。`respondToOffer()` に部分参加後の残りオファー通知ロジック追加。`respondToOfferAll()` / `declineOfferAll()` メソッド追加 |
| `LineNotificationService` | `sendWaitlistOfferNotification()` を統合版に改修（`List<PracticeParticipant>` + トリガー情報を受け取る）。残りオファー通知用メソッド追加。`buildConfirmMessage()` に新アクション対応追加 |
| `LineWebhookController` | `CONFIRMABLE_ACTIONS` に `waitlist_accept_all` / `waitlist_decline_all` 追加。`handleConfirmationRequest()` / `executeOriginalAction()` に新アクションのハンドリング追加 |
| `LotteryController` | キャンセル一括処理後のオファー統合通知送信。`respondToOfferAll` エンドポイント追加 |
| `LotteryService` | `adminEditParticipants()` 内の `promoteNextWaitlisted()` を通知抑制版に切り替え |
| `DensukeImportService` | `AdminWaitlistNotificationData` の `promotedParticipant` をプレイヤーIDでグルーピングし、統合オファー通知を送信 |
| `NotificationService` | `createOfferNotification()` を統合版に対応（複数試合分の情報を含む通知を1件作成） |

## 4. 影響範囲

### 影響を受ける既存機能

| 機能 | 影響 | 対策 |
|-----|------|------|
| LINE連携後の未送信オファー一斉送信（LineNotificationService:687） | `sendWaitlistOfferNotification()` のシグネチャ変更時に壊れる | 既存メソッドを維持するか、内部で統合版を呼ぶようにする |
| OfferExpiryScheduler → expireOffer() | 期限切れ1件ごとに次のオファー送信（単発） | 影響なし |
| Web版 OfferResponse | URLパラメータ `?id=participantId` が変わる可能性 | 後方互換維持。1件の場合は現状と同じ表示 |
| 確認ダイアログフロー | 新アクション追加 | 既存アクションには手を加えない |

### 破壊的変更の有無

- APIの破壊的変更: なし（新規エンドポイント追加のみ、既存は維持）
- DBスキーマの破壊的変更: なし
- LINEメッセージのpostbackデータ: 既存の `waitlist_accept` / `waitlist_decline` は維持。新規アクション追加のみ

## 5. 設計判断の根拠

| 判断 | 理由 |
|------|------|
| 管理者通知と同じSuppressedパターンを採用 | 既存の仕組み（`cancelParticipationSuppressed` + `sendBatchedAdminWaitlistNotifications`）が実績あり。同じパターンを踏襲することで一貫性を保つ |
| 応答期限は最も遅い期限に統一 | 各試合に個別対応する余地を残すため。最も早い期限だと、一部の試合で本来の期限より早く切れてしまう |
| 部分参加時に未応答オファーは継続 | プレイヤーに試合ごとの選択肢を残す。一括辞退は別途ボタンで提供 |
| 辞退ボタンは常に一括辞退 | 統合メッセージ内での個別辞退はUXが複雑になる。辞退したい場合はまとめて辞退するのが自然 |
| 部分参加後に毎回残りオファー通知を送信 | プレイヤーが残りのオファーを忘れないようにするため |
| 1試合のみの場合「すべての試合に参加」を非表示 | 1試合しかないのに「すべて」は不自然 |
