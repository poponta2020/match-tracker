# 管理者向けキャンセル待ち状況通知 Flex Message 改修 要件定義書

## 1. 概要

### 目的
管理者向けキャンセル待ち状況通知のLINE Flex Messageを改修する。同一セッション・同一プレイヤー・同一トリガーの複数試合を1通のメッセージにまとめ、ヘッダー・本文の文言を刷新して視覚的にわかりやすくする。

### 背景・動機
現在は1試合ごとに1通の通知が送信されるため、伝助同期等で同一セッションの複数試合が同時にキャンセルされると管理者に大量の通知が飛んでしまう。同じセッションの情報であれば1通にまとめたい。

## 2. ユーザーストーリー

### 対象ユーザー
- SUPER_ADMIN + 該当セッションの団体に所属するADMIN（LINE連携済み・`ADMIN_WAITLIST_UPDATE` 通知設定ON）
- WAITLISTEDユーザー（順番繰り上がり通知として同じFlexを受信）

### ユーザーの目的
- キャンセル・降格・辞退等が発生した際に、どのセッションのどの試合で何が起きたかを1通で把握したい

### 利用シナリオ
- ある選手が同一セッションの1, 2, 3試合目に参加していてキャンセル → 3通ではなく1通で「1, 2, 3試合目」と通知される
- 異なるセッションのキャンセルは別メッセージ
- 異なる選手のキャンセルは別メッセージ
- 異なるトリガー種別（キャンセルとオファー辞退等）は別メッセージ

## 3. 機能要件

### 3.1 まとめの単位

通知のグルーピングキー: **セッションID × トリガー種別 × プレイヤーID**

同一キーに該当する複数試合のイベントを1通のFlex Messageにまとめる。

### 3.2 Flex Message 構成（改修後）

#### ヘッダー（紫 `#8E44AD`）

トリガー種別に応じたテキスト:

| triggerAction | ヘッダーテキスト |
|---|---|
| `"キャンセル"` | キャンセル発生通知 |
| `"キャンセル（当日補充）"` | 当日キャンセル発生通知 |
| `"降格"` | 練習参加者手動入替通知 |
| `"オファー辞退"` | オファー辞退通知 |
| `"キャンセル待ち辞退"` | キャンセル待ち辞退通知 |
| `"オファー期限切れ"` | オファー辞退通知（期限切れ） |

#### ボディ

1. **セッション情報**: `〇月〇日（会場名：定員〇〇名）`
   - 現在の `getSessionLabel` とは異なる専用フォーマット（他の通知には影響しない）

2. **該当試合**: `1, 2, 3試合目`（カンマ区切り）

3. **イベント内容**: `{選手名} が {イベント文言}`
   - 「発生イベント」ラベルは削除
   - イベント文言はヘッダーに合わせる:

| triggerAction | イベント文言 |
|---|---|
| `"キャンセル"` | キャンセル |
| `"キャンセル（当日補充）"` | 当日キャンセル |
| `"降格"` | 管理者操作 |
| `"オファー辞退"` | オファー辞退 |
| `"キャンセル待ち辞退"` | キャンセル待ち辞退 |
| `"オファー期限切れ"` | オファー辞退（期限切れ） |

4. **キャンセル待ち列**（「繰り上げオファー」セクションは削除）
   - ラベル: 「キャンセル待ち列」
   - 全試合でキャンセル待ちメンバーが同一の場合:
     - ラベルに「（※キャンセル待ちのメンバーは全試合で同一）」を付記
     - 1つにまとめて表示
   - 試合ごとに異なる場合:
     - 試合ごとにセクション分け（「1試合目:」「3試合目:」等）
   - 表示形式:
     - 1番の人: `1番: {名前}（オファー応答待ち）` — 緑色 `#27AE60` 太字（オファーが送られている場合のみ）
     - 1番の人にオファーが送られていない場合: 通常表示（緑太字なし、「オファー応答待ち」なし）
     - 2番以降: `{番号}番: {名前}` — 通常表示
   - キャンセル待ち列が空の場合: 「なし」

#### altText（通知プレビュー）

`【管理者通知】〇月〇日: {選手名}が{イベント文言}`

### 3.3 既存の通知条件は維持

- キャンセル時: WAITLISTEDがいて繰り上げが発生した場合のみ通知
- キャンセル待ち辞退時: 繰り上げの有無に関係なく常に通知
- その他（降格、オファー辞退、オファー期限切れ）: 現行通り

## 4. 技術設計

### 4.1 API設計

API変更なし（LINE通知のみの改修）。

### 4.2 DB設計

DB変更なし。

### 4.3 バックエンド設計

#### 方針: 呼び出し元で蓄積してからまとめ送信（案A）

#### 変更対象クラス

**`WaitlistPromotionService`**
- `notifyAdminsAboutWaitlistChange` のシグネチャを変更し、複数試合分の情報をまとめて受け取れるようにする
- 各呼び出し元（`cancelParticipation`, `demoteToWaitlist`, `declineWaitlistBySession`, `respondToOffer`, `expireOffer`）で、試合番号と繰り上げ結果をリストに蓄積してからまとめて通知メソッドを呼ぶ
- 単一試合のケース（`respondToOffer`, `expireOffer`, `demoteToWaitlist` 等）はリストに1件だけ入れて呼ぶ（共通インターフェース）

**`LineNotificationService`**
- `sendAdminWaitlistNotification` のシグネチャを変更: `int matchNumber` → `List<Integer> matchNumbers` + 各試合のキャンセル待ち列情報
- `buildAdminWaitlistFlex` を改修: 新しいFlex構成に合わせて再構築
  - セッション情報の専用フォーマット（`〇月〇日（会場名：定員〇〇名）`）
  - 複数試合の列挙
  - ヘッダー・イベント文言のトリガー別切り替え
  - キャンセル待ち列の全試合同一判定 & 表示分岐
  - 1番の人のオファー状態による表示切り替え
- `sendWaitlistPositionUpdateNotifications` も同様に改修（同じFlexを使用）

#### 1番の人のオファー状態の判定

現在の `promoteNextWaitlisted` は繰り上げ対象をOFFERED状態に変更した後に `PracticeParticipant` を返す。通知構築時にWAITLISTED一覧を取得する際、1番の人がOFFERED状態であれば「オファー応答待ち」表示にする。

具体的には、通知メソッドに渡す情報として各試合の `offeredPlayer`（繰り上げオファー先、`null` の場合はオファーなし）を保持し、キャンセル待ち列の1番表示時にこの情報を参照する。

### 4.4 フロントエンド設計

変更なし。

## 5. 影響範囲

### 変更が必要な既存ファイル
- `WaitlistPromotionService.java` — 通知呼び出しのバッチ化、`notifyAdminsAboutWaitlistChange` の改修
- `LineNotificationService.java` — `sendAdminWaitlistNotification`, `buildAdminWaitlistFlex`, `sendWaitlistPositionUpdateNotifications` の改修

### 既存機能への影響
- 通知の送信条件は変更しない（キャンセル時のWAITLISTEDチェック等は維持）
- `getSessionLabel` は変更しない（他の通知で使用されているため）
- 他の通知メソッド（`sendWaitlistOfferNotification`, `sendSameDayCancelNotification` 等）には影響なし
- WAITLISTEDユーザー向け通知 (`sendWaitlistPositionUpdateNotifications`) も同じ新Flexを使用

### テストへの影響
- `LineNotificationServiceAdminTest` — シグネチャ変更に伴う修正が必要
- `WaitlistPromotionServiceTest` — 通知呼び出しのverify部分の修正が必要
- `LineNotificationRoutingTest` — 必要に応じて修正

## 6. 設計判断の根拠

- **案A（呼び出し元で蓄積）を採用**: バッファリング方式（案B）よりシンプルで、flushタイミング管理の複雑さを避けられる。まとめが必要な呼び出し元は限られているため、呼び出し元の改修コストは許容範囲。
- **セッション情報の専用フォーマット**: `getSessionLabel` を変更すると他の通知（20箇所以上で使用）に影響が及ぶため、この通知専用のフォーマットとした。
- **トリガー種別ごとに別メッセージ**: ヘッダーがトリガー別に変わるため、混在すると分かりにくくなる。
- **既存の通知条件は維持**: 挙動の変更は最小限に留め、Flex Messageの構成・文言の改修に集中する。
