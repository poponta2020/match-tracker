# 対戦組み合わせ編集のキャンセル者アラート＆保存エラー明示（pairing-cancel-alert-and-save-errors）要件定義書（ドラフト・改修）

> **改修モード**。対象は既存画面 `/pairings`（[PairingGenerator.jsx](../../../karuta-tracker-ui/src/pages/pairings/PairingGenerator.jsx)）。
> 既存機能 [pairing-cancelled-opponent](../pairing-cancelled-opponent/requirements.md)（キャンセルの read-time 反映）を踏まえた**差分**として定義する。

## 1. 概要

### 目的
対戦組み合わせ編集画面（`/pairings`）の「詰まりやすい2点」を UX ライティングの観点から改善する。

1. **キャンセル者アラート**: 作成済みの組み合わせにキャンセルした参加者が残っている状態で画面を開いたとき、灰色の取消線チップで黙って示すのではなく、**能動的なアラート**で「誰が・どの試合で・キャンセルしているか」を伝え、ワンタップで組から外して整理できるようにする。
2. **保存エラーの明示**: 「確定して保存」が押せない理由が分かりづらい問題を解消する。ボタンは**基本的に常に押せる**ようにし、保存できない場合は**押下後に具体的な理由**（誰の対戦相手が未設定か・どうすればよいか）を明示する。

### 背景・動機
- 現状、キャンセル者は閲覧モードで取消線＋「キャンセル」タグで表示されるだけで、管理者が気付きにくく・外す操作（対戦編集）に自力で辿り着く必要がある。
- 現状、片方だけ埋まった組があると「確定して保存」ボタンが `disabled` になる（[pairingLockLogic.hasBlockingIncompletePair](../../../karuta-tracker-ui/src/pages/pairings/pairingLockLogic.js)）。**なぜ押せないのか画面に何も出ない**ため、原因不明のまま操作が止まる。

### 方式（最重要の設計判断）
- **read-time（非破壊）方式は維持**（[pairing-cancelled-opponent](../pairing-cancelled-opponent/requirements.md) を踏襲）。`match_pairings` は変更せず、スキーマ変更・migration・本番DB適用は**なし**。
- **純フロントエンド改修**。バックエンド（DTO・API・createBatch）は無改修。既存の純粋関数（`materializeCancelledSlots` / `hasBlockingIncompletePair` / `buildSaveRequests` / `hasNothingToSave`）を再利用する。

## 2. 変更後の挙動

### 2.1 【変更1】キャンセル者アラート

> 承認済みの設計判断（`AskUserQuestion` 2026-07-24）:
> - OK 押下＝**現在の試合を編集モードへ**（キャンセル者を「空き」に実体化）。
> - PLAYER・ADMIN で**同一挙動**（現状 `/pairings` はロール差が無く、PLAYER も編集・保存できる。ロール制御は新設しない）。

- **発火条件**: **閲覧モードで表示中の試合**にキャンセル済み参加者（`player1Cancelled` または `player2Cancelled` が true の組）が1件以上あるとき、アラートを表示する。
  - 画面初回表示時に、その時点で表示している試合が該当すれば発火。
  - 別の試合タブへ切り替えて閲覧したとき、その試合が該当し、かつ**まだアラートを確認していなければ**発火。
- **スコープは「表示中の試合」単位**（＝選択された OK 挙動「現在の試合を編集モードへ」と一致させる）。
  - ⚠️ 依頼文の例（「〇試合目と〇試合目に…」）は複数試合をまたいだ集約文言だったが、選ばれた「現在の試合を編集モードへ」方式では、**試合ごとに1回ずつ確認**する形になる（同一人物が複数試合をキャンセルしていれば、各試合を開いたときにそれぞれ確認が出る）。この点は承認時に確認する。
- **表示内容**: iOS のアラートビュー的な**モーダル**（単一 OK ボタンのみ・閉じる/キャンセルは出さない）。
  - 文言（案）: 「**{N}試合目に参加予定の{選手名}は練習をキャンセルしています。対戦組み合わせから外してよろしいですか？**」
  - 同じ試合に複数のキャンセル者がいれば全員を挙げる（例:「{A}・{B}は…」）。
  - 正確な最終コピーと複数人レイアウトは[デザインへの宿題](#デザインへの宿題--design-screen-pairing-cancel-alert-and-save-errors)。
- **OK 押下時の挙動**:
  - その試合を**編集モードに切り替える**（既存の `enterEditModeForExisting()` 相当＝`materializeCancelledSlots`）。キャンセル者のスロットは**「空き」**になる（両方キャンセルの組は行ごと除去。結果入力済みの組は結果を保持し対象外）。
  - 管理者はその場で別の選手をドラッグ配置して組み直せる。何もせず「確定して保存」した場合は、壊れた組（生存側 vs 空き）は削除され、生存側の選手は待機（参加者プール）に戻る（**データ消失なし**）。
  - OK 済みの試合では**同一画面表示中（同一日付ロード中）はアラートを再表示しない**（編集モードでキャンセル→閲覧へ戻ったときの無限ループを防ぐ）。
- **PLAYER でも同挙動**: PLAYER も編集モードに入れて外せる（現状の実装能力どおり）。ロール分岐は設けない。

### 2.2 【変更2】保存ボタンを常に押せるように＋押下後エラー明示

> 承認済みの設計判断（`AskUserQuestion` 2026-07-24）: エラーは**未設定の選手名を挙げる**（複数なら全員列挙）。

- **ボタンの有効化**: 「確定して保存」ボタンから、片方だけ埋まった組による無効化（`hasBlockingIncompletePair`）を**外す**。ボタンは `loading`（保存処理中）を除き**常に押下可能**にする。
- **押下時バリデーション**: 保存を押したとき、**対戦相手が未設定の組**（片方だけ選手が入っている・キャンセル由来の空き `cancelledEmptied` を除く）があれば、保存を実行せずエラーを表示する。
  - 文言（案）: 「**{選手名}の対戦相手が未設定のため保存できません。対戦相手を選ぶか、待機中の枠に移動してください。**」
  - 未設定の選手が複数いれば**全員を列挙**する（例:「{A}・{B}の対戦相手が…」）。
  - 正確な最終コピー・表示位置は[デザインへの宿題](#デザインへの宿題--design-screen-pairing-cancel-alert-and-save-errors)。
- **キャンセル由来の空き組は対象外**: 変更1の OK で作られる「生存側 vs 空き」（`cancelledEmptied`）は未設定エラーの対象にしない。そのまま保存でき、壊れた組は削除・生存側は待機へ戻る（既存挙動）。
- **既存の他エラーは維持**: 「保存する組み合わせがありません」（`hasNothingToSave`）、サーバー起因の保存失敗メッセージ（`err.response.data.message`）は従来どおり。

## 3. 変わらないもの（Non-goals / 回帰対象）

- **キャンセル判定ロジック（read-time 方式・BE enrich）は変更しない**。`match_pairings` 非破壊・スキーマ変更なし・migration なし・BE 無改修。
- **LINE 送信用テキスト（[PairingSummary.jsx](../../../karuta-tracker-ui/src/pages/pairings/PairingSummary.jsx)）は対象外**（[pairing-cancelled-opponent](../pairing-cancelled-opponent/requirements.md) のスコープ外を踏襲。キャンセル者も従来どおりテキストに出る）。
- **複数試合を横断した一括自動保存（AskUserQuestion の選択肢B）は行わない**。OK は現在の試合の編集モード移行のみ。
- **ロール制御を新設・変更しない**。現状のプロトタイプ認証のまま。PLAYER の編集可否は現状維持。
- **自動組み合わせ（autoMatch）・再シャッフル・ロック・削除の各ロジックは無改修**。
- キャンセル者がいない通常の閲覧・編集・保存フローは従来どおり（アラート追加・ボタン有効化による退行なし）。

## 5. 技術的制約・契約

- **read-time 非破壊方式を維持**（`match_pairings` 不変・スキーマ変更なし・migration 不要・本番DB適用不要）。
- **純フロントエンド改修**。変更対象は `pages/pairings/PairingGenerator.jsx` と純粋ロジック（`pairingDisplayLogic.js` / `pairingLockLogic.js`）＋テスト。BE（DTO・controller・service・createBatch）は無改修。
- **既存純粋関数の再利用**: OK 挙動は `materializeCancelledSlots`（＝`enterEditModeForExisting`）を再利用。保存バリデーションは既存 `hasBlockingIncompletePair`（`cancelledEmptied` 除外を既に内包）を「disabled」ではなく「押下後エラー」に転用する。判定は本番・テスト共有の純粋関数として切り出す（既存方針踏襲）。
- **後方互換**: DTO・API 契約は不変。FE⇔BE のリクエスト/レスポンス形式に変更なし。
- ユーザー向け文言は日本語。UX ライティングの観点（誰が・何が問題か・次にどうすればよいか）を満たす。

## デザインへの宿題（→ /design-screen はユーザー判断でスキップ。実装時に既存パターンで解決）

- **キャンセル者アラートのモーダル**【解決】: 既存の選手追加モーダルと同じ styled modal（`fixed inset-0` オーバーレイ＋クリーム地カード）、単一 OK ボタン。複数キャンセル者は「・」区切りで列挙。
- **変更2 の保存エラーの表示位置**【解決】: 上部の共通 error バナーは編集エリア先頭にあり、組数が多いと保存ボタン（最下部）から見切れて「押しても無反応」に見える。→ 保存起因のフィードバック（未設定／保存対象なし／サーバー失敗）は専用 state `saveError` に分離し、**編集モードのフッター（保存ボタン直上）**に描画する。未設定選手名は複数なら「・」区切りで列挙。
- 現行のデザイン正典 [docs/design/design.md](../../design/design.md)（脱AIスロップ）を厳守。

## 6. 設計判断の根拠

| 判断 | 根拠 |
|------|------|
| **OK＝現在の試合を編集モードへ（自動保存しない）** | ①read-time 非破壊のため OK で保存しない限り再訪で再発火するが、選択肢OK（no cancel）は「破壊的な確定」ではなく「確認（acknowledge）」の性質。実際の削除は明示的な「確定して保存」で行うのが自然。②編集ロックは「一度に1試合」制約があり、全試合横断の一括操作は編集モードでは不可能。③既存 `enterEditModeForExisting`／`materializeCancelledSlots`／`cancelledEmptied` をそのまま再利用でき、変更2とも噛み合う。 |
| **アラートは試合単位** | OK の作用が現在の試合に閉じるため、メッセージ（スコープ）と作用（スコープ）を一致させ「OKしたのに他試合に残る」混乱を避ける。 |
| **保存ボタンを常時押下可＋押下後エラー** | disabled は「なぜ押せないか」を伝えられない。押下→具体的な理由（誰の相手が未設定か・どう直すか）を出す方が UX ライティングとして明快。既存 `hasBlockingIncompletePair` の判定（`cancelledEmptied` 除外込み）を disabled から click バリデーションへ転用するだけで実現でき、退行リスクが小さい。 |
| **ロール分岐なし** | 現状 `/pairings` は FE にロール判定が無く、BE も `createBatch` 等で PLAYER を許可（プロトタイプ認証）。ユーザーも「player と admin で同じでよい」と明示。既存の実態に合わせる。 |
