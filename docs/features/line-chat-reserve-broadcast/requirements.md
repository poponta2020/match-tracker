# LINEチャット予約送信による札分け全体配信（line-chat-reserve-broadcast）要件定義書

## 1. 概要

- **目的**: 札分けの全体LINEグループ配信を、Messaging API push（通数課金）ではなく **LINE Official Account Manager の「チャット予約送信」（無料メッセージ通数の対象外・通数無制限）** で行う。実送信はLINE側の予約機構が担い、アプリ＋常駐ワーカーは「事前の予約登録」だけを行う。
- **背景・動機**:
  - グループpushは人数分課金（70名グループ≒70通/回）で、無料枠200通/月では月2回しか配信できない。
  - 前作 [[card-division-group-broadcast]]（PR #1083・マージ済み）の「bot 10体ローテで枠を吸収する」作戦は、**LINEの仕様上グループに参加できる公式アカウントは1体まで**であることが実装後に判明し不成立（seed 未投入のまま）。
  - 代替として、グループ内の1体のOAが使える「チャット」は通数対象外・無制限であることに着目。チャットの予約送信を Playwright による管理画面自動操作で登録する（元ネタ: kagetra_new/docs/line送信作戦.txt。技術構成は本要件で再設計済み）。
- **実行基盤（作戦書からの主要変更点）**: GitHub Actions ではなく、**Oracle Cloud VM（東京・kagetra_new 稼働ホスト）上の常駐ワーカー**で実行する。固定国内IP＋永続ブラウザプロファイルによりセッション失効リスク（本方式の最大リスク）を最小化し、認証状態のSecrets往復・本文のGitHub露出も構造的に排除する。
- **対象スコープ**: v1 は**札分け全体配信の配信手段差し替えのみ**。予約キュー（本文＋送信日時＋対象グループ）自体は汎用に設計し、将来の任意メッセージ配信への拡張余地を残す（管理UIからの任意登録は Non-goal）。

## 2. ユーザーストーリー

- **配信の受信者** = 団体の全体LINEグループのメンバー全員（約70名）。練習当日、1試合目開始30分前（開始時刻不明なら朝8:00）に札組がグループへ流れてくる。**受信者から見た体験は前作と同一**（送信者が単一の「お知らせ」OAになる分、むしろ自然）。
- **設定・運用者** = 管理者（ADMIN / SUPER_ADMIN）。管理画面で予約状況（登録済み/失敗/要確認）を確認し、失敗時のアラートを受けて再試行または手動対応する。LINEログイン失効時は再ログイン手順（runbook）で認証状態を更新する。
- **目的**: 無料枠の制約を受けずに毎練習日の全体配信を成立させる。月15回前後の練習をすべてカバーする（push方式では月2回が上限だった）。

## 3. 機能要件

### 3.1 画面と遷移

- **管理画面「全体LINE配信管理」（既存・PR #1083）に予約状況セクションを追加**（ADMIN+、既存 /admin/line 系パターン踏襲・design-spec なし）:
  - 予約一覧: 対象セッション・送信予定時刻・状態（pending / reserving / reserved / failed / manual_review_required / cancelled / fallback_sent）・エラーコード・最終更新。
  - アラート表示: 失敗・要確認（manual_review_required）・LINEログイン失効。
  - 手動再試行: failed の予約を pending に戻す（送信予定時刻まで余裕がある場合のみ）。
- 既存の bot ローテ関連 UI（残枠バー・次配信bot等）は**bot 1体構成でもそのまま表示可能なため変更しない**（表示が冗長になる程度）。
- 受信者向け画面の変更なし。

### 3.2 ビジネスルール

**予約キューの生成（アプリ側・自動）**
- **前日20:00（JST）の日次バッチ**で、有効な配信グループ × 翌日の当該団体セッションについて、(配信グループ, セッション) 単位の予約レコードを一意に作成する（冪等・重複作成しない）。
- **配信は練習会（セッション）ごとに独立**。同日に同一団体が複数セッションを持つ場合、それぞれの会場・試合数に応じた本文と送信時刻で別々に予約・配信される。
- 本文は**個人通知と完全同一の札分けテキスト**（`CardDivisionTextService.buildTextForSession(session)`）。生成ロジックは一切変更しない。
- 送信予定時刻は既存ルールを踏襲: **1試合目開始時刻（`venue_match_schedules` match_number=1 → 無ければ `PracticeSession.startTime`）の30分前**、いずれも無ければ**朝8:00**。さらに**10分境界へ切り捨てる**（早める方向・遅らせない）。「30分前」は元の分を保つため 1試合目 :15/:45 開始だと非境界（例 09:45）になり得るが、LINE OAM の予約時刻入力が10分単位（`step=600`）でしか受け付けず非境界値を無言でスナップする（実測 09:45→09:50・phase2-dom-findings §4）ため、BE側（`CardDivisionScheduleResolver.resolveScheduledSendAt`）で先に境界へ寄せる。push配信のウィンドウ判定は生の開始時刻を使うため不変。
- セッションが前日20:00より後に新規作成されることはない（ユーザー確認・2026-07-17）ため、**補完チェックは設けない**。万一予約が存在しないセッションがあっても、予約未作成＝フォールバックpush経路が拾う構造のため配信は欠落しない。

**予約の登録（ワーカー・自動）**
- 常駐ワーカーが数分間隔でアプリAPIをポーリングし、pending の予約を1件ずつ処理する:
  1. 対象グループのチャットルームを開き、**グループ識別情報（名称＋初回登録時に保存した識別情報）を検証**。不一致なら TARGET_CHAT_MISMATCH で停止。
  2. 既存の同一予約（同一日時・同一本文冒頭）がないか確認。あれば DUPLICATE_RESERVATION_FOUND として manual_review_required。
  3. 本文入力 → 予約日時設定 → 予約確定 → **チャット画面に「送信予定」が表示され、日時・本文が一致することを検証**して初めて reserved を報告する。予約ボタン押下だけでは成功と判定しない。
  4. 予約確定後の結果不明（タイムアウト等）は**自動再試行せず** manual_review_required。
- 実送信はLINE側の予約機構が行う（ワーカーは送信時刻に何もしない）。

**変更・取消**
- 予約済みセッションの削除・会場変更・時刻変更をアプリが検知したら、予約を cancel_pending にする。ワーカーがLINE側の予約を削除し（一覧から消えたことを確認）、変更の場合は新しい内容で再予約する（**「編集」はサポートせず取消→再予約に正規化**）。
- 旧予約を特定できない場合は manual_review_required（勝手に追加予約しない）。

**フォールバック（同一OAの自動push・既存経路の再利用）**
- グループに参加する1体のOAは、チャット予約とMessaging API pushの両方を使える。既存の `CardDivisionBroadcastScheduler`（30分前/8:00ウィンドウ・枠ゲート200通・dedupe）を**フォールバック経路として存置**し、判定を1つ追加する:
  - 予約が **reserved** → pushしない（LINE側が送信する。二重配信防止）。
  - 予約が **pending / failed / 未作成** → 既存どおりpush（枠ゲート内。月200通≒70名グループへ月2回分が上限）。
  - 予約が **manual_review_required** → **pushしない**（予約が生きている可能性があり二重配信リスクがあるため）。管理者通知のみ。
- ワーカー自体が停止していても、この経路により「単一botの枠内で配信＋アラート」まで自動で劣化する（全滅しない）。

**セッション失効・異常時（ワーカー）**
- ログイン画面・QR/二段階認証・CAPTCHA・本人確認画面を検出したら操作を**即中止**し、LINE_AUTH_EXPIRED として報告・管理者通知。**認証の自動突破・CAPTCHA回避は行わない**。
- 再ログインは手動運用: ローカルPCで headed ログイン → storageState を VM の永続ボリュームへ配置（runbook 化）。
- 失敗時はスクリーンショット＋Playwright TraceをVMボリュームに保存（保持期間つき）。アプリへはステータス・エラーコードのみ送る（画像・本文をログに出さない）。

**dry-run**
- ワーカーに dry-run モードを持つ: 予約確定ボタンを**押さない**ところまで全手順を実行し、入力済み画面のスクリーンショットを保存して dry_run_succeeded を報告する。テストグループ＋dry-runで本番投入前の検証を行う。

**通知**
- 失敗・要確認・ログイン失効・フォールバック発動は、管理画面アラート＋既存のベストエフォート管理者LINE通知（`ADMIN_` プレフィックス種別を新設）で知らせる。

**段階ゲート（実装順序の絶対条件）**
1. **Phase 1（手動・コード着手前のGO/NO-GO）**: テストグループで手動のチャット予約送信を実施し、(a) グループチャットでも予約送信が使えること、(b) 「利用と請求」で無料メッセージ通数が増えないこと、を実測確認する。**NGなら本機能は中止**（その場合の代替は要再検討）。
2. **Phase 2（ローカルPoC）**: ローカルPCの Playwright でテストグループへの予約登録→検証→削除を自動化できることを確認。
3. **Phase 3（VMデプロイ＋本番投入）**: VM上でワーカーを起動し、(a) サービストークン疎通（トークン無し=401／不正=403／正当=200 を実測で判別）と (b) **VMのIPから**テストグループへの dry-run 成功（`LINE_AUTH_EXPIRED` でない）を確認してから本番グループへ投入する。**2週間の事前観測は行わない**（2026-07-18 ユーザー判断・下記「設計判断の根拠」参照）。セッション寿命は本番運用の中で観測し、失効時は検知→通知→再ログイン運用と既存pushフォールバックで劣化を吸収する。本番投入後の**最初の1〜2回は送信前に人が予約内容を目視確認**する。

## 5. Non-goals（今回やらないこと）

- **管理画面からの任意メッセージ予約登録**（汎用配信基盤のUI）。キュー設計は汎用だがUIは作らない。
- **LINE側予約の「編集」操作**（取消→再予約に正規化。Page Object実装面積を半減させる）。
- **送信後の実送信確認**（sent_confirmed。予約成功の確認まで。実送信はLINE任せ）。
- **bot複数体のローテーション**（LINE仕様上グループ1体制約で不成立。前作の残存実装は削除せずbot 1体構成で流用）。
- **GitHub Actions を実行基盤にすること**（作戦書原案。IP変動によるセッション失効リスクのため不採用）。
- **CAPTCHA・二段階認証等の自動突破**（検出したら停止・通知。恒久的に対象外）。
- **kagetra_new との機能連携**（VMを間借りするだけ。プロセス・データは完全独立）。
- **LINEヤフーへの規約照会**（ユーザー判断でリスク受容・2026-07-17。専用OA運用で影響を限定）。
- **個人LINE通知（3時間前・購読制）の変更・廃止**（併存）。

## 6. 技術的制約・契約

- **再利用する既存資産**: 札分けテキスト＝`CardDivisionTextService`／送信時刻解決＝`CardDivisionBroadcastScheduler.resolveFirstMatchStartTime` 相当／フォールバックpush＝既存 `CardDivisionBroadcastService`＋枠ゲート／グループID捕捉＝既存 join Webhook／配信グループ管理＝`LineBroadcastGroup`（bot 1体割当で運用）。
- **変更禁止**: 札組生成ロジック（`cardRules.js`/`CardDivisionTextService`）、個人通知経路、既存管理APIの公開契約。既存push経路の変更は「予約状態による発火ガード追加」のみに留める。
- **OA**: 既存プールbot 1体を転用・リネームして全体グループに参加させる（`GROUP` 種別・配信グループ割当は既存管理UIで実施）。チャットON（応答メッセージはOFF）＋Webhook ON を併用。LINE Business ID の認証情報はアプリ・リポジトリ・VMのいずれにも保存しない（storageState のみVMボリュームに置く）。
- **ワーカー**: Node.js/TypeScript + Playwright(Chromium)。`line-chat-worker/` として match-tracker リポジトリに同居。VM（Oracle Cloud 東京・kagetra_new稼働ホスト）へ docker compose で手動デプロイ（runbook必須）。TZ=Asia/Tokyo。kagetra_new のコンテナ群とはネットワーク・ボリュームを共有しない。
- **ワーカー⇔アプリ間**: Render上のREST APIをポーリング（**DB直読みはしない**＝Render無料DBの月次ローテーションの影響を受けない）。認証は専用サービストークン（環境変数・十分長いランダム値）。本文・認証情報をワーカーのstdout/アプリログに出力しない。
- **冪等・排他**: (配信グループ, セッション) 単位の一意制約で予約レコードは1件。ワーカーは直列処理（並行実行しない）。結果不明時は再試行せず manual_review_required（作戦書 §9 の方針を踏襲）。
- **Playwright ロケーター方針**: getByRole/getByLabel/テキスト優先、自動生成クラス名依存禁止、複数一致はエラー（作戦書 §11 踏襲）。LINE管理画面固有操作は Page Object に集約し、実DOMは Phase 2 PoC で調査してから確定する（**識別子を推測で実装しない**）。
- **CI**: ワーカーは lint＋ユニットテスト（純ロジック・Page Objectモック）のみCI対象。実ブラウザでのLINE操作はCIで行わない。
- **ドキュメント更新（実装と同一コミット）**: `docs/spec/notifications.md`（予約経路・フォールバック条件）、`docs/design/db.md`（新テーブル）、`docs/SCREEN_LIST.md`（管理画面追記）、[[card-division-group-broadcast]] requirements への変更履歴追記（bot 10体ローテ不成立＝1グループ1bot制約、役割は単一botフォールバックに変更）、ワーカー運用runbook（初回ログイン・storageState更新・デプロイ・障害対応）。
- **未解決の技術論点（→技術計画で確定）**: 予約テーブルのスキーマ詳細／サービストークン認証の実装方式（フィルタ vs インターセプタ）／ワーカーのポーリング間隔と安全マージンの値／チャット画面での複数行テキスト入力の忠実性（Phase 2 PoC確認項目）／スクリーンショット・Traceの保持期間。

## 7. 設計判断の根拠

- **チャット予約送信の採用**: グループ1体制約でbotローテが不成立となり、無料で月15回配信できる手段はチャット（通数対象外）のみ。予約送信を使うことで「実送信時刻の正確性」をLINE側に委ね、自動化部分は事前登録に限定できる（実行基盤の遅延が送信品質に影響しない）。
- **VM常駐ワーカー（作戦書のGitHub Actionsから変更）**: 本方式の最大リスクはLINEセッション失効。実行毎にIPが変わる GitHub-hosted runner は失効を誘発する最悪の構成であり、固定国内IP＋永続プロファイルのVMが構造的に優位。認証状態のSecrets往復・本文のGitHub露出も消える。
- **API ポーリング（DB直読みしない）**: Render無料DBは月次ローテーションで接続情報が変わるため、ワーカーがDB資格情報を持つと毎月壊れる。アプリのURL＋サービストークンは安定。
- **フォールバックは同一OAのpush**: グループ内の1体は両方式を使えるため、前作のpush基盤（枠ゲート・dedupe・ウィンドウ）が「月2回分の保険」としてそのまま活きる。ワーカー全停止でも配信が全滅しない。manual_review_required でpushしないのは二重配信（70名への誤爆）が配信欠落より重大なため。
- **取消→再予約への正規化**: 編集フローの Page Object 実装・検証を丸ごと省略でき、UI変更リスクの面積も減る。
- **段階ゲート**: 成立性が「LINE仕様の実測」と「セッション寿命の実測」に依存するため、Phase 1（手動）→Phase 2（ローカル）→Phase 3（VM）を通過するまで本番グループに触れない。
- **2週間観測を廃し「smoke＋初回目視」に差し替え（2026-07-18）**: 当初の Phase 3 は2週間の事前観測を課していたが、これは**2つの別問題を混ぜていた**。①「VMのIPでスタック＋セッションが通るか」＝本質的リスクだが**数分で実測可能**（VMからの dry-run 1回）。②「セッションが2週間もつか」＝待つしかないが、失効しても**既存pushフォールバック＋`ADMIN_`アラート**が拾うため、静かな全滅ではなく「push枠制限の旧状態への劣化」に留まり、自己申告的に露見する。加えて本番は実練習日ごとにタスクが走るため**セッションは自然に叩かれる**（アイドル時はワーカーがLINEに触れず、観測が空回りする問題は"人工的な観測"にのみ存在した）。よって①をVM上のsmokeで潰し、②は本番運用で観測する。70名の本番グループへ未検証のまま流さないための担保は「テストグループdry-run→初回1〜2回の目視確認」で維持する。
- **専用OA（プールbot転用）**: 規約リスクの影響半径を「お知らせ配信」に限定。個人通知bot群・LINE Login とは分離。
