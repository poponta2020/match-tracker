# 試合番号スワイプ移動 要件定義書

## 1. 概要

### 目的
「その日の対戦結果一覧」「結果の一括入力」「個人の結果入力」の3画面で、画面上部の **試合番号タブ（1試合目 / 第N試合）** を、**左右スワイプ**でも切り替えられるようにする。

### 背景・動機
現状、試合番号の切り替えは上部タブのタップのみ。スマホで使う場面が多く、タブをいちいち狙ってタップするより、コンテンツ上を左右にスワイプして前後の試合へ移動できる方が速く直感的。タブUIは残したまま、スワイプを「追加の操作手段」として提供する。

## 2. ユーザーストーリー

- **対象ユーザー**: 練習の対戦記録を閲覧・入力する選手および管理者（主にスマホ利用）
- **ユーザーの目的**: 前後の試合番号へ素早く移動したい
- **利用シナリオ**:
  - 結果一覧画面で、各試合の結果を順に確認したいとき、スワイプで次々と試合を送る
  - 一括入力画面で、1試合目→2試合目→…と入力しながらスワイプで移動する
  - 個人入力画面で、自分の各試合の結果を入力する際にスワイプで試合を移動する
- **ゴール**: スマホ操作で、タブタップと同等以上に快適に試合番号を行き来できる

## 3. 機能要件

### 3.1 画面仕様

#### 共通仕様（3画面）
- スワイプの対象は **試合番号の切替**（= `currentMatchNumber` / `formData.matchNumber`）。**日付は対象外**（結果一覧の日付前後ボタンは現状維持）。
- **方向**: 左スワイプ（指を右→左）＝**次の試合番号（+1）**、右スワイプ（指を左→右）＝**前の試合番号（−1）**。
- **端の挙動**: 1試合目で「前へ」、最終試合で「次へ」はスワイプしても**何も起きない（端で止まる）**。ループや日付移動はしない。
- **既存タブUIは存続**。スワイプで試合が切り替わったら、上部タブのアクティブ表示も連動し、**アクティブタブが画面内に見えるよう自動スクロール**する。
- **対象は横スワイプのみ**。縦スクロール・タップと明確に区別する（誤発動防止）:
  - タッチ開始位置から横移動量が一定（約10px）を超え、かつ「横移動量 > 縦移動量」のときだけ横スワイプとして扱う。
  - 縦方向が優勢な場合は通常の縦スクロールとして扱い、スワイプは発動しない。
  - 移動量が小さい場合はタップとして扱う（ボタン・セレクト操作を阻害しない）。
- **対象デバイス**: タッチ操作（スマホ/タブレット）。PCのマウス操作は対象外（PCは従来どおりタブで切替）。
- `totalMatches`（その日の試合数）が1以下の場合、スワイプは実質無効（移動先がない）。

#### 結果一覧（MatchResultsView） … 指追従カルーセル
- 試合ごとの内容（対戦組ペアリング一覧・抜け番・下部の「結果を一括入力 / 対戦組み合わせを作成」ボタン）を **指の動きに追従**して左右にスライドさせ、隣の試合がチラ見えする。
- 指を離した時点で、移動量が **コンテナ幅の約25%（または素早いフリック）** を超えていれば隣の試合へスナップ確定。未満なら元の試合へ戻る。
- 日付前後ボタン・カレンダーピッカー・FAB（自分の結果を入力）はスワイプ対象外（従来どおり固定）。

#### 一括入力（BulkResultInput） … 指追従カルーセル
- 試合ごとの入力行（勝者選択・枚数差セレクト・抜け番活動）を指追従でスライド。挙動は結果一覧と同じ。
- **データは消えない**: 入力結果は全試合分を保持しているため、スワイプで試合を移動しても入力済み内容は保持される（タブ切替と同じ）。
- 下部固定の保存バー（保存する N件）はスワイプ対象外（全試合の変更件数を表すため固定のまま）。
- セレクトやボタンのタップはスワイプ判定と区別し、従来どおり操作可能とする。

#### 個人入力（MatchForm） … スライドイン＋警告
- **方式B（スライドイン）**: 指を離したときに移動量を判定し、確定したら新しい試合の内容が左右から **スッと入る（約0.2秒のトランジション）**。指追従はしない。
  - 理由: MatchFormは「1試合分のフォーム」しか保持しない構造のため指追従カルーセルが困難。また下記の警告は「確定前に割り込む」必要があり、確定前に隣を見せる指追従とは両立しないため。
- **入力途中の警告**: 入力途中（フォームに未保存の変更がある状態）で **スワイプまたはタブタップ** により試合番号を変えようとした場合、「入力中の内容は破棄されます。移動しますか？」と**確認ダイアログ**を表示する。
  - OK → 試合番号を切り替え（既存 `applyMatchData` で対象試合のデータを反映）＋（スワイプ起点なら）スライドインアニメ。
  - キャンセル → 移動しない（現在の試合に留まる）。
  - 未保存の変更がない場合は確認なしで即移動。
  - スワイプ・タブタップの**両経路を共通ガード**（後述 `requestMatchNumberChange`）に通す。
- **適用範囲**: 試合番号タブが機能する**新規入力フロー**でのみ有効（編集モードはタブ切替自体が現状機能しないため対象外）。

### 3.2 ビジネスルール

- スワイプによる移動は**フロントエンドの表示状態のみ**を変更する。サーバーへの保存・送信は行わない（結果の保存は従来の各保存ボタンのまま）。
- 結果一覧・一括入力では、スワイプで試合を移動しても入力中・取得済みデータは破棄しない。
- 個人入力(MatchForm)のみ、未保存変更がある状態で**スワイプ・タブタップのいずれでも**試合番号を移動しようとしたとき確認ダイアログを挟む。
- 端（最初/最後の試合）を超える方向のスワイプは無効。

## 4. 技術設計

> **バックエンド・DB・APIの変更は一切不要。フロントエンドのみの改修。**

### 4.1 API設計
- 変更なし（新規エンドポイントなし）。

### 4.2 DB設計
- 変更なし（スキーマ変更・マイグレーションなし）。

### 4.3 フロントエンド設計

#### 新規（共通ロジック / コンポーネント）
1. **`swipeGesture.js`（純粋関数モジュール）**
   - タッチ座標（開始・現在・終了）と設定（活性化閾値・確定閾値・方向）から、「横スワイプか」「方向（prev/next）」「確定するか」を判定する純粋関数群。
   - 既存の `byePlayersLogic.js` / `pairingDragLogic.js` と同様、UIから切り離して `swipeGesture.test.js` で単体テスト可能にする。

2. **`MatchCarousel.jsx`（指追従カルーセル：結果一覧・一括入力で共用）**
   - props: `totalMatches`, `currentMatchNumber`, `onChange(matchNumber)`, `renderPanel(matchNumber)`。
   - 役割: 試合番号パネルを横並びトラックとして描画（現在±1のウィンドウ描画）。touch イベントでドラッグ追従（translateX）、離したときにスナップ／戻し、端でクランプ、確定時に `onChange` を呼ぶ。
   - 高さは表示中パネルに合わせる（パネルごとの高さ差による余白を防ぐ）。
   - `swipeGesture.js` を利用。

3. **MatchForm用 スライドイン＋確認の仕組み**
   - 軽量なスライドイン用の状態（方向・キー）＋確認ダイアログ用の状態。指追従はしないため `MatchCarousel` は使わず、スワイプ検出（touchstart/move/end）→確定判定→（dirtyなら確認）→`formData.matchNumber` 更新＋アニメ。

#### 改修（既存3画面）
- **MatchResultsView.jsx**: 試合ごとの表示部（`currentPairings` の描画＋抜け番＋下部ボタン）を `renderPanel(matchNumber)` として切り出し、`MatchCarousel` でラップ。`onChange` で `setCurrentMatchNumber`。タブのアクティブ自動スクロールを追加。
- **BulkResultInput.jsx**: 試合ごとの入力部を `renderPanel(matchNumber)` に切り出し、`MatchCarousel` でラップ。`onChange` で `setCurrentMatchNumber`。
- **MatchForm.jsx**:
  - `isDirty` 状態を追加。ユーザー操作（結果ボタン・枚数差・お手付き・メモ・対戦相手選択・抜け番活動）で true、保存成功時と確認後の試合切替時に false。`applyMatchData` 自体では立てない。
  - 試合番号変更の共通ガード `requestMatchNumberChange(num, { fromSwipe })` を追加: `isDirty` なら確認ダイアログ→OKで `setFormData({matchNumber})`、キャンセルで据え置き。dirtyでなければ即変更。**既存タブタップのonClickとスワイプ確定の両方をこのガード経由にする**。スワイプ起点（`fromSwipe`）のときのみスライドインアニメを付ける。
  - タブのアクティブ自動スクロールを追加。

#### 閾値（初期値、実装時に微調整）
- 横スワイプ活性化: 横移動 約10px かつ 横 > 縦
- 確定: コンテナ幅の約25%以上（または素早いフリック）
- スライドイン時間: 約0.2秒

### 4.4 バックエンド設計
- 変更なし。

- 各画面のスワイプ挙動テスト（`*.swipe.test.jsx` 等）

### 既存機能への影響（デグレ注意点）
- **タブタップ**: 既存の `setCurrentMatchNumber` / `setFormData({matchNumber})` 経路はそのまま維持（スワイプはこれを呼ぶ追加手段）。
- **縦スクロール**: コンテンツが縦に長い画面で、スワイプ判定が縦スクロールを阻害しないこと（横優勢時のみ発動）。
- **タップ操作**: 一括入力のセレクト/勝者ボタン、結果一覧のリンク/動画バッジ、個人入力のフォーム操作がスワイプ判定で誤反応しないこと。
- **一括入力の保存バー / 結果一覧のFAB・日付ナビ**: 固定要素はカルーセルの外に置き、スライドの影響を受けないこと。
- **MatchFormの既存ロジック**（抜け番判定・既存試合上書き・参加登録ダイアログ・指導試合）に影響を与えないこと。
- ドキュメント更新: `docs/SPECIFICATION.md` / `docs/SCREEN_LIST.md` / `docs/DESIGN.md` の該当箇所を実装と同コミットで更新（CLAUDE.md ルール）。

## 6. 設計判断の根拠

- **スワイプ対象を「試合番号」に限定**: ユーザー要望が「前の試合／後ろの試合番号に移動」。結果一覧の日付移動は既存の前後ボタンがあり、役割を分離した方が混乱しない。
- **結果一覧・一括入力＝指追従／MatchForm＝スライドイン**: 指追従カルーセルは隣パネルの同時描画が前提。読み取り専用の結果一覧、全試合分の状態を持つ一括入力は対応しやすいが、1試合分のフォームしか持たないMatchFormは大改修＆高リスク。さらにMatchFormは「確定前の警告」が必要で、確定前に隣を見せる指追従とは両立しない。よってMatchFormはスライドイン＋警告とした。
- **端で止まる**: 最もシンプルで誤操作が少ない。ループや日付跨ぎは挙動が複雑で予期しない移動を生むため不採用。
- **共通化（swipeGesture.js / MatchCarousel）**: 既存コードベースが純粋ロジック切り出し＋単体テストを重視しているため、ジェスチャ判定を純粋関数化し、カルーセルを共通部品化して重複と回帰リスクを抑える。
