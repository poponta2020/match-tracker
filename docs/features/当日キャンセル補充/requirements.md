# 当日キャンセル補充 要件定義書

## 1. 概要

### 目的
練習当日にWON参加者がキャンセルした場合、空いた枠を迅速に埋めるための補充フローを実装する。

### 背景・動機
現在、当日キャンセルが発生しても繰り上げフローが発動せず、枠が空いたまま放置される。当日は従来のオファー方式（24h期限）では間に合わないため、全体募集＋先着順の即時補充方式に切り替える。

## 2. ユーザーストーリー

### 対象
北大かるた会（MONTHLY型）の全メンバー

### シナリオ
1. **練習参加者として**: 当日12:00に参加者が確定し、メンバーリストをLINEで受け取りたい
2. **練習参加者として**: 同じ試合のメンバーに変更があったら知りたい
3. **練習に参加しない人として**: 枠が空いたら参加のチャンスを得たい（LINEボタンで即応答）
4. **管理者として**: 当日のメンバー変動を把握したい

## 3. 機能要件

### 3.1 12:00確定フェーズ

#### 3.1.1 オファー強制失効
- 当日12:00（JST）に、その日のセッションでOFFERED状態の参加者を全てDECLINEDに変更する
- スケジューラ（cron）で自動実行

#### 3.1.2 参加者確定通知
- 12:00時点でWONの参加者に、試合ごとのメンバーリストをLINE通知
- **1通のFlex Message**に全試合分をまとめる
- 送信先: その日のセッションでWONの全参加者
- この通知はオンオフ可能（独立トグル）

### 3.2 当日キャンセル→補充フェーズ（12:00以降）

#### 3.2.1 キャンセル通知
- **トリガー:** WONの参加者が12:00以降にキャンセル
- **送信先:** そのセッションの全参加者（キャンセル本人除く）
- **内容:** 「〇〇が今日の第〇試合をキャンセルしました」

#### 3.2.2 空き募集通知
- **トリガー:** 3.2.1と同時
- **送信先:** そのセッションの非WONの参加者（キャンセル本人除く）
- **内容:** Flex Message「今日の第〇試合目に〇名分の空きがあります。参加しますか？」＋「参加する」ボタン
- 複数試合で空きがある場合は試合ごとに別メッセージ
- CANCELLEDステータスの人も対象（自分がキャンセルした別試合の空き募集は受け取れる）
- **応答期限:** 練習開始時間まで。期限後にボタンを押しても無効（エラー表示）。ただし期限切れの通知は送らない

#### 3.2.3 参加登録（ボタン押下時）
- **先着1名のみ**がWONに変更される
- 2人目以降は「先を越されました」エラーメッセージを表示
- ステータス遷移: 元のステータス（WAITLISTED, DECLINED, WAITLIST_DECLINED, CANCELLED等）→ WON
- 同じ練習日の別試合でWONの人も参加可能（複数試合参加OK）

#### 3.2.4 参加登録後の通知

**a. 既存WONメンバーへの通知**
- **送信先:** その試合でWONの人（参加登録した本人除く）
- **内容:** 「〇〇さんが今日の第〇試合に参加します」

**b. 枠状況通知**
- **送信先:** そのセッションの非WONの人（参加登録した本人除く）
- **内容（空きが残っている場合）:** Flex Message「今日の第〇試合目は残り〇枠です。参加しますか？」＋「参加する」ボタン
- **内容（枠が埋まった場合）:** Flex Message「今日の第〇試合目の枠が埋まりました」（ボタンなし）

### 3.5 12:00以降のアプリからの参加登録（定員未満の試合）

- 12:00時点で定員に満たない試合は、アプリから自由に参加登録可能（従来通り）
- 12:00以降にアプリ経由で参加登録された場合、その試合の既存WONメンバーに「〇〇さんが今日の第〇試合に参加します」通知を送信
- 空き募集通知は不要（そもそも定員未満で自由参加できるため）

### 3.3 フロントエンド（アプリ側）

#### 3.3.1 当日12:00以降のキャンセル確認ダイアログ
- 当日12:00以降に、その日の試合を含むキャンセル処理を確定しようとした場合に表示
- メッセージ: 「直前のキャンセルとなりますがよろしいですか？」（心理的圧を与える程度の軽い表現）
- 既存のわすらもち会（SAME_DAY型）の12:00以降確認ダイアログを北大かるた会にも適用する形

### 3.4 LINE通知設定トグル（新規）

以下を**全て独立したトグル**として追加:
| トグル名 | 説明 | デフォルト |
|---|---|---|
| 参加者確定通知 | 当日12:00のメンバーリスト | ON |
| 当日キャンセル通知 | 当日のキャンセル発生通知 | ON |
| 空き募集通知 | 当日の空き枠募集 | ON |

## 4. 技術設計

### 4.1 新規スケジューラ
- **SameDayConfirmationScheduler**: 毎日12:00 JST実行
  - 当日のセッションでOFFERED → DECLINED に強制変更
  - WON参加者に確定通知を送信

### 4.2 API設計

#### 新規エンドポイント
| メソッド | URL | 説明 |
|---|---|---|
| POST | `/api/lottery/same-day-join` | 空き募集への応答（LINEボタンのpostback経由） |

#### リクエスト
```json
{
  "sessionId": 123,
  "matchNumber": 2,
  "playerId": 456
}
```

#### レスポンス
- 成功: `200 OK` + WONステータス
- 先着負け: `409 Conflict` + エラーメッセージ
- 期限切れ: `400 Bad Request` + エラーメッセージ

### 4.3 LINE Flex Message設計

#### 参加者確定通知（Bubble）
- ヘッダー: 「本日の練習メンバー」
- ボディ: 試合ごとのメンバーリスト（第1試合: A, B, C / 第2試合: D, E, F ...）

#### 空き募集（Bubble）
- ヘッダー: 「空き枠のお知らせ」
- ボディ: 「今日の第〇試合目に〇名分の空きがあります」
- フッター: 「参加する」ボタン（postback: `action=same_day_join&sessionId=X&matchNumber=Y`）

#### 残り枠通知（Bubble）
- ヘッダー: 「空き枠のお知らせ」
- ボディ: 「今日の第〇試合目は残り〇枠です」
- フッター: 「参加する」ボタン（空きがある場合のみ）

#### 枠埋まり通知（Bubble）
- ヘッダー: 「空き枠のお知らせ」
- ボディ: 「今日の第〇試合目の枠が埋まりました」
- フッターなし

### 4.4 DB設計
- テーブル変更なし（既存のPracticeParticipant, LineNotificationPreferenceで対応）
- LineNotificationType enumに以下を追加:
  - `SAME_DAY_CONFIRMATION` （参加者確定通知）
  - `SAME_DAY_CANCEL` （当日キャンセル通知）
  - `SAME_DAY_VACANCY` （空き募集通知）
- LineNotificationPreferenceテーブルに対応するboolean列を追加

### 4.5 バックエンド設計

#### 新規クラス
- `SameDayConfirmationScheduler` — 12:00実行スケジューラ

#### 変更クラス
- `WaitlistPromotionService` — 当日12:00以降の分岐ロジック追加
- `LineNotificationService` — 新規Flex Message構築・送信メソッド追加
- `LotteryController` — `/same-day-join`エンドポイント追加
- `LineWebhookController` — postbackハンドラに`same_day_join`アクション追加
- `LineNotificationType` — 3つのenum値追加
- `LineNotificationPreference` — 3つのboolean列追加

### 4.6 既存フローとの境界

| 時間帯 | キャンセル時の挙動 |
|---|---|
| 12:00より前 | 従来通り: waitlistNumber順にOFFERED（24h or 当日23:59期限） |
| 12:00ちょうど | スケジューラがOFFERED全失効 + 確定通知送信 |
| 12:00以降 | 新フロー: 全体募集 + 先着ボタン方式 |

## 5. 影響範囲

### 変更が必要な既存ファイル
- `WaitlistPromotionService.java` — cancelParticipationの当日判定を12:00基準に変更
- `LineNotificationService.java` — 新規通知メソッド群追加
- `LineWebhookController.java` — postbackアクション追加
- `LotteryController.java` — エンドポイント追加
- `LineNotificationType.java` — enum追加
- `LineNotificationPreference.java` — 列追加
- `LotteryDeadlineHelper.java` — MONTHLY型にも12:00判定を適用
- フロントエンド: キャンセル確認ダイアログの条件拡張

### 既存機能への影響
- `OfferExpiryScheduler`: 12:00強制失効と競合しないよう制御が必要（12:00前に5分ポーリングで失効させないよう注意）
- 伝助同期: 当日補充でWONになった人も`dirty=true`で反映される（タイミング的に伝助側が間に合わない可能性あるが許容）
- 12:00以降はwaitlistNumberの意味がなくなるが、ステータスはWAITLISTEDのまま残す。12:00以降のロジックでwaitlistNumberに依存しないよう注意

## 6. 設計判断の根拠

| 判断 | 理由 |
|---|---|
| 全体募集＋先着方式（オファー順番なし） | 当日は時間的余裕がなく、順番にオファーしていては間に合わない |
| 12:00を境界にする | 北大かるた会の練習は午後が多く、12:00が「当日確定」の自然なタイミング |
| 通知トグルを全て独立 | 通知量が増えるため、ユーザーが細かく制御できるようにする |
| 先着負けはエラー表示のみ | 発生頻度が低いため、シンプルなエラー処理で十分 |
| CANCELLEDの人も空き募集対象 | 「やっぱり参加したい」ケースに対応。別のキャンセルで空いた枠に再登録可能 |
