# match-list-column-alignment 要件定義書

## 1. 概要

### 目的
対戦履歴一覧（MatchList）の各行を、行ごとに各要素のテキスト長で位置がガタつかないよう「列揃え」のテーブル風レイアウトに整える。同時に、勝敗（〇N/×N/△N）を「対戦相手名の右」に移動させ、会場・試合数を勝敗の右に置く。

### 背景・動機
- 現状の MatchList の行内表示は flex の `flex-1` や `max-w-[40%]` でテキスト幅に応じて伸縮するため、対戦相手名や会場名の長さに応じて勝敗・会場・お手付き等の x 座標が行ごとに動き、縦に視線を走らせたときに不揃いに感じられる。
- 直前の PR #759（[c4e83b3](../../../) `fix(matches): 対戦履歴一覧の行内表示順を変更し、試合数を「N試合目」表記に`）で行内順を一度入れ替えたが、列が揃わない問題は残っていた。
- ユーザーは「日付・名前・勝敗・会場名などが縦に見たときにずれてほしくない（左側がそろえばよい）」と要望。

---

## 2. ユーザーストーリー

### 対象ユーザー
- 競技かるた選手（自分の対戦履歴を一覧で振り返るプレイヤー）
- メンター／管理者（他選手の対戦履歴を一覧で確認する立場の人）

### ユーザーの目的
- 過去の対戦結果を一覧で素早くスキャンしたい。
- 各列が縦に揃っていることで「いつ・誰と・何勝何敗で・どの会場の何試合目だったか」を行間で比較しやすくしたい。

### 利用シナリオ
- 練習後に自分の MatchList を開き、当日の試合結果を上から順に眺めて勝敗の流れを把握する。
- 他選手のページに切り替えて、その選手の対戦履歴を上から下にスクロールしながら確認する。
- メンターが担当メンティーの試合を縦に確認し、特定の対戦の詳細にアクセスする。

## 3. 機能要件

### 3.1 画面仕様

#### 画面
- 対象画面: `/matches`（`MatchList.jsx`）の試合一覧セクションのみ。
- 対戦詳細（`MatchDetail.jsx`）、一括入力（`BulkResultInput.jsx`）、結果一覧（`MatchResultsView.jsx`）は対象外。

#### 1行のレイアウト
各行は以下の **6 列構造** とし、すべての行で各列の左端 x 座標が揃うように描画する。

| 列 | 内容 | 表示例 | 幅ポリシー |
|----|------|--------|-----------|
| 1 | 日付 | `5/23` | 固定幅（`M/D` が常に収まる幅） |
| 2 | 対戦相手名 | `山田太郎` | 可変幅。残スペース内で伸縮し、超過時は末尾を `…` でトランケート |
| 3 | 勝敗 | `〇3` / `×2` / `△1` | 固定幅（記号 + 1〜2桁の枚数差が常に収まる幅） |
| 4 | 会場 N試合目 | `さくら会場 3試合目`<br>会場不明時は `3試合目` | 可変幅。残スペース内で伸縮し、超過時は末尾を `…` でトランケート |
| 5 | メモアイコン（詳細遷移ボタン） | 📝（`StickyNote` アイコン） | 固定幅。非表示条件のときも幅は確保（列の左端を揃えるため） |
| 6 | お手付き回数 | `手2` | 固定幅。`null` のときも幅は確保 |

- 列 2（名前）と列 4（会場）の具体的な幅比は実装者裁量。モバイル幅（〜360px）でも 6 列が無理なく収まるようにする。レビュー時の見映えで微調整可能とする。

#### 列の揃え方
- 各行は CSS Grid によるテーブル風レイアウトで描画する。
- 列の左端 x 座標は行が変わっても同じになることを必須条件とする（ユーザー要望：「左側がそろえばよい」）。
- 列の右端や中央寄せは要件外（実装者裁量）。

#### 各列の挙動詳細

##### 列 1: 日付
- 形式: `M/D`（既存 `formatDate()` 関数をそのまま使う）。
- 表示色・サイズは現状維持（`text-xs text-[#9ca3af]`）。

##### 列 2: 対戦相手名
- 既存仕様を維持:
  - 通常の対戦相手は `<button>` でレンダリングし、タップで `/matches?playerId=<opponentId>` に遷移。
  - ゲスト選手（`opponentId === 0` または `null`）はリンクにならず `<span>`。
- リンク表示時は下線（既存）を維持。
- 超過時は `truncate`（`overflow: hidden; text-overflow: ellipsis; white-space: nowrap`）。

##### 列 3: 勝敗
- 形式は現状維持: `〇N` / `×N` / `△N`（既存 `getResultDisplay()` を流用）。
- 色は既存 `getResultColor()` を流用（勝ち=緑、負け=赤、引き分け=グレー、太字）。
- 表示位置を現状の「行の最右端」から「対戦相手名のすぐ右」に移動する。

##### 列 4: 会場 N試合目
- 形式は現状維持:
  - `venueName` がある場合: `{venueName} {matchNumber}試合目`
  - `venueName` が空の場合: `{matchNumber}試合目`
- 表示色・サイズは現状維持（`text-xs text-[#9ca3af]`）。
- 超過時は `truncate`。

##### 列 5: メモアイコン
- 表示条件: 既存仕様を維持（`showDetailButton && mentorCheckCompleted`）。
  - 自分閲覧時: 常に表示。
  - 他選手閲覧時: ACTIVE メンター関係がある場合のみ表示。
  - メンター関係 API ロード中・失敗時: 非表示。
- タップ動作: 既存仕様維持。`/matches/<id>` へ遷移。他選手閲覧時は `?playerId=<targetPlayerId>` 付き。
- 色分け: 既存仕様維持。`hasNote` で `text-gray-600`（メモあり）/`text-gray-300`（メモなし）。
- **列幅は条件で非表示の行でも確保する**（行間で列の左端を揃えるため）。

##### 列 6: お手付き回数
- 表示条件: `otetsukiCount != null`（既存仕様維持）。
- 表示形式: `手{otetsukiCount}`（既存維持）。
- 表示色・サイズは現状維持（`text-xs text-[#9ca3af]`）。
- **列幅は `null` の行でも確保する**。

### 3.2 ビジネスルール
- データ取得・並び順・フィルタ動作は変更しない（既存 `useEffect` ロジックそのまま）。
- 統計表示・FilterBottomSheet 等の周辺 UI も変更しない。
- 行のタップ動線は既存仕様を維持:
  - 行全体タップで詳細遷移はしない（廃止済み）。
  - 対戦相手名タップ = 相手の MatchList へ。
  - メモアイコンタップ = 対戦詳細へ。

### 3.3 エラー・境界条件
| ケース | 振る舞い |
|--------|---------|
| `opponentName` が長い | 列 2 内で `…` 省略 |
| `venueName` が長い | 列 4 内で `…` 省略 |
| `venueName` が空 | 列 4 は `N試合目` のみ表示 |
| `opponentId` が 0 または null（ゲスト） | 列 2 はリンクにならず `<span>` |
| メンター関係 API ロード中／失敗 | 列 5 は非表示（ただし列幅は確保） |
| `otetsukiCount` が null | 列 6 はテキスト非表示（ただし列幅は確保） |
| 試合 0 件 | 既存の空状態 UI を表示（変更なし） |

## 6. 設計判断の根拠

### 6.1 なぜ CSS Grid を使うのか
- 「列ごとに左端 x 座標を揃える」要件には、各列の幅をフレームレベルで決め打ちできる Grid が最も素直。
- 既存の flex + `flex-1`/`max-w-[40%]` は内容の長さで折り返し・縮小が起きるため、列の x 座標が行ごとに動く（=今回の問題の根本）。

### 6.2 なぜ勝敗を「名前のすぐ右」に移すのか
- ユーザー要望: 「日付、名前、勝敗、その右に会場〇試合目」。
- 視線の流れ（日付→誰と→どうだった→どこで何戦目）が左から自然に読めるため、結果の把握速度が上がる。
- 直前の PR #759 で「勝敗を最右端に」一度変更されていたが、列揃え要件と合わせて再配置する。

### 6.3 なぜメモ・お手付きが非表示の行でも列幅を確保するのか
- 非表示にして詰めると、行ごとに後続列の x 座標が動いてしまい「列揃え」要件が崩れる。
- 表示・非表示は同一画面内では全行で揃う条件（メモアイコン）またはデータに依存（お手付き）なので、空のセルがあっても視覚的に違和感は少ない見込み。

### 6.4 なぜ MatchDetail / BulkResultInput / MatchResultsView は対象外なのか
- ユーザーから「対戦履歴一覧（MatchList）だけ」と明示された。
- 他画面は性質が異なる（詳細表示・入力フォーム・別ビュー）ため、共通の「列揃え」要件は当てはまらない。

### 6.5 名前列・会場列の幅比を実装者裁量にする理由
- モバイル中心の利用想定で、画面幅・実データ分布によって最適比が変わる。
- ユーザーから「実装者裁量で適切に」の指示を受領済み。レビューで実機の見映えを確認しつつ調整する想定。
