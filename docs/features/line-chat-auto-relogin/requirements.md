# line-chat-auto-relogin 要件定義書

## 1. 概要
- **目的**: LINEチャット予約ワーカーが chat.line.biz の「ログインから絶対24時間」セッション失効で停止する問題を解消する。**24時間セッションはワーカーが無人のクリックスルー再ログインで自己回復**させ、**30日SSOの手動更新は期限アンカー型の先回り通知で人に促す**という二層運用で、練習毎（週2回）の手動再ログインを不要にする。
- **背景・動機（実測）**: 2026-07-20 の本番VM storageState での実測により、
  1. `__is_login_sso`/`__Host-SID`（30日有効）が生きていれば **「LINE account」→「Log in」の2クリック（パスワード無し・reCAPTCHA無し）で新 `__Host-chat-ses` が再発行される**（クリックスルー成立）。
  2. ただし **そのクリックスルーは30日SSO Cookie を1秒も延長しない（`SSO_ABSOLUTE`・実測 DELTA=0.00日）**。30日SSOは前回“手動フルログイン”からの絶対期限で、自動化では延ばせない（延長にはパスワード＋CAPTCHAのフルログインが必要＝安全制約上も自動化しない）。
  - ゆえに **自動化で回せるのは24時間セッションのみ**。30日SSOは約月1回、人が手動ログインする必要がある。詳細は `.claude/memory/impl_line_chat_reserve_broadcast.md` の「【突破口】…」「【SSO_ABSOLUTE 確定】…」節。

## 2. ユーザーストーリー
- **運用者（管理者）** として、練習毎の手動再ログイン無しで札分け全体配信が自動で届いてほしい。手動作業は約月1回のSSO更新ログインだけにしたい。
- SSO更新が必要になる**前に**「そろそろ手動ログインを」と気付けるようにして、配信失敗や有料フォールバックを未然に防ぎたい。

## 3. 機能要件

### 3.1 画面と遷移
- **UI なし**。VM常駐ワーカーのバックエンド挙動＋管理者への LINE 通知のみ。管理画面・フロントの変更なし。

### 3.2 ビジネスルール

**A. 24時間セッションの自動クリックスルー再ログイン（遅延・オンデマンド）**
- **トリガー**: 予約/取消タスクの処理中に `detectAuthWall` が壁（`OK` 以外）を返したサイクルで**のみ**再ログインを試みる。定期ウォームアップはしない。
- **再ログイン手順**（同一 browser context 内・メモリ保持の30日SSO Cookie を利用）:
  1. アカウントルート `https://chat.line.biz/<accountPath>` へナビゲート
  2. **壁の種別を host で判定（最重要）**: `detectAuthWall` は「認証面リダイレクト（`account.line.biz`/`access.line.me`）」**または**「`editorMissingAfterOpen`（SPA遅延・一時的エラー等・**セッションは有効**）」の2経路で壁を返す。ナビ後に **`isOnAuthSurface()`（host ベース）で分岐**する:
     - **chat.line.biz に居る（＝認証面でない・transient wall）** → 何もクリックせず **`SUCCEEDED`**（セッションは有効。リトライで room を開き直せば通る）。
     - **認証面に居るときのみ** 以下 3〜4 を実行する。
  3. `account.line.biz/login` の方式選択で **「LINE account」をクリック**
  4. `access.line.me` の「Continue as …／Log in」で **「Log in」をクリック** → `chat.line.biz` へ帰着＝新 `__Host-chat-ses` 再発行を積極確認 → `SUCCEEDED`
- **成功時**: 当該タスクを **1回だけリトライ**（`openChat` からやり直し）→ 通常どおり予約/取消を完遂。フォールバックpushは発火しない。
- **失敗時**（**認証面に居る**状態で手順3/4の**パスワード欄・reCAPTCHAチャレンジが出現**、または期待ボタン不在＝**SSO失効**）: 一切入力・突破せず**即失敗扱い**で中止。既存の `FAILED`/`LINE_AUTH_EXPIRED` 挙動を維持（＝**既存フォールバックpushが発火し配信は届く**＝安全網）。
- **重要な区別**: 「期待ボタン不在＝SSO失効」と判定してよいのは **認証面に居るときだけ**。`editorMissingAfterOpen` 由来の transient wall（chat.line.biz上）でボタンが無いのを SSO失効と誤判定すると、健全なセッションで誤ってフォールバックpush（67通課金）＋偽アラートを誘発する（AC-13）。
- **安全境界（最重要）**: パスワード欄には**絶対に入力しない**・reCAPTCHAチャレンジは**操作しない**（検出したら即失敗）。クリックするのは「LINE account」「Log in」の2ボタンのみ。
- **レート制御 / ループ防止**: 1サイクルにつき再ログインは**最大1回**。失敗サイクルは以降のタスクを処理せず打ち切る（既存 `abortCycle` 準拠）。成功後リトライも当該タスク**1回まで**。
- **冪等・非破壊**: 再ログインは予約アクションの**前**（壁検出＝まだチャット未到達）に走るため進行中の予約に影響しない。storageState は**既存のサイクル末尾書き戻し**で自動更新（新規ファイル処理を足さない）。

**B. 30日SSOの期限アンカー型 先回り通知**
- ワーカーは自身の in-memory context が保持する **`__is_login_sso` の失効日時を各サイクルで参照**する。
- **失効の N 日前（既定 3 日・env で可変）以内**になったら、アプリのサービスAPI経由で**管理者へ「手動再ログインが必要（SSO期限接近）」を LINE 通知**する。
- **間引き**: 同一“接近ウィンドウ”内は **1日1回まで**（5分ポーリングで多重送信しない。ワーカー側の in-memory throttle）。
- **自己追従**: 通知は常に**実際の SSO Cookie 失効日時**を根拠にする。人が手動ログイン（→ RUNBOOK どおりワーカー restart）すると、新しい storageState の新しいSSO期限を読み直し、**通知状態は自然にリセット**される（固定周期cronのようなズレが出ない）。

## 5. Non-goals（今回やらないこと）
- **PR #1102**（予約有無を DOM→予約一覧API で判定）の統合 — 別PR（1PR=1機能）
- 複数OAの再ログイン（v1は単一OA）
- **30日SSO天井の自動化撤廃**（LINE側の外部制約。約月1回の手動ログインは残す。自動延長は実測で不可＝`SSO_ABSOLUTE`）
- パスワード自動投入・CAPTCHA自動突破（安全制約上も恒久的に行わない）
- 予約検出のレース根治・送信時刻floor 等の既出改修（別PRで対応済/対応予定）
- 先回り通知の固定周期cron方式（期限アンカー型を採用するため）

## 6. 技術的制約・契約
- **壊してはいけない既存挙動**: 通常時（セッション有効）の予約/取消/dry-run/重複検出/`verifyScheduledEntry`/サイクル打ち切り、storageState のサイクル末尾書き戻し、`detectAuthWall` の positive判定（初回 about:blank 誤爆しない性質）、SSO失効時の既存フォールバックpush。
- **変更領域（2レイヤー）**:
  - `line-chat-worker`: (A) 認証壁時のクリックスルー再ログイン機構＋orchestration のリトライ判定、(B) SSO期限監視＋先回り通知の発火（in-memory throttle）。
  - `karuta-tracker`: 先回り通知を管理者へ届けるための**サービスAPIエンドポイント新設**（worker の既存サービストークン認証を踏襲）＋既存の管理者向け LINE push ヘルパーの再利用。**DB/entity スキーマ変更は不要を目標**（throttleはワーカー in-memory・通知はリレーのみ）。もし必要と判明したら停止・報告。
- **公開契約**: worker→app API（`reportResult` 等）の既存契約は不変。通知用エンドポイントは新設（後方互換）。
- **設定（env）**: 先回り通知の閾値 `SSO_WARNING_THRESHOLD_DAYS`（既定 3）。任意でキルスイッチ `AUTO_RELOGIN_ENABLED`（既定 true）を検討（Step4判断）。
- **安全境界**: ワーカーはパスワード投入・CAPTCHA突破を行わない（検出＝即失敗）。秘匿値（Cookie/token/本文）はログに出さない（既存規約）。
- **ToS**: LINEログインの自動化はグレー領域。本人アカウント・既存の有効セッション（SSO）利用・パスワード非保存で低リスク。ユーザー承認済（本要件の前提）。
- **未解決の技術論点（Step4で解決）**: (a) 管理者アラート用エンドポイントの形（新規 vs 既存拡張）と既存 admin-push ヘルパーの特定。(b) 再ログイン flow を page-object メソッドにするか独立 usecase にするか（テスト容易性）。(c) SSO期限の読み取り箇所（in-memory context の cookies を各サイクルで参照）。

## 7. 設計判断の根拠
- **二層運用の分離**: 24hは自動化可能・30日SSOは自動化不可（`SSO_ABSOLUTE` 実測）という物理的事実に合わせ、前者を自動クリックスルー、後者を人への先回り通知に分ける。
- **遅延トリガー（24h側）**: LINEログイン回数を最小化しbot検知リスクを下げる。既存の「壁検出→中止」フローへの統合が自然。
- **同context内メモリSSO利用**: 単一context再利用＋サイクル末尾書き戻しという既存設計に乗るだけで、ファイル更新/restartレースを構造的に回避。
- **期限アンカー型 先回り通知（30日側）**: 固定28日cronは人の周期外ログインでズレて早すぎ/遅すぎ・取りこぼしが起きる。実際のSSO Cookie失効日時を根拠にすれば自己追従し、有料フォールバックも配信欠落も未然に防げる。
- **既存フォールバックは触らず安全網として残す**: 先回り通知を人が見逃しても配信は落とさない（実装最小・二重の安全）。
- **2ボタンのみ・password/CAPTCHA不介入**: 安全制約を遵守しつつ、SSO失効を「失敗」として明確に検出できる（堅牢な分岐）。
