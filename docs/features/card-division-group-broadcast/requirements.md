# 札分けの全体LINE一斉配信（card-division-group-broadcast）要件定義書

## 1. 概要

- **目的**: その日の練習の「札分け（各試合の札組ルール）」を、団体の **全体LINEグループ**（1団体につき1グループ・約70名）に自動で一斉配信する。
- **背景・動機**: 既存の「個人LINE通知」（[[card-division-line-reminder]]・購読制・1試合目3時間前）は、各プレイヤーに専用botを1対1で割り当てて配信するため、購読していない/LINE未連携のメンバーには届かない。全体グループに流せば、当日の全メンバーが札組を確実に受け取れる。
- **制約の核（LINE無料枠）**: LINE公式アカウントの無料枠は **200通/月/チャネル**。グループ配信は「**トークルームの人数分**」を消費する（70名グループ＝約70通/回。ブロック済み/存在しないユーザーは非カウント）。したがって **bot 1体で月2回まで**。これを **未使用botを10体グループに投入し、枠を使い切ったら次のbotへローテーション**することで、月最大15回の練習を吸収する（10体×2回＝20回分 ≥ 15回）。**無料枠は超過しても課金されず超過分が届かないだけ**なので、ローテの最悪ケースは「配信の一部欠落」であり想定外の請求は発生しない。
- **対象スコープ（ユーザー確定・2026-07-16）**: 実装は**両団体（わすらもち会／北海道大学かるた会）に対応できる per-org 設計**（配信グループはN個持てる）とする。ただし**今回セットアップするのは北海道大学かるた会の全体グループ 1つのみ・bot 10体**を割り当てる。わすらもち会の全体グループは将来、同じ管理UIから追加する（データ登録のみ・コード改修不要）。

## 2. ユーザーストーリー

- **対象ユーザー**:
  - **配信の受信者** = 各団体の全体LINEグループのメンバー全員（購読・LINE連携の要否を問わない。グループに居れば届く）。
  - **設定・運用者** = 管理者（ADMIN / SUPER_ADMIN）。グループの登録・bot群の割当・稼働状況の確認を行う。
- **目的**: 受信者は当日の札組を練習前に把握・準備したい。運用者は「無料枠内で毎回きちんと全体配信が届いているか」を把握し、枠が尽きる前に手当てしたい。
- **利用シナリオ**:
  - （運用者・初回セットアップ）管理画面で団体の全体配信グループを作成 → 未使用botを10体割り当て → LINEアプリでその10体を全体グループに招待。以降は自動運用。
  - （受信者）練習当日、1試合目開始の30分前（開始時刻未設定なら朝8:00）に、全体グループへ札組が自動で流れてくる。
  - （運用者・日常）管理画面で「現在の稼働bot・各botの当月残枠・次に切り替わるbot・直近の配信結果」を確認。全botが枯渇したらアラートで気づく。

## 3. 機能要件

### 3.1 画面と遷移（画面インベントリ＋ナビゲーション地図。見た目は design-spec 参照）

- **管理画面：全体LINE配信管理（新規、ADMIN+。既存 `/admin/line` 系のLINEチャネル管理UIに隣接）**
  - **配信グループ一覧・登録**: 団体ごとに1つの配信グループを作成/編集/有効・無効。各グループの状態（グループID捕捉済みか・有効か・当日配信の有無）を表示。
  - **bot群（配信チャネルプール）の割当**: 未使用チャネル（`GROUP` 種別）をこのグループに割り当て/解除。各botの当月送信数・残枠・捕捉済みグループID・「グループ参加許可＋Webhook設定済みか」の状態を表示。
  - **稼働状況・ローテーション表示**: 「次に配信するbot（＝残枠が最大 or 未使用の先頭）」「各botの残枠バー」「今月の配信回数/残り可能回数」。
  - **配信ログ**: 直近の全体配信（日時・対象セッション・使用bot・受信数・結果）を一覧。
  - **枯渇アラート**: 全botが今月分を使い切り「これ以上配信できない」状態を画面上に警告表示（＋ログ、ベストエフォートで管理者へLINE通知）。
- 既存の個人向け「札分け確認」画面（`/settings/card-division`）・`/pairings/summary` は**変更しない**。

### 3.2 ビジネスルール（処理ルール・制約条件・エラーケース・例外処理）

**配信対象と本文**
- 配信は **団体（＝配信グループ）ごと**に独立して行う。ある団体の全体グループには、その団体の**当日セッション**の札分けテキストのみを流す（会場名・試合数は団体で異なるためセッション単位）。
- 本文は**個人通知と完全同一の札分けテキスト**（`CardDivisionTextService.buildTextForSession(session)`）。ヘッダ `M/D(曜) 会場名` ＋各試合の札組。**接頭辞・装飾・リード行は一切付けない**（プレーンテキスト。ユーザー確定＝個人通知と同じ・2026-07-16）。
- 札組生成ロジックは既存踏襲（日付シード決定論・団体非依存）。**このアルゴリズムは一切変更しない**。

**トリガー・タイミング（自動）**
- 各配信グループについて、その団体の当日セッションの **1試合目開始時刻の30分前**（JST）に自動配信する。
- 1試合目開始時刻は `venue_match_schedules`（会場×`match_number=1`）を第一情報源、無ければ `PracticeSession.startTime`。**いずれも無ければ、その日の朝8:00（JST）に配信する**（＝個人通知の「送信しない」とは異なり、全体配信はフォールバックして必ず流す）。
- 数分間隔のポーリングで、トリガー時刻に入った最初の実行で1回だけ配信する（二重送信は下記dedupeで収束）。
- **取りこぼし時の扱い（ユーザー確定・2026-07-16／既定案②）**: 各トリガーには**時間帯ウィンドウ**を設ける（30分前トリガー＝`[開始-30分, 開始)`／8:00フォールバック＝`[8:00, 正午]` 程度）。スケジューラのダウン等でこのウィンドウを過ぎた場合は**遅延配信せず**、その (グループ, セッション) を **「未配信」として管理画面に残す**だけにする（古い/場違いな時刻の全体配信を避ける）。ウィンドウ端の値は技術計画で確定。

**bot ローテーション（無料枠の吸収）**
- 配信グループに割り当てられた `GROUP` 種別botのうち、**当月の残枠が十分にある**（`当月送信数 + 想定受信数 ≤ 200`）botを選び、その1体でグループへpush配信する。
- 選択方針: 残枠のあるbotの中から選ぶ（原則「今使っているbotを使い切ってから次へ」＝ユーザー指定の『2回で次のbot』を、残枠しきい値で表現）。1回の配信で消費するのは**1体のみ**。
- **想定受信数**は実グループ人数に基づく（LINEのグループ人数取得APIまたは設定値。設計で確定）。当月送信数はLINE APIの実測（既存 `LineMessageCountSyncScheduler` が毎時同期）を基準にしつつ、1時間の同期ラグによる枠超過を避けるため**送信直後に消費分を即時反映**する。
- **全botが枯渇**（どのbotも `残枠 < 想定受信数`）なら、その回の配信は**スキップ**し、ログ＋管理画面アラート（＋ベストエフォートで管理者LINE）に記録する。**課金は発生しない**。

**二重送信防止（冪等）＋クラッシュ回復**
- 同一 **(配信グループ, セッション)** の全体配信は**1回のみ**（原子的な送信権確保）。ポーリングや再起動で複数回トリガーされても1回に収束する。
- 個人通知（`CARD_DIVISION_REMINDER`）の dedupe は player_id スコープのため流用不可。全体配信専用の送信ログ/dedupe機構を用いる（§6参照）。
- **送信権確保後・送信完了前にプロセスが落ちた場合の回復（必須）**: 予約（RESERVED）が残留すると次回ポーリングが「送信済み」とみなし、**その回が永久に未送信**になる（貴重な枠なので最悪の取りこぼし）。個人版の `releaseStaleReservations` 相当（RESERVED タイムアウト→FAILED 解放）を全体配信にも設け、**同一ウィンドウ内で再試行**できるようにする。解放猶予がウィンドウ幅を超える場合はフォールバック②のとおり未配信で残す（設計でウィンドウ幅と解放猶予の整合を確認）。

**同日に同一団体が複数セッションを持つ場合（枠に関わる明示決定）**
- dedupe は (配信グループ, セッション) 単位のため、**同日に同一団体が2セッションある場合は2回配信**され、その日は枠を2回分消費する（各セッションで札組＝会場・試合数が異なりうるため妥当）。稀なケースだが、枠が唯一の制約であることを踏まえ明示する。

**グループID・bot参加（運用セットアップ）**
- botが全体グループに招待されると LINE から `join` Webhookイベントが届く。これを**新規ハンドラで捕捉し、そのbot（チャネル）に紐づく形でグループIDを保存**する。グループIDが未捕捉のbotは配信に使えない（管理画面で「未参加」表示）。
- **アプリでは自動化できない手作業**（運用者が行う。管理画面に手順を明示）:
  1. 対象botのLINEチャネル設定で「グループトーク・複数人トークへの参加を許可」を有効化＋Webhook有効化（既存 `migrateWebhookUrls` で一括設定可）。
  2. 全体グループに対象botを人手で招待する（bot自動参加はLINE仕様上不可）。

**権限・スコープ**
- 管理画面・配信設定APIは ADMIN+（`@RequireRole`）。ADMINは自団体の配信グループのみ、SUPER_ADMINは全団体。

## 5. Non-goals（今回やらないこと）

- **手動「今すぐ全体配信」ボタン**は作らない（トリガーは自動＝30分前/8:00フォールバックのみ、というユーザー指定に従う。将来追加可）。
- **botの全体グループへの自動参加**はしない（LINE仕様上不可。人手で招待）。
- **有料プランの導入・200通超の配信**はしない（無料枠内でローテ吸収）。
- **札組生成アルゴリズム（`cardRules.js` / `CardDivisionTextService`）の変更**はしない。
- **個人LINE通知（購読制・3時間前）の変更・廃止**はしない（併存）。
- **Flex/リッチメッセージでの全体配信**はしない（個人通知と同じプレーンテキスト）。
- **受信到達保証・既読取得・個別受信者の特定**はしない（グループpushの性質上、誰が読んだかは扱わない）。
- **1団体に複数の全体グループ**、**過去日/任意日付の再配信**は今回スコープ外。

## 6. 技術的制約・契約

- **再利用する既存資産**: 札分けテキスト＝`CardDivisionTextService.buildTextForSession`／グループpush＝`LineMessagingService.sendPushMessage(token, groupId, text)`（`to` にグループIDを渡すのみ）／当月実測枠の同期＝`LineMessageCountSyncScheduler`（全チャネル対象・そのまま流用）／開始時刻解決＝既存 `resolveFirstMatchStartTime` 相当。
- **新スケジューラ**: 個人通知の `CardDivisionReminderScheduler`（3時間前・開始時刻なしはスキップ）とは**タイミングが異なる**（30分前＋8:00フォールバック）ため、**別スケジューラ**（例 `CardDivisionBroadcastScheduler`）を新設する。`zone="Asia/Tokyo"`・`JstDateTimeUtil` 使用・`today/now` を引数化して決定論テスト化。
- **チャネルプールのカニバリ防止**: 新 `channel_type = GROUP` を導入し、`line_channels.channel_type` の CHECK 制約を拡張する（**本番introspectで現行許容値を確認してから張り直す**。過去にテンプレ古さで有効値を潰した実害あり＝[[reference_prod_matches_no_check_constraint]] / [[impl_card_division_line_reminder]]）。個人割当 `findFirstByStatusAndChannelTypeOrderByIdAsc(AVAILABLE, PLAYER)` は種別で分離済みのため `GROUP` を掴まない。流用は本番の `PLAYER/AVAILABLE`（47体在庫）を `GROUP` に付け替えて行う。
- **データモデル（案・詳細は技術計画で確定）**:
  - `line_broadcast_group`（配信グループ設定）: `organization_id`・名称・有効フラグ・想定受信数（任意）等。
  - `line_channels` 拡張: `broadcast_group_id`（nullable FK）・`line_group_id`（捕捉したグループID・nullable）を追加。`channel_type` に `GROUP`。**entity変更とmigration SQLは同一PR＋本番適用**。
  - `line_broadcast_send`（全体配信の送信ログ兼dedupe）: `broadcast_group_id`・`session_id`・使用 `line_channel_id`・受信数・status（RESERVED/SUCCESS/FAILED/SKIPPED）・error・sent_at。**部分ユニークインデックス `(broadcast_group_id, session_id) WHERE status IN (SUCCESS, RESERVED)`** で原子的に一度きりを担保（`INSERT ... ON CONFLICT DO NOTHING`）。player_id スコープの `line_message_log` dedupe（`player_id` NOT NULL・部分ユニーク index がplayer単位）は流用しない。この表が管理画面の配信ログ・監査の実体も兼ねる。
- **グループID捕捉**: `LineWebhookController` に `join`（および必要なら `memberJoined`）イベントハンドラを追加。Webhookはチャネル別URL（`/api/line/webhook/{lineChannelId}`）で既にルーティング済み → イベント発火チャネルに `line_group_id` を保存。`leave`/`memberLeft` の扱いは設計で定める（配信不能検知）。
- **枠会計と想定受信数**: 送信前ゲート `当月送信数 + 想定受信数 ≤ 200`。想定受信数の情報源は LINEのグループ人数取得API（`/v2/bot/group/{groupId}/members/count`）または設定値（技術計画で確定）。毎時同期のラグ対策として送信直後に消費分を即時加算する。
- **エンドポイント権限**: 管理系APIは `@RequireRole(ADMIN+)`、団体スコープ（ADMINは自団体・SUPER_ADMINは全団体）。
- **運用依存（既知）**: 早朝発火（8:00フォールバック・早い1試合目）は Render 無料枠のスピンダウンを外部監視（UptimeRobot `/ping`）で回避している前提。本番ログで早朝帯まで無停止発火を確認済み（[[impl_card_division_line_reminder]] §6）。
- **ドキュメント更新（実装と同一コミット）**: `docs/spec/notifications.md`（新スケジューラ・全体配信・ローテ・グループID捕捉）、`docs/design/db.md`（新テーブル・新カラム・CHECK拡張）、`docs/SCREEN_LIST.md`（管理画面）。

## 7. 設計判断の根拠

- **botローテで無料枠を吸収**: LINEのグループ配信は人数分課金（今回の一次調査で確認）＝70名で約70通/回＝月2回/bot。有料化しない制約下で、未使用botを複数投入しローテするのが唯一現実的（ユーザー設計）。無料枠は超過非課金＝最悪でも配信欠落のみ、が安全側の裏付け。
- **専用送信ログ表で冪等＋監査を両取り**: 全体配信はplayer非依存のため、既存のplayer単位dedupeを流用せず `(グループ, セッション)` 単位の専用表で原子的に一度きり。同じ表が管理画面の配信履歴・枯渇判定の実体も担う。
- **別スケジューラ**: トリガーが個人通知（3時間前・無ければスキップ）と異なる（30分前・無ければ8:00配信）。分岐で既存を汚さず、決定論テスト可能な新スケジューラに切り出す。
- **`GROUP` 種別で分離**: 個人割当プールを侵食しないため。在庫47体から付け替えるだけで新規発行不要。
- **手動送信は今回入れない**: ユーザーがトリガーを自動（30分前/8:00）と明示指定したため。管理画面は登録・監視・アラートに徹する。
