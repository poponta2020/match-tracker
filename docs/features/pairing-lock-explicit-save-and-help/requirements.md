# 組み合わせ作成：ロックの明示保存化＆使い方ヘルプ（pairing-lock-explicit-save-and-help）要件定義書

## 1. 概要

### 目的
組み合わせ作成画面（`PairingGenerator` / `/pairings`）について、以下の3点を改善する。

1. **ロックの自動保存を廃止**し、「ロック → 確定して保存」という明示的な保存フローに統一する。
2. ロックボタンの**冗長な「ロック」テキストを削除**し、アイコンのみにする。
3. カレンダー画面（練習カレンダーの「記号の見方」）と同様の**小さな ⓘ ボタン**を設置し、押すと**この画面の使い方**が表示されるようにする。

### 背景・動機
- 現状、各組の「ロック」ボタンを押すと、その回戦の組み合わせが**即座にDBへ保存**される（[PairingGenerator.jsx:519-535](../../../karuta-tracker-ui/src/pages/pairings/PairingGenerator.jsx#L519-L535) の `handleLockPairing` が、未保存時に `createBatch` を呼んでから lock API を実行）。ユーザーの意図しないタイミングで保存が走り、「確定して保存」ボタンとの役割が曖昧になっている。
- 「ロック」ボタンはアイコンと「ロック」テキストが併記されており冗長。テキストを削るとアイコンの意味が伝わりにくくなるため、代わりに使い方ヘルプ（ⓘ）で補う。
- 画面の操作（ドラッグでの入替、ロックの意味、保存の流れ、日付列の見方）は初見では分かりにくく、ヘルプ導線がない。

### ヒアリングで確定した方針
| 論点 | 決定 |
|------|------|
| ロック/解除の反映タイミング | **「確定して保存」を押すまでDBに反映しない**（下書き状態） |
| ロック直後の見た目 | **即グレーアウト＋編集不可**（現状のロック済み表示と同じ）。もう一度押す（解除）で編集可に戻る |
| 「ロック」テキスト削除の対象 | **ロックボタンのみ**アイコン化。状態バッジ（「ロック」「結果入力済」）の文言は残す |
| 使い方ヘルプの内容 | **4項目すべて**（選手の入れ替え方／ロックの意味と使い方／保存の流れ／日付列の見方） |
| ヘルプの初回自動表示 | **あり**（端末ごとに初回のみ自動表示。`localStorage` で既読管理。カレンダーと同方式） |
| 表示形式 | カレンダーと同じ**ドロップダウンパネル**（ⓘボタン直下、外側タップ・✕で閉じる） |

## 2. ユーザーストーリー

### 対象ユーザー
- **選手（PLAYER）／管理者（ADMIN / SUPER_ADMIN）**: 組み合わせ作成画面を使う全ロール。ロック/解除・編集・保存は既存どおり全ロールが可能。

### 利用シナリオ
1. ユーザーが `/pairings` で対象日・回戦の組み合わせを編集する。
2. 「この組は確定したい」と思った組の鍵アイコンを押すと、その組が**その場でグレーアウト**してロック状態になる（この時点ではまだ保存されない）。
3. 必要な組をすべてロックし、他の組も整えたら、最後に**「確定して保存」**を押して初めてDBに反映される。
4. ロックを解除したい場合は「解除」を押すと編集可能に戻る（これも保存するまで反映されない）。
5. 操作に迷ったら、画面上部の **ⓘ「使い方」** を押すと、入替・ロック・保存・日付列の見方が表示される。初回訪問時は自動でパネルが開く。

## 3. 機能要件

### 3.1 ロックの下書き化（自動保存の廃止）

**ロック操作（未ロックの組）**
- 両選手が揃っている組にのみ鍵アイコンボタンを表示する（現状どおり）。
- 押下で**ローカル状態の `locked` を true にするだけ**。サーバ通信は行わない。`hasUnsavedChanges = true` とし、未保存ドラフトに反映する。
- ロックした組は即座にグレーアウト表示（現状のロック済み表示の分岐 `(hasResult || locked)` に入る）に切り替わり、ドラッグ・選手変更不可になる。
- 現状の「未完成の組があるとロック不可（`hasIncompleteUnlockedPairings` で無効化）」は廃止してよい（即時保存しなくなるため）。最終的な不整合は「確定して保存」ボタンの活性条件（未完成の未ロック組があると保存不可）で担保する。

**解除操作（ロック済みの組）**
- 手動ロック中の組（`locked && !hasResult`）に表示される「解除」ボタンは、**ローカル状態の `locked` を false にするだけ**。サーバ通信は行わない。`hasUnsavedChanges = true` とする。
- DB保存済みのロック組（`id` あり）でも、未保存の新規ロック組（`id` なし）でも、同じくローカルで解除できる（現状の `pairing.id` 必須条件を撤廃）。
- 「解除」ボタンの文言は残す（テキスト削除対象はロックボタンのみ）。

**保存操作（確定して保存）**
- 「確定して保存」押下時、**手動ロック組を含む全ての編集対象組**を保存リクエストに含める（`locked` フラグ付き）。結果入力済み組（`hasResult`）は従来どおり保護対象で送信しない。
- 保存後はキャッシュ更新・`hasUnsavedChanges` リセット・次回戦への自動遷移など、現状の `handleSave` の後続挙動を維持する。

**保護挙動（自動組み合わせ・回戦削除）— 現状維持**
- 「自動組み合わせ」「全削除（回戦削除）」は引き続き**手動ロック組（locked）と結果入力済み組（hasResult）を保護**する（保存済みのロック状態に対して有効）。本変更では挙動を変えない。

### 3.2 ロックボタンのアイコン化

- 未ロック組のロックボタンから「ロック」テキストを削除し、鍵アイコン（`Lock`）のみにする（[PairingGenerator.jsx:1150-1162](../../../karuta-tracker-ui/src/pages/pairings/PairingGenerator.jsx#L1150-L1162)）。
- アクセシビリティのため `aria-label="ロック"` と `title`（説明）を付与する。
- 状態を示すバッジ（「ロック」＝amber／「結果入力済」＝blue）の文言は**残す**。
- 「解除」「リセット」「全削除」など他のボタンの文言も**変更しない**。

### 3.3 使い方ヘルプ（ⓘ）

**配置・表示**
- 画面上部（`PageHeader` 直下・右寄せ）に、`Info` アイコン＋「使い方」ラベルの小さなボタンを置く。カレンダーの「記号の見方」と同じ見た目・トーン。
- 押下でボタン直下に**ドロップダウンパネル**を開閉する。パネルは外側タップおよび右上の ✕ で閉じる。
- **初回訪問時のみ自動で開く**。端末単位で `localStorage`（キー例: `pairingHelpSeen`）に既読フラグを保存し、2回目以降は手動表示のみ。`localStorage` が使えない環境では毎回自動表示にフォールバック（カレンダーと同方式）。

**内容（4セクション）** ※文言はドラフト。レビューで調整。
1. **選手の入れ替え方**: 選手チップをドラッグ＆ドロップ、またはタップで選んでから移動先をタップすると、組み合わせ・待機中の選手を入れ替えできる。
2. **ロックの意味と使い方**: 🔒（鍵アイコン）を押すとその組をロックする。ロックした組は自動組み合わせ・一括保存・回戦削除から保護され変更されない。解除は「解除」を押す。ロック・解除は「確定して保存」を押すまで保存されない。
3. **保存の流れ**: 編集内容は「確定して保存」を押すまで保存されない。ある試合の編集中は、保存するまで他の試合は編集できない。
4. **日付列の見方**: 各組の右の表示はその2人の直近対戦日。「初」＝対戦記録なし、「⚠今日」＝今日すでに対戦済み、「MM/DD」＝直近対戦日。

### 3.4 ビジネスルール / エラーケース
- ロック可能なのは両選手が揃った組のみ（未完成組はロック不可）。
- 「確定して保存」は、未完成（片側空欄）の**未ロック組**がある場合は無効（現状の活性条件を維持）。
- 保存対象が1組もなく待機者もいない場合はエラー表示（現状の「保存する組み合わせがありません」を踏襲。判定は「完成した組（ロック含む）が0かつ待機者0」に一般化）。
- 二重ブッキング（同一選手が同回戦の複数組）はUI上構造的に発生しない（選手移動で元から除去される）ため、保存時の追加検証は不要（現状の `createBatch` の重複排除に委ねる）。

## 4. 技術設計

### 4.1 DB設計
- **スキーマ変更なし**。`match_pairings.locked` カラムは手動ロック機能（pairing-manual-lock）で追加済み・本番適用済み（現に稼働中）。**新規マイグレーション不要・本番DB適用不要**。

### 4.2 API設計

**変更エンドポイント**

| メソッド | パス | 変更内容 |
|----------|------|----------|
| POST | `/api/match-pairings/batch` | リクエスト要素に `locked` を追加。保存時に `locked` を反映して新規作成。保存時の保護対象を「結果入力済み（hasResult）のみ」に変更（手動ロック組はリクエストで送られてくるため、削除→再作成で `locked` を再現） |

**未使用化するエンドポイント（本変更では削除しない）**

| メソッド | パス | 扱い |
|----------|------|------|
| PATCH | `/api/match-pairings/{id}/lock` | PairingGenerator から呼ばれなくなる（即時ロックを廃止）。影響範囲縮小のため**エンドポイントは残置**。将来クリーンアップ候補 |
| PATCH | `/api/match-pairings/{id}/unlock` | 同上 |

**変更しないエンドポイント**: `auto-match`（手動ロック保護を維持）、`date-and-match` の DELETE（全削除＝手動ロック保護を維持）、各 GET。

### 4.3 Entity / DTO
- `MatchPairingCreateRequest.java`（バッチ要素DTO）: `private Boolean locked;` を追加（null は false 扱い）。
- `MatchPairingDto.java`: `locked` は既存（変更不要。フロントは既に `p.locked` を読む）。

### 4.4 バックエンド設計（`MatchPairingService.createBatch`）
- 保存時の保護判定を `isLockedPairing`（hasResult OR locked）から **`hasResult`（結果入力済み）のみ**に変更する。
  - 結果入力済み組は従来どおり保護（削除しない・両選手を新規ペアから除外）。
  - 手動ロック組はリクエストに含まれて送られてくるため、既存レコードは削除→リクエストから再作成する（`locked` フラグを再現）。これにより**解除（locked=false での保存）も正しく反映**される。
- 新規ペア構築時、`MatchPairing.builder()...` に `.locked(Boolean.TRUE.equals(request.getLocked()))` を設定する。
- 重複排除・ゾンビ掃除・待機者登録・組織スコープ検証は現状ロジックを維持する。
- `auto-match` / `deleteByDateAndMatchNumber` の保護判定は**変更しない**（hasResult OR locked のまま）。

### 4.5 フロントエンド設計

**`api/pairings.js`**
- `createBatch` の各 pairing 要素に `locked` を含めて送る（`{ player1Id, player2Id, locked }`）。
- `lock(id)` / `unlock(id)` は呼び出されなくなる（メソッドは残置可。任意で削除）。

**`PairingGenerator.jsx`**
- `handleLockPairing` → **ローカル状態トグル**に変更。対象組の `locked` を true にし、`setPairings` / `setHasUnsavedChanges(true)` / `saveDraft` を実行。`createBatch`・`pairingAPI.lock` 呼び出しと再取得を削除。`hasIncompleteUnlockedPairings` による無効化を撤廃。
- `handleUnlockPairing` → **ローカル状態トグル**に変更。対象組の `locked` を false にし、同様に状態更新。`pairingAPI.unlock`・再取得・`pairing.id` 必須条件を削除。
- 「解除」ボタンの表示条件から `pairing.id` を外し、`!isReadOnly && !isViewMode && pairing.locked`（手動ロック）で表示。
- `handleSave` のリクエスト生成を `filter(p => !(p.hasResult || p.locked))` → **`filter(p => !p.hasResult)`** に変更し、`map` で `locked: !!p.locked` を付与。空判定（保存対象なし）を「完成した組（ロック含む）が0かつ待機者0」に一般化。
- ロックボタン: 「ロック」テキスト削除（アイコンのみ）、`aria-label` / `title` 付与。
- 使い方ヘルプ: `showHelp` state、`localStorage`（`pairingHelpSeen`）による初回自動表示、外側タップ/✕で閉じる `useEffect`・`ref`、`Info` アイコンボタン＋ドロップダウンパネル（4セクション）を追加。カレンダー（[PracticeList.jsx:63-72,154-175,680-748](../../../karuta-tracker-ui/src/pages/practice/PracticeList.jsx#L680-L748)）の実装を踏襲。

### 4.6 状態遷移（ロック）
| 操作 | locked（ローカル） | DB反映タイミング | 見た目 |
|------|---------|----------|--------|
| 鍵アイコン押下 | false → true | 「確定して保存」時 | グレーアウト＋「ロック」バッジ＋「解除」 |
| 「解除」押下 | true → false | 「確定して保存」時 | 通常の編集可能行に戻る |
| 「確定して保存」 | 確定 | 即時（createBatch に locked 同梱） | キャッシュ更新・次回戦へ |
| キャンセル | キャッシュから復元 | なし | 保存前の状態に戻る |

### 既存機能への影響
- **手動ロック（pairing-manual-lock）**: ロックの「即時サーバ反映」から「下書き＋保存時反映」へ挙動変更。保護効果（自動組み合わせ・一括保存・回戦削除からの保護）自体は維持。
- **自動組み合わせ / 全削除**: 保護判定は不変（hasResult OR locked）。保存済みロックに対しては従来どおり保護。
- **結果入力済みロック（match-result-lock）**: 影響なし（hasResult はそのまま保護）。
- **PairingSummary・他画面**: 影響なし。
- **PATCH lock/unlock エンドポイント**: 未使用化（残置）。他からの呼び出しが無いことを確認済み（PairingGenerator のみ）。

### リスク・注意点
- `createBatch` の保護判定変更により、手動ロック組は保存のたびに削除→再作成され `id` が変わる。手動ロック組の `id` を参照する箇所は無い（結果は `matchId`、ペア履歴は選手ID基準）ため安全。回帰テストで担保する。
- ロックボタンのアイコンのみ化に伴い、意味の伝達をヘルプ（ⓘ）と `aria-label`/`title` で担保する。
- DBスキーマ変更なし（CLAUDE.md のマイグレーション適用ルールは非該当）。

## 6. 設計判断の根拠
| 判断 | 理由 |
|------|------|
| ロックを下書き化し保存時に確定 | ユーザー要望（自動保存をやめ、明示的な「確定して保存」に統一）。編集途中の予期せぬ保存を防ぎ、保存ボタンの役割を一貫させる。 |
| ロック直後に即グレーアウト（現状表示を流用） | ヒアリング結果。既存のロック済み表示分岐 `(hasResult || locked)` を再利用でき、実装が最小・見た目が一貫。 |
| 保存APIに `locked` を同梱（PATCH lock/unlock を使わない） | 1回の保存でロック状態まで確定でき、解除（locked=false）も同一経路で反映できる。複数APIコールやID解決の複雑さを回避。 |
| createBatch の保存時保護を hasResult のみに変更 | 手動ロック組がリクエストで送られてくる新モデルでは、既存ロック組を保護（残置）すると解除が反映できない。削除→再作成に統一することで lock/unlock 両方が正しく永続化される。 |
| auto-match / 全削除 の保護は変更しない | これらは保存済みロック状態に対する保護であり、ユーザー要望の対象外。挙動を変えず影響範囲を最小化。 |
| ロックボタンのみテキスト削除（バッジは残す） | ヒアリング結果。操作ボタンは省スペース化しつつ、状態表示（ロック済/結果入力済）の可読性は維持。 |
| 使い方ヘルプはカレンダーと同方式（ドロップダウン＋初回自動表示） | ユーザー要望（「カレンダー画面のように」）。既存実装を踏襲でき、UXとコードの一貫性が高い。テキスト削減分の発見性も担保。 |
