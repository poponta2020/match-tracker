# 練習カレンダーの「出欠登録」を1日分の参加＋キャンセル画面に変更 要件定義書

## 1. 概要

- **目的:** 練習カレンダー（`/practice`）で練習日をタップして開くセッション詳細ポップアップの「出欠登録」ボタンを押したとき、**月まとめの一括入力画面（`/practice/participation`）へ飛ばすのをやめ**、押した日付のそのセッションだけで「参加登録」と「キャンセル（理由付き）」の両方を完結できる専用画面へ遷移するように変更する。
- **背景・動機:** 特定の1日の出欠を直したいだけなのに、月全体のチェック表（一括入力画面）に飛ばされて対象日を探し直す必要がある。さらに参加は一括入力画面、キャンセルは別のキャンセル画面（`/practice/cancel`）と導線が分かれており、1日分の出欠を素直に扱えない。押した日付に閉じた1画面で参加もキャンセルもできるようにする。

## 2. ユーザーストーリー

- **対象ユーザー:** 練習に参加する全ロール（PLAYER / ADMIN / SUPER_ADMIN）の本人。
- **目的:** カレンダーで押した「その日」の自分の出欠（どの試合に出る／出た枠をやめる）を、その場で完結させたい。
- **利用シナリオ:**
  1. カレンダーで練習日をタップ → セッション詳細ポップアップが開く
  2. 「出欠登録」ボタンを押す → **その日のそのセッション専用の出欠登録画面**へ遷移
  3. 参加したい試合をトグルで選んで保存する／既に確定している枠を理由を選んでキャンセルする
  4. 完了オーバーレイの「カレンダーに戻る」でカレンダーへ戻る

## 3. 機能要件

### 3.1 画面と遷移（画面インベントリ＋ナビゲーション地図。見た目は design-spec 参照）

**変更する導線:**
- `/practice` セッション詳細ポップアップ内「出欠登録」ボタン（`isPastDate=false` のときのみ表示、既存維持）
  - Before: `AttendanceRegisterModal`（参加登録/キャンセル登録の2択）を開く
  - After: **新しい1日分出欠登録画面へ直接遷移**（`AttendanceRegisterModal` を経由しない）

**新規画面:**
- **1日分出欠登録画面**（新ルート）
  - 仮ルート: `/practice/attendance?sessionId=<セッションID>`（最終的なパス・パラメータ名は技術計画で確定）
  - 対象は**ポップアップで開いていたそのセッション1件**（同日に複数団体のセッションがある場合も、開いていた1件のみを対象とする）
  - 上部: 対象セッションのヘッダー（日付・会場名・団体）
  - 参加登録エリア: 各試合番号（1〜`totalMatches`）の参加トグル
  - キャンセルエリア: 登録済み（確定枠）の試合を理由付きでキャンセル（**当月扱いのときのみ表示**、後述 3.2）
  - 保存/キャンセル実行は `SaveProgressOverlay`（保存中／完了／エラー）を表示、完了で「カレンダーに戻る」→ `/practice`

**変更しない導線（据え置き）:**
- `/practice` 右下フローティングボタン: **ラベルのみ「出欠一括登録」に変更**し、機能は不変（従来どおり `AttendanceRegisterModal` を開き、参加登録 → `/practice/participation`（月まとめ）、キャンセル登録 → `/practice/cancel`（月まとめ）へ）
- `Home.jsx` の参加率カードからの「参加登録」リンク（`/practice/participation`）
- `/practice/participation`・`/practice/cancel` の各画面自体の内部ロジック（新画面はこれらのロジックを流用するが、既存画面は据え置き）

### 3.2 ビジネスルール（既存挙動の union。新画面はこれらを再現する）

- **セクションの排他振り分け（design 収束で確定）:** 各試合は「未参加＝参加する側／参加中＝キャンセル側」のどちらか一方にだけ表示する（重複させない）。結果として **全試合参加中→参加セクション非表示（キャンセルのみ）／全試合未参加→キャンセルセクション非表示（参加のみ）** が自然に成立する。ただし来月扱いは例外（下記）。
- **満員でも抽選前はチェック可（design 収束で確定・既存挙動）:** capacity 到達（満員）でも抽選前は参加登録できる（超過はキャンセル待ち→抽選で解決）。満員は情報表示にとどめ、チェックボックスを無効化しない。登録不可は「抽選確定済み（ステータス表示）」「伝助削除（×）」のみ。
- **参加トグル（登録）:**
  - 選択可能な試合は 1〜`min(totalMatches, N)`。伝助側で削除承認済みの試合番号はトグルを出さず `×` 表示（既存 `densukeDeletionCandidateMatchNumbers`）。
  - 保存は `registerParticipations` を呼ぶ。**送信ペイロードは月・団体全体の participations を積む**（対象セッションのトグル差分のみ反映し、同月・同団体の他日の参加は保持したまま送る）。API が月・団体単位の全置換方式のため、対象日だけを送ると他日の参加が消える（→ これを防ぐことが最重要の回帰防止項目）。
  - `expectedVersion`（楽観ロック版）を送信。409 は既存同様にエラー表示し最新を再読込する。
- **既存参加の取り消し方（当月扱い／来月扱いの区別を維持）:** 判定は既存 `resolveAttendanceMode(year, month, hasAnyExecutedLotteryInMonth)` を流用。
  - **当月扱い**（現在月、または未来月で月内に抽選確定済みが1つ以上）: 既存のアクティブ参加はトグルではロックして外せない。取り消しは**理由付きキャンセルエリア**（WON / PENDING が対象）で行う。
  - **来月扱い**（未来月で抽選確定なし）: 既存参加はトグルを外すだけで理由なしに取り消せる（`registerParticipations` の全置換で反映）。**理由付きキャンセルエリアは表示しない**。
  - **抽選確定済みセッション**（`lotteryExecuted[sessionId]=true`）: 参加トグルは操作不可・ステータスバッジ表示のみ。WON / PENDING は理由付きキャンセル可。
- **理由付きキャンセル:** 既存 `PracticeCancelPage` と同一。
  - 対象試合（WON / PENDING）を選択 → 理由（体調不良／仕事・学業／家庭／交通機関／その他）を選択 → 実行。`その他` は自由記述必須。
  - 実行は `cancelMultiple(participantIds, 理由, 詳細)`（対象試合の `participantId` を集約）。
  - 実行前に確認（既存の `window.confirm` 相当）。
- **SAME_DAY 当日12:00以降の確認ダイアログ:** 参加保存・キャンセル実行の両方で既存の追加確認ダイアログを維持（参加は `needsSameDayConfirm`、キャンセルは当日12時判定）。
- **過去日:** 過去日ではこの画面に到達しない（ポップアップの「出欠登録」ボタンが非過去日のみ表示、既存維持）。
- **副作用:** 登録/キャンセルに伴う既存の副作用（伝助書き戻し同期、補充募集通知など）はバックエンド側でそのまま発火（本改修で新規追加・変更はしない）。

## 5. Non-goals

- 月まとめ参加登録画面（`/practice/participation`）・月まとめキャンセル画面（`/practice/cancel`）自体の内部ロジック改修（新画面はロジックを流用するが既存画面は据え置き）
- 新規バックエンドAPI・DBスキーマ・マイグレーションの追加（既存 `registerParticipations` / `cancelMultiple` を流用。**フロントのみ**）
- 同一日に複数セッションがある場合に「日付単位で全セッションまとめて」操作する機能（対象はポップアップで開いた1セッションのみ）
- 参加登録とキャンセルを1つのボタンで同時コミットする統合トランザクション（参加とキャンセルは別アクション。各アクション独立に実行→完了でカレンダーへ戻る）
- 来月扱いでの取り消しに理由を課す挙動変更（今回は既存の「理由なしで外せる」を維持）
- LINE通知・伝助同期など既存副作用の新規追加・仕様変更

## 6. 技術的制約・契約

- **`registerParticipations` は月・団体単位の全置換**（`PracticeParticipantService.registerParticipations`）。新画面は必ず月全体の participations を読み込み・送信する。対象セッションのみのペイロードは禁止（他日データ消失事故になる）。
- **`cancelMultiple` は participantId 単位**（`lotteryAPI.cancelMultiple`）。`getPlayerParticipationStatus` が返す `participations[sessionId][].participantId` を使う。
- 既存ヘルパー・共通コンポーネントを流用: `resolveAttendanceMode`、`needsSameDayConfirm`（`utils/sameDayConfirm.js`）、`SaveProgressOverlay`、`getInitialDateFromQuery` 相当のクエリ解決。
- 操作対象は `currentPlayer` 本人のみ（他者の出欠は操作しない）。新規バックエンドエンドポイントは追加しないため `@RequireRole` 付与対象なし。
- 対象データ取得は既存API流用: `getById`（セッション詳細）、`getPlayerParticipations` / `getPlayerParticipationStatus`（月内の参加・ステータス・version・lotteryExecuted・hasAnyExecutedLotteryInMonth）。

## 7. 設計判断の根拠

- **新ロジックではなく consolidation（再配線）:** 参加登録は `PracticeParticipation`、理由付きキャンセルは `PracticeCancelPage` の既存ロジックの union を1セッションにスコープして再構成する。よって主要リスクは新機能バグより**既存挙動の回帰**であり、AC-3/5/6/8/10/13 を回帰防止として明示する。
- **UI は専用画面（遷移）:** 理由ラジオ・進捗オーバーレイ・2アクション（参加/キャンセル）を無理なく置け、既存の全画面パターンを流用できる（ユーザー選択）。
- **当月/来月の区別を維持:** 「確定した枠を手放す＝理由を聞く／未確定の申込を外す＝理由なし」という既存の意味づけを保ち、挙動変更を避ける（ユーザー選択）。
- **FAB は機能維持・改名のみ:** まとめて入力したいニーズの導線を残しつつ、1日分導線（ポップアップ内ボタン）と役割を言葉で分ける（ユーザー選択）。1PR=1機能で影響範囲を最小化。
