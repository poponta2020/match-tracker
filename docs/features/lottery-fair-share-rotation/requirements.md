# 抽選の公平化（一巡保証 ＋ 直近30日の重み付き抽選）要件定義書（改修）

> **用語:** 競技かるたでは試合をすることを **「取る」** と言う（「打つ」は誤り）。本書では試合の実施回数を「取れた試合数」と表記する。

## 1. 概要

- **目的:** 「同じ人が続けて落選する」を解消し、落選の負担をメンバー間で公平に分配する。
- **背景・動機:** 現行 `LotteryService.processMatch` は (1) 同セッション先行試合の落選者を **cascade（連鎖落選）** で低優先度に落とし、(2) 月内落選を **二値の救済（monthlyLosers）** で補償する2階建て。この cascade により、希望者が定員を超える日には**同じ人が1日の全試合で落選**する。
- **ユーザーが解消したい痛み:**
  - **(A) 1日全落ち** — その日1試合も取れない人が出る
  - **(B) 連日落ち** — 毎回同じ人が弾かれる
- **前提の転換:** 過去改修（Issue #233/#234、`docs/features/lottery/fix-requirements-v2.md`）は「移動コスト考慮でまとめて落選が望ましい」として cascade を意図的に導入した。今回ユーザーは**無駄足（会場に来て1試合だけ）を許容すると明言**（2026-07-18）したため、この前提を撤回する。

## 2. 変更の要旨（before → after）

| 観点 | 現行（before） | 変更後（after） |
|---|---|---|
| 連鎖落選（cascade） | 先行試合の落選者を後続で低優先度＝落ちやすい | **廃止**（真逆にし、落選者を先に座らせる） |
| 月内救済（monthlyLosers） | 「落ちたか否か」の二値 | **廃止**し、直近30日の取得数（連続量）へ一般化 |
| 当落の決め方 | 3層（管理者優先／救済／一般）＋一般枠30% | **2層**（① その日の一巡保証＝絶対 ② 直近30日の重み付き抽選＝確率） |
| 集計期間 | 暦月（月初リセット） | **その練習日から遡る直近30日のローリング** |
| 重みの基準 | 固定設定値 `lottery_normal_reserve_percent`（30%一般枠） | **その試合の候補者の上位◯%をフラット化**（キャップのパーセンタイルは新設定 `lottery_weight_cap_percentile`、デフォルト30＝上位70%フラット。管理者が画面から変更可能） |
| 管理者指定優先 | 最上位 | **最上位のまま維持** |
| 乱数 | グループ内シャッフル | **同条件の候補内での重み付き抽選**（シード再現性は維持） |

## 3. 機能要件（変更後の挙動）

### 3.1 画面と遷移
- 抽選管理（`/admin/lottery`）・抽選結果（`/lottery/results`）の**レイアウト・遷移は不変**。当選/キャンセル待ちの顔ぶれだけが新ロジックで変わる。
- 抽選管理画面の「システム設定」ボタンから遷移する既存の**システム設定画面（`/admin/settings`, `SystemSettings.jsx`）を変更**する（詳細は §3.7）。新規ルート・新規画面は作らない。

### 3.2 実行中に保持するカウンタ

| カウンタ | 定義 | リセット |
|---|---|---|
| `todayTaken[選手]` | **そのセッション（1日）で取れた試合数** | セッションごとに0 |
| `recentTaken[選手]` | **そのセッション日付から遡って30日間**に、同一団体で取れた試合数（WON件数） | リセットなし（常にローリング） |

- どちらも、抽選中に当選が確定するたびに +1 する。
- `recentTaken` の初期値は、抽選実行前から存在する WON（管理者手動追加・当日join・繰り上げ承諾・確定済みセッション分）を算入する。**`OFFERED`（応答待ち）は定員には算入するが `recentTaken` には数えない。**

### 3.3 ビジネスルール（processMatch の新挙動）

処理順序（セッション日付昇順 → 試合番号昇順 → 参加者ID昇順）と、A-2 の定員控除は現行踏襲。

```
入力: applicants（この試合のPENDING）, capacity, adminPriorityIds, rng（seedで初期化）

1. 定員調整（現行A-2を踏襲）
   capacity = max(0, capacity - (この試合の既存WON数 + 既存OFFERED数))

2. capacity == null または applicants.size <= capacity なら全員当選して終了

3. 座席割り当て
   adminPool = applicants ∩ adminPriorityIds
   restPool  = それ以外
   seated  = 選抜(adminPool, min(capacity, adminPool.size))
   seated += 選抜(restPool, capacity - seated.size)

4. seated → WON / それ以外 → WAITLISTED

選抜(pool, k):
   k回くりかえす:
     # 【ルール1・絶対】その日まだ取れていない人から
     m = pool 内の todayTaken の最小値
     候補 = pool のうち todayTaken == m の者

     # 【ルール2・確率】同点のときだけ、直近30日の取得数で重みづけ
     if 候補が1人: 当選者 = その人
     else:
        cap  = 候補の recentTaken の p パーセンタイル（p = 設定値、算出は 3.4）
        w[i] = 1 / (min(recentTaken[i], cap) + 1)
        当選者 = 重み付き抽選（rng）で1人

     seated へ追加 / pool から除外 / todayTaken +1 / recentTaken +1
```

**ルール1が候補を絞り込み、ルール2はその候補の中でしか働かない。** これにより「その日は必ず一巡する」という保証と、「日をまたいだ公平性」を両立させる。

### 3.4 キャップ（上位70%のフラット化）★重要

重みの基準となる `cap` は、**その試合の候補者だけ**から毎回算出する。

```
p = システム設定 lottery_weight_cap_percentile（整数, デフォルト 30）
1. 候補者の recentTaken を昇順ソート:  v[0] <= v[1] <= ... <= v[n-1]
2. idx = clamp(ceil((p/100) * n) - 1, 0, n-1)   # nearest-rank による p パーセンタイル
3. cap = v[idx]
4. cap == v[0]（最小値と一致）の場合は cap = v[0] + 1 とする
```

結果として、**`recentTaken` が cap 以上の候補（＝おおむね上位70%）は全員まったく同じ重み**になり、cap を下回る候補だけが、少なさに応じて有利になる。

- **一番練習に来ている人が、毎回真っ先に落選候補になることがなくなる**（上位70%が大きな同着グループを形成するため、その中の誰が落ちるかは日によって変わる）
- **しばらく参加できていなかった人（下位30%）は保護される**
- 基準がその試合の顔ぶれで毎回決まるため、**人間が調整する固定値が不要**になり、暦月の位置（月初/月末）にも左右されない

**手順4のガードについて:** 候補者が少数のとき、p パーセンタイルが最小値と一致して「全員フラット＝保護が消える」ことが起こりうる。これを防ぐため、cap が最小値と一致する場合のみ +1 する。
**同値が多い場合:** `recentTaken` は離散値のため同値が並びやすく、実際にフラット化される割合は 70% ちょうどにはならない（例: 候補11人で cap=3 のとき9人＝82%がフラット）。これは仕様として許容する。

### 3.5 キャンセル待ち番号

落選者に対して**同じ選抜手続きを続行**し、引かれた順に 1, 2, 3… を付与する。繰り上げの優先度も当落と同じ公平基準に揃う。
**バケット順は当選時と同じ（管理者優先プール → その他プール）**とし、管理者優先の落選者がキャンセル待ちの最上位に入る現行挙動を維持する。
**現行の「連続する試合番号では前試合のキャンセル待ち順番を引き継ぐ」ルールは廃止し、本手続きに一本化する**（毎試合の公平基準で順番が決まるため、引き継ぎと二重管理になる）。

### 3.6 期待される効果（検証対象）
- **1日内:** 落選が全員に1回ずつ回りきるまで、同じ選手が2回落選しない（構造的保証）。
- **日またぎ:** 直近30日で取れていない人ほど有利になり、「毎回同じ人が弾かれる」が起きない。
- **暦月の境界に依存しない**（月初・月末で基準が切り替わらない）。

### 3.7 システム設定画面の変更（`/admin/settings`）

対象は抽選管理画面（`/admin/lottery`）の「システム設定」ボタンから遷移する**既存の `SystemSettings.jsx`**。権限は既存どおり **ADMIN（自団体のみ）/ SUPER_ADMIN（団体選択で全団体）**。設定は既存の per-org KV（`system_settings`）に保存し、`systemSettingsAPI.getAll(orgId)` / `update(key, value, orgId)` にそのまま乗る。

- **① 「一般枠の最低保証割合」カードを削除**（`lottery_normal_reserve_percent`）。
- **② パーセンタイル設定カードを追加**：
  - 新設定キー `lottery_weight_cap_percentile`（整数、**デフォルト 30**、バリデーション 0〜100）
  - 既存の締切日数カードと同じ体裁（数値入力＋保存は既存の一括「保存」ボタンで反映）
  - **効果の方向をインライン説明で明示**：値を**小さくするほど**多くの人が横並びになり、よく参加している人も不利になりにくい。**大きくするほど**参加回数の少ない人がより強く優先される。
- **③ 「抽選の仕組み」説明セクションを追加**（本要件の目玉）：
  - ルール1（その日の一巡保証）・ルール2（直近30日の重み付き抽選）・パーセンタイル設定の意味を、管理者・会員が読んで理解できる日本語で記載する
  - 記載内容は §3.3 の2ルール・§3.4 のキャップと整合させる（実装時に本要件を出典とする）

**未設定団体の扱い:** `lottery_weight_cap_percentile` が未保存の団体はデフォルト 30 が使われる（`SystemSettingService` のデフォルト値付き getter を新設）。
**旧設定行の扱い:** 既存の `lottery_normal_reserve_percent` 行は参照されなくなるだけで無害なため、明示削除は行わない（任意）。

## 6. 技術的制約・契約
- **DB スキーマ変更なし・migration 不要:** 設定は `system_settings` の汎用 KV（`settingKey`/`settingValue`/`organizationId`）。旧 `lottery_normal_reserve_percent` の撤廃も新 `lottery_weight_cap_percentile` の追加も、KV への行操作のみでカラム追加は不要。
  - 新キー `lottery_weight_cap_percentile` は `SystemSettingService` にキー定数＋デフォルト 30 の getter を追加。保存・取得は既存の `systemSettingsAPI.getAll` / `update` にそのまま乗る（新エンドポイント不要）。
- **公開契約は不変:** `/api/lottery/preview`・`/confirm`・`/re-execute/{sessionId}`・`/results`・システム設定の取得/保存エンドポイントの I/O 形状は変えない。
- **再現性契約を維持:** seed → 同一結果。候補リストを ID 昇順に固定した上で重み付き抽選（累積和の走査）を行う。パーセンタイル設定はプレビュー時と確定時で同一団体の同一値が読まれる前提（確定操作の直前に設定変更されるレースは既存の母集団シグネチャ検証の範囲外＝許容）。
- **権限不変:** `@RequireRole`・`AdminScopeValidator` の適用は現行どおり。システム設定画面・API は既存の ADMIN（自団体）/ SUPER_ADMIN（全団体）スコープを踏襲する。
- **スコープ:** MONTHLY 型（抽選運用）の団体のみ影響。SAME_DAY 型（先着運用・わすらもち会）は `processMatch` を通らないため無影響。
- **未解決の技術論点（→ 技術計画フェーズで解決）:**
  - `recentTaken`（日付範囲での WON 件数）を取得するクエリ／リポジトリメソッドの要否。現行は月単位の finder しか無い可能性があり、日付範囲版の追加が要るか確認する。
  - 実行中のインメモリ集計構造（既存 WON の初期ロード＋当選時インクリメント）の持ち方。
  - `reExecuteLottery` の `findMonthlyLoserPlayerIds` を新基準へ置換する具体。
  - 既存 `LotteryServiceTest` の cascade / rescue / 30%一般枠 前提テストの改廃範囲。
  - システム設定の保存 API に整数バリデーション（0〜100）がバックエンド側にあるか（フロントのみか）の確認と、必要なら追加。

## 7. 設計判断の根拠
- **2層構造にした理由:** 「1日全落ちを絶対に無くす」はユーザーの明示要件のため確定ルール（ルール1）で保証する必要がある。一方で日またぎの公平化まで確定ルールにすると、行列の最後尾の人が「確定で落ちる」ことになり不公平が別の形で発生する。よって**1日内は確定・日またぎは確率**に役割分担した。
- **キャップを採用した理由:** 生の累計回数をそのまま重みにすると、**一番練習を頑張っている人が毎回必ず真っ先に落選候補**になる（累計が最大＝重み最小のため）。固定のキャップ値でも解決するが、暦月リセットと組み合わせると月初は全員フラット・月末は差が最大となり、同じ「久しぶりの人」でも来る時期で扱いが変わってしまう。**その試合の候補者から算出する相対キャップ**なら母集団に応じて自動調整され、調整用の固定値も不要になる。
- **キャップ位置を中央値ではなく30パーセンタイル（上位70%フラット）を既定にした理由:** 中央値だとフラット化される同着グループが約半数にとどまり、常連の中でも参加回数の多寡で差が残る。上位70%まで広げることで**常連層をより大きな同着グループにまとめ、狙い撃ちをさらに薄める**（ユーザー判断、2026-07-18）。**トレードオフとして、久しぶりの参加者の優遇幅は小さくなる**（例の11人で 7倍 → 4倍）。
- **パーセンタイルを設定化（画面から変更可）にした理由:** 効き具合の最適値は団体の運用・参加者数分布で変わり、事前に決め切れない。固定値ではなく**団体ごとに管理者が調整できる**ようにした（ユーザー要望、2026-07-18）。既存のシステム設定画面と KV 機構にそのまま乗るため追加コストが小さい。デフォルトは 30。**画面に抽選の仕組みの説明も併記**することで、管理者が値の意味を理解した上で調整できるようにする。
- **直近30日ローリングを採用した理由:** 暦月だと月初・月末に段差が生じ、月をまたいだ瞬間に実績が消える。「最近どれだけ取れたか」という感覚にも近い。しばらく参加できなかった人の保護が時間経過で自然に回復する利点もある。
- **キャンセル待ち番号の引き継ぎを廃止した理由:** 新方式では順番が毎試合の公平基準で決まるため、前試合からの引き継ぎと二重管理になり、どちらが優先かが不明瞭になる。
- **移動コスト前提の撤回:** ユーザーが無駄足許容を明言（2026-07-18）。
- **design_required: false:** 新規画面・新規ルートは無い。変更は既存システム設定画面（`SystemSettings.jsx`）への **カード差し替え（一般枠→パーセンタイル）＋説明セクション追加**で、既存カードの体裁を踏襲する定型 UI のため `/design-screen` の収束ループは不要と判断（凝った視覚表現が必要になれば別途 `/design-screen` を回す）。
