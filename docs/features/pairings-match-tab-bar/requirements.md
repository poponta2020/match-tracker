# 組み合わせ画面 試合番号タブの整形（レイアウト＋切替モーション） 要件定義書

> 親機能: `pairings-ui-change`（PR #1023 で導入した「試合番号は下線タブ＋連結パネル」）の純UI改修。ロジック変更なし。

## 1. 概要
- **目的:** `/pairings`（`PairingGenerator.jsx`）の試合番号タブの、(1) 画面幅いっぱいに等分される不自然なレイアウトと、(2) タブ切替時に大きさ・並びがガタつく不自然なモーション、の2点を解消する。
- **背景:** 現状のタブは、非アクティブが `flex-1`（残り幅を等分）・アクティブだけ `flex-none` で大きい文字＋「N試合目」ラベルに変化する。3試合しかなくても幅いっぱいに3分割され、切替のたびに幅・文字サイズ・ラベルが変わって並びが跳ねる。`transition-colors` しか効いていないためジオメトリの変化は瞬間的に起こる。

## 2. 変更の動機と内容（現状 → 変更後）
| 観点 | 現状（before） | 変更後（after） |
|---|---|---|
| タブ幅 | 非アクティブ `flex-1` で等分（3試合でも全幅3分割） | 全タブ **content幅で左寄せ**。3試合は左に寄り、余白は右に残る |
| アクティブ表現 | `flex-none`＋21px 太字＋「N試合目」ラベル＋cream連結パネル（＝サイズが変わる） | **全タブ同一フットプリント**（切替でサイズ・位置が変わらない）。アクティブは色（紺 `#1A3654`）＋太字＋**滑るインジケーター**で表現 |
| 切替モーション | 色のみ transition、幅/文字サイズ/ラベルは瞬間変化 → 並びが跳ねる | ジオメトリのリフロー（跳ね・ずれ）を**除去**しつつ、アクティブインジケーターが**スッと滑る**アニメーションで移動 |
| 試合数が多い時 | `totalMatches`（既定10・会場 `defaultMatchCount`）分を等分 | ≤7 は横スクロールなしで1行に収まる。8以上は**横スクロール**にフォールバック（タブは隠さない） |

- 「N試合目」の意味（この連結パネルが第N試合のものだという情報）は、アクティブタブが肥大化する形ではなく、**連結パネル上部の小見出し等の静的表示**で保持する（切替でレイアウトが動かない形にする）。連結 cream パネル（`#ebe4d8`）自体は維持。

### 承認済み決定（2026-07-12）
- **アクティブインジケーターは ②「cream 連結ハイライトが滑る」**（ユーザー選択）。全タブは数字のみ・同一フットプリントとし、cream（`#ebe4d8`・角丸上）のハイライトが計測値に基づきアニメーションでアクティブタブ位置へ滑走し、下辺で連結パネルに接続する。下線バー案（①）は不採用。

## 3. 変更後の挙動（機能要件）
- タブは `matchNumber` 1..min(totalMatches, 実データ) を content 幅・左寄せ・小さめ gap で描画。コンテナは横スクロール可（`overflow-x-auto`。≤7 では発火しない）。
- タブ押下＝`setMatchNumber(num)`（現状と同一。選択ロジック・キャッシュ・参加者更新は不変）。
- アクティブの視覚表現は **滑るインジケーター（下線または cream ハイライト）＋色/太字**。切替時、全タブのフットプリントは不変でインジケーターだけがアニメーションで移動する。
- リサイズ・フォント読込後もインジケーター位置が破綻しないこと。

## 5. Non-goals（今回やらないこと）
- `totalMatches` の上限変更・練習会設定（`PracticeForm` / 会場 `defaultMatchCount`）の変更。**8試合以上でも動くようスクロールで担保するのみ**で、上限を7に固定する変更はしない。
- `/pairings/summary`（`PairingSummary.jsx`）の試合番号タブの整形。今回のスコープは `/pairings`（`PairingGenerator.jsx`）のみ。同型なので必要なら別 PR で追随。
- タブ以外の脱カードUI・ペア行・チップ・LINE導線・ヘッダ等の見た目変更。
- 試合切替のロジック、未保存ドラフト/読み取り専用の判定、ペアリング生成・保存ロジックの変更。

- 変更対象は原則 `karuta-tracker-ui/src/pages/pairings/PairingGenerator.jsx` のタブ描画部（`:864-896` 付近）のみ。BE・API・DB 変更なし。
- 親機能の documented 挙動（未保存=オレンジ / 完了=緑チェック / isReadOnly ガード）を壊さない＝回帰 AC（AC-5〜AC-7）で担保。

- デザイントークン: `tailwind.config.js` / `src/index.css`（既存の cream `#ebe4d8`・紺 `#1A3654`・muted トークンを流用。新色は原則追加しない）。

## 7. 設計判断の根拠
- 「動きが不自然」の根因はアクティブ/非アクティブでフットプリントが変わること。**サイズ変化（リフロー）を無くす**方向が正で、リフローをアニメで誤魔化す方向は不採用。ユーザー要望「どっちも（ガタつき除去＋滑らかな動き）」は、フットプリント不変＋インジケーターのみ滑走、で両立する。
- 「7試合まで・スクロール不要」というユーザー指定に対し、データ上 `totalMatches` は7を超え得る（既定10）。ハードキャップ変更はスコープ外の副作用が大きいため、**≤7 は1行に収まり・8以上はスクロール**という非破壊フォールバックで満たす。
