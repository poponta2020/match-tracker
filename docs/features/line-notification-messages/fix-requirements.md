# LINEの通知メッセージの仕様 改修要件定義書

## 1. 改修概要

- **対象機能:** LINE通知メッセージ（LineNotificationService / LineReminderScheduler / 関連UI）
- **改修の背景:** 監査レポートにて、仕様書と実装の不整合、コード重複、UI上の表示の曖昧さ、リマインダーメッセージの不自然な日本語等が検出された
- **改修スコープ:** 全7項目（仕様書整備4項目 + コード改修3項目）

## 2. 改修内容

### 2.1 SPECIFICATION.md 旧エンドポイント削除（項目1）

- **現状の問題:** SPECIFICATION.md 7.15節に `POST /api/admin/line/send/lottery-result` が記載されているが、実装には存在しない。実際は `POST /api/lottery/notify-results` にアプリ内通知と統合済み
- **修正方針:** 7.15節から旧エンドポイントを削除し、統合先エンドポイントへの参照を注記する
- **修正後のあるべき姿:** 仕様書のAPIエンドポイント一覧が実装と一致している

### 2.2 sendOfferResponseConfirmation 仕様追記（項目2）

- **現状の問題:** Webアプリから繰り上げオファーに応答した際にLINEに確認メッセージを送信する `sendOfferResponseConfirmation` が実装済みだが、SPECIFICATION.mdに記載がない
- **修正方針:** SPECIFICATION.md 4.3節のLINE通知セクションに「繰り上げ応答確認通知」の仕様を追記する。内容は以下:
  - トリガー: Webアプリから繰り上げオファーに応答（承諾/辞退）
  - メッセージ:
    - 承諾時: `"{日付}の練習 試合{番号}の繰り上げ参加を承諾しました。"`
    - 辞退時: `"{日付}の練習 試合{番号}の繰り上げ参加を辞退しました。"`
  - 通知種別: WAITLIST_OFFER（繰り上げ通知と同一の通知設定を使用）
- **修正後のあるべき姿:** 全LINE通知種別が仕様書に網羅されている

### 2.3 sendToPlayer / sendFlexToPlayer の共通処理抽出（項目3）

- **現状の問題:** `sendToPlayer`（行447-491）と `sendFlexToPlayer`（行397-442）で、LINKED状態チェック・通知設定チェック・チャネル取得・月間上限チェック・ログ記録がほぼ同一のコードで重複している
- **修正方針:** 共通前処理（チャネル取得・各種チェック）を `private` メソッドに抽出し、両メソッドから呼び出す形にリファクタリングする
- **修正後のあるべき姿:** 送信チェックロジックが一箇所に集約され、変更時の修正漏れリスクが解消されている

### 2.4 LINE送信結果を人数ベースに変更（項目4）

- **現状の問題:** `sendLotteryResults()` の `sentCount` はメッセージ通数を返すが、UIでは `LINE送信: ${lineSent}件` と表示しており、「人数」か「通数」かが曖昧。管理者にとってミスリーディング
- **修正方針:**
  - `sendLotteryResults()` 内でプレイヤー単位のカウントに変更（1人に複数通送信しても1人としてカウント）
  - `LineSendResultResponse` のフィールドを `sentCount` → `sentPlayerCount` にリネーム（failed/skippedも同様）
  - UIの表示を `LINE送信: ${lineSent}人` に変更
  - `notifyStatus` のレスポンスの `sentCount` も人数ベースに変更
- **修正後のあるべき姿:** 管理者が「何人にLINE通知が届いたか」を正確に把握できる

### 2.5 締め切りリマインダー未実装の明記（項目5）

- **現状の問題:** `LineReminderScheduler` に締め切りリマインダーの構造のみ存在し送信処理は未実装だが、SPECIFICATION.mdに未実装である旨の記載がない
- **修正方針:** SPECIFICATION.mdの該当箇所に「未実装（PracticeSessionに締め切り日フィールドが未追加のため）」と明記する
- **修正後のあるべき姿:** 未実装機能の状態が仕様書で明確になっている

### 2.6 参加予定リマインダーの日数ラベル改善（項目6）

- **現状の問題:** `LineReminderScheduler` 行65で `days == 1 ? "明日" : days + "日後"` としており、「2日後5月3日は練習日です。」のような不自然な日本語になる
- **修正方針:** 日数ラベルを以下に変更:
  - 1日前 → `"明日"`
  - 2日前 → `"明後日"`
  - 3日以上前 → ラベルなし（日付のみ）
- メッセージ例:
  - 1日前: `"明日5月3日は練習日です。準備をお忘れなく！"`
  - 2日前: `"明後日5月3日は練習日です。準備をお忘れなく！"`
  - 3日以上前: `"5月3日は練習日です。準備をお忘れなく！"`
- **修正後のあるべき姿:** 自然な日本語のリマインダーメッセージが送信される

### 2.7 チャネルアクセストークン平文保存の現状明記（項目7）

- **現状の問題:** 要件定義書ではAES-256-GCM暗号化保存を規定しているが、実装は平文保存。SPECIFICATION.mdに現状の記載がない
- **修正方針:** SPECIFICATION.md 4.3節に「現状は平文保存。将来的にAES-256-GCM暗号化を予定」と明記する
- **修正後のあるべき姿:** セキュリティ方針と現状が仕様書で明確化されている

## 4. 影響範囲

- **項目1,2,5,7（仕様書のみ）:** 既存機能への影響なし
- **項目3（リファクタリング）:** LINE通知送信全般に関わるが、外部動作は不変。リファクタ後にテストで動作確認が必要
- **項目4（カウント変更）:** 抽選結果通知のAPIレスポンスの意味が変わる（通数→人数）。フロントエンド・バックエンド同時変更で吸収。JSONキー名は変えないため外部互換性は維持
- **項目6（日数ラベル）:** 参加予定リマインダーのメッセージ文言が変わる。ユーザー向け改善のみで他機能への影響なし
- **破壊的変更:** なし

## 5. 設計判断の根拠

- **JSONキー名を変更しない判断:** フロントエンドとバックエンドを同時にデプロイできる前提だが、キー名変更は不要なリスクを生むため、意味の変更のみで対応する
- **LINE通知とアプリ内通知のメッセージ差異は仕様:** 監査レポートでは懸念として挙げられたが、意図的な仕様であるため改修対象外とする
- **チャネルアクセストークンの暗号化は今回見送り:** Render.comのマネージド環境であるためリスクは限定的。仕様書に現状を明記するにとどめる
- **`sendOfferResponseConfirmation` の通知種別:** 既存の `WAITLIST_OFFER` を継続使用。専用の通知種別を新設すると通知設定UIの変更も必要になり、費用対効果が低い
