## 妥当性で注意が必要な箇所

### 1. 書き戻しトリガー問題（1-7, 1-8, 1-10, 1-11, 3-22, 3-32）

**問題:** レコードが存在しない（未登録）状態で、伝助に×を書き戻したい。しかし現在の書き戻しは**dirty=trueのレコードを持つプレイヤーのみ**が対象。レコードがなければ書き戻しがトリガーされない。

**例:** ある人が伝助で△をつけた。アプリには取り込まない。しかし×を書き戻すには、その人のdirtyレコードが必要。

**解決案:**
- (A) △/×で取り込み時に、ダミーの`CANCELLED`レコードを`dirty=true`で作成し、書き戻し後に削除
- (B) 書き戻しロジックを変更し、伝助メンバーマッピングがあるプレイヤー全員について書き戻す（dirtyでなくても）
- (C) △の人のみ×を書き戻す必要がある場合に限り、dirty=trueのマーカーレコードを作る

**推奨:** (A)が最もシンプルだが副作用に注意。(B)は確実だが書き戻しの負荷が増える。どちらが良いか判断をお願いします。

### 2. 3-19/3-20: CANCELLED/DECLINEDからの復帰

**問題:** 一度アプリでキャンセル/辞退した人が伝助で再度○をつけた場合、再登録として扱う。これ自体は要件通りだが、**キャンセル理由の記録が上書きされる**点と、**意図しない再登録**の可能性がある。

**リスク:** 伝助で○を入れ直すだけでキャンセルが無かったことになる。悪用というより誤操作のリスク。

**判断:** 要件通り(a)で実装するが、**ログを残す**ことで追跡可能にする。妥当と判断。

### 3. 3-28: OFFEREDの人が伝助で×にした場合

**問題:** アプリでオファー中（応答待ち）の人が伝助で×にした場合、`WAITLIST_DECLINED`にして次の人に繰り上げる。これは妥当だが、**OFFEREDをDECLINEDにする方が既存の処理フロー（`respondToOffer(false)`相当）と整合する**。

**推奨:** `DECLINED`にして`respondToOffer`の辞退ロジックを再利用する方が安全。`WAITLIST_DECLINED`はOFFERを受ける前の辞退なので意味が異なる。→ **`DECLINED`にすべきか？**

### 4. 1-3: 全取消時の書き戻し

**問題:** 締め切り前の全取消で月の全レコードが削除される。削除後はdirtyレコードがないため、書き戻しがトリガーされない可能性がある。

**解決案:** 全取消時に`triggerWriteAsync()`が呼ばれるタイミングで、削除前のdirtyレコードが書き込みキューに載っていれば問題ない（現在のコードでは`registerParticipations`の最後で`triggerWriteAsync()`が呼ばれる）。ただし、書き戻し時点でレコードが存在しないため`toJoinValue(null)`=空白（現在）or ×（新）になる。→ **書き戻しトリガー問題（1と同じ）**
