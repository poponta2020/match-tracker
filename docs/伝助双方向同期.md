## 妥当性で注意が必要な箇所

### 1. 書き戻しトリガー問題（1-7, 1-8, 1-10, 1-11, 3-22, 3-32）

**問題:** レコードが存在しない（未登録）状態で、伝助に×を書き戻したい。しかし現在の書き戻しは**dirty=trueのレコードを持つプレイヤーのみ**が対象。レコードがなければ書き戻しがトリガーされない。

**例:** ある人が伝助で△をつけた。アプリには取り込まない。しかし×を書き戻すには、その人のdirtyレコードが必要。

**解決案:**
- (A) △/×で取り込み時に、ダミーの`CANCELLED`レコードを`dirty=true`で作成し、書き戻し後に削除
- (B) 書き戻しロジックを変更し、伝助メンバーマッピングがあるプレイヤー全員について書き戻す（dirtyでなくても）
- (C) △の人のみ×を書き戻す必要がある場合に限り、dirty=trueのマーカーレコードを作る

**推奨:** (A)が最もシンプルだが副作用に注意。(B)は確実だが書き戻しの負荷が増える。どちらが良いか判断をお願いします。

### 2. 3-19/3-20: CANCELLED/DECLINEDからの復帰

**問題:** 一度アプリでキャンセル/辞退した人が伝助で再度○をつけた場合、再登録として扱う。これ自体は要件通りだが、**キャンセル理由の記録が上書きされる**点と、**意図しない再登録**の可能性がある。

**リスク:** 伝助で○を入れ直すだけでキャンセルが無かったことになる。悪用というより誤操作のリスク。

**判断:** 要件通り(a)で実装するが、**ログを残す**ことで追跡可能にする。妥当と判断。

### 3. 3-28: OFFEREDの人が伝助で×にした場合

**問題:** アプリでオファー中（応答待ち）の人が伝助で×にした場合、`WAITLIST_DECLINED`にして次の人に繰り上げる。これは妥当だが、**OFFEREDをDECLINEDにする方が既存の処理フロー（`respondToOffer(false)`相当）と整合する**。

**推奨:** `DECLINED`にして`respondToOffer`の辞退ロジックを再利用する方が安全。`WAITLIST_DECLINED`はOFFERを受ける前の辞退なので意味が異なる。→ **`DECLINED`にすべきか？**

### 4. 1-3: 全取消時の書き戻し

**問題:** 締め切り前の全取消で月の全レコードが削除される。削除後はdirtyレコードがないため、書き戻しがトリガーされない可能性がある。

**解決案:** 全取消時に`triggerWriteAsync()`が呼ばれるタイミングで、削除前のdirtyレコードが書き込みキューに載っていれば問題ない（現在のコードでは`registerParticipations`の最後で`triggerWriteAsync()`が呼ばれる）。ただし、書き戻し時点でレコードが存在しないため`toJoinValue(null)`=空白（現在）or ×（新）になる。→ **書き戻しトリガー問題（1と同じ）**

## 練習セッション自動作成時の Venue デフォルト値適用 (`DensukeImportService.findOrCreateSession`)

伝助→アプリ同期で `practice_sessions` を新規作成・補完する際、`Venue` エンティティの `defaultMatchCount` / `capacity` を以下のルールで反映する。これにより、伝助で練習日が追加された際の自動セッションでも当日12:00以降の WAITLISTED→WON 自動昇格や空き枠通知が正常動作する。

### 新規セッション作成時

| 会場名マッチ | `totalMatches` | `capacity` | `venueId` |
|------------|---------------|-----------|-----------|
| マッチ | `venue.defaultMatchCount`（null なら伝助スケジュール最大、なければ 3） | `venue.capacity`（null 許容） | `venue.id` |
| 未マッチ | 伝助スケジュールの最大試合番号（なければ 3） | null | null |

### 既存セッションの補完

| ケース | 既存 `venueId` | 既存 `capacity` | 会場名マッチ | 処理 |
|-------|--------------|----------------|------------|------|
| A | null | null | マッチ | `venueId` と `capacity` を同時補完 |
| A' | null | 設定済み | マッチ | `venueId` のみ補完（`capacity` は保護） |
| B | 設定済み | null | — | Venue から id 逆引きで `capacity` のみ補完 |
| — | 設定済み | 設定済み | — | 何もしない（save 呼び出しなし） |

- `totalMatches` および設定済みの `venueId` は**常に保護**（管理者の手動調整を尊重）
- `capacity` が既設定なら Venue 値で上書きしない
- `practiceSessionRepository.save()` は `changed` フラグで集約し、不要な呼び出しを抑制する

### 設計判断

- **「補完」は null 時のみ充填**: 管理者が手動で設定した値（capacity, totalMatches）を Densuke 同期が上書きすることによる事故を防ぐ
- **試合数は Venue 優先**: 伝助スケジュールの最大試合番号は「現時点で参加者が登録されている試合の最大値」でしかなく、Venue 側の値が「管理者が明示的に設定した正の値」であるため、Venue を優先する

## 練習日 push 同期 (DensukeScheduleWriteService)

参加者出欠の双方向同期（本ドキュメントのメイン）とは独立した、**練習日（スケジュール）自体のアプリ→伝助 push 同期**。
出欠（`dirty` フラグ駆動）とは別軸の同期方向で、本マトリクスとは独立して動作する。
詳細仕様は `docs/features/densuke-schedule-write-sync/fix-requirements.md` 参照。

### 同期方向

- **アプリ→伝助**: アプリで新規追加した練習日を伝助ページの候補日程欄に末尾追記（追加のみ）
- **伝助→アプリ**: 既存 `DensukeImportService.findOrCreateSession` で対応（変更なし）
- **削除同期**: スコープ外（伝助ページ管理者編集 UI 経由で手動運用）

### push 経路

| トリガー | メソッド | 失敗通知 |
|---------|---------|---------|
| `PracticeSessionService.createSession` の `afterCommit` | `pushNewSchedulesToDensukeAsync` (`@Async`) | 管理者 LINE 通知 (`ADMIN_DENSUKE_PUSH_FAILED`) |
| `DensukeSyncService.syncAll` の 5 分スケジューラ | `pushAllForCurrentAndNextMonth` → `pushSilently` | WARN ログのみ（フラッディング防止） |

### 重複防止と過去日制約

- push 前に `DensukeScraper.scrape(url, year)` で伝助の現スケジュールを read
- アプリ側 `practice_sessions` のうち伝助に存在しない日付のみ POST `/update` で送信
- 既存日程・既存参加者・`densuke_row_ids` のインデックスは保護される（伝助 `/update` は末尾追記であることをスパイク調査で実証済み）
- **過去日は push しない**: 伝助 `/update` は候補日程を**末尾追記**しかできないため、伝助の既存最大日付より前の
  新規日付を push すると伝助 DOM 出現順 `[既存..., 過去日]` とアプリ側日付昇順 `[過去日, 既存...]` がずれ、
  `DensukeWriteService.parseAndSaveRowIds` の row id 対応が誤って参加者出欠が別日に書き込まれるデータ破壊が
  発生する。差分セッションのうち伝助既存最大日付以前のものは push 対象から除外し、即時 push 経路では
  `ADMIN_DENSUKE_PUSH_FAILED` 通知で管理者に「伝助ページの管理画面から手動追加してください」を伝える
  （スケジューラ経路はフラッディング防止で通知抑制）

### 無限ループ防止

- `DensukeImportService.findOrCreateSession` は `practiceSessionRepository.save` を直接呼び
  `PracticeSessionService.createSession` を経由しないため、`afterCommit` フックが発火しない
- 結果として、伝助→アプリ取り込みで作成された練習日が再度伝助に push されることはない（構造的保証）

### 並行制御

- `DensukeUrlRepository.findByYearAndMonthAndOrganizationIdForUpdate`（`@Lock(PESSIMISTIC_WRITE)` 行ロック）で
  同一 (year, month, organizationId) の並行 push をシリアライズ
- 伝助 read → write の間に他リクエストが割り込んで差分計算がズレるリスクを回避

### スケジューラ実行順序（`DensukeSyncService.syncAll`）

1. **スケジュール push** （本機能 — `pushAllForCurrentAndNextMonth`）
2. 参加者書き込み（既存 — `densukeWriteService.writeToDensuke`）
3. 伝助→アプリ取り込み（既存 — `syncForMonth` × 当月・翌月）

スケジュールがそろってから参加者を書き、最後に伝助の最新状態を取り込む順序。
