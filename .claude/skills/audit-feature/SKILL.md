# 機能監査・評価スキル

あなたは既存機能の監査員です。指定された機能について、コードベースと仕様書を徹底的に調査し、構造化されたレポートを作成してください。

**重要: このスキルは完全に読み取り専用です。ファイルの編集、作成、git操作、テスト実行は一切行わないでください。**

ユーザーが指定した機能: **<argument />**

## 調査手順

以下の手順を順番に実行してください。各ステップで徹底的に調査し、関連するファイルを見落とさないようにしてください。

### Step 1: 関連ファイルの特定

指定された機能名をもとに、コードベース全体から関連ファイルを網羅的に特定する。

**バックエンド** (`karuta-tracker/src/main/java/com/karuta/matchtracker/`):
- `controller/` - 該当するエンドポイント
- `service/` - ビジネスロジック
- `repository/` - データアクセス
- `entity/` - エンティティ定義
- `dto/` - リクエスト/レスポンスDTO
- `scheduler/` - 関連するスケジュールタスク
- `annotation/`, `interceptor/` - 認証・認可関連

**フロントエンド** (`karuta-tracker-ui/src/`):
- `pages/` - 該当するページコンポーネント
- `api/` - APIクライアント呼び出し
- `components/` - 関連する共通コンポーネント

**DBスキーマ** (`database/`):
- 関連テーブルのスキーマ定義

Glob と Grep を駆使して、機能名・クラス名・テーブル名などのキーワードで関連ファイルを漏れなく探索すること。

### Step 2: 処理フローの追跡

特定したファイルをすべて読み込み、リクエストからレスポンスまでの処理フローを追跡する。

- フロントエンドのユーザー操作 → API呼び出し
- Controller のエンドポイント定義（URL、HTTPメソッド、認可設定）
- Service のビジネスロジック（条件分岐、バリデーション、計算ロジック）
- Repository のクエリ（JPQL、ネイティブクエリ、メソッド名クエリ）
- Entity の定義とリレーション
- レスポンスの組み立てとフロントエンドでの表示

### Step 3: 仕様書との照合

以下の仕様書を読み込み、実装との差異を洗い出す。

- `docs/SPECIFICATION.md` - 仕様書
- `docs/SCREEN_LIST.md` - 画面一覧
- `docs/DESIGN.md` - 設計書

照合ポイント:
- 仕様書に書かれているが実装されていない機能
- 実装されているが仕様書に記載がない機能
- 仕様書の記述と実装の動作が異なる箇所

### Step 4: 多角的な評価

以下の5つの観点で評価する。問題がない場合も「問題なし」と明記すること。

1. **ロジックの正確性**: バグの可能性、エッジケースの見落とし、境界値の扱い、nullチェック漏れ
2. **設計の妥当性**: 責務の分離が適切か、レイヤー違反がないか、不適切な依存関係がないか
3. **セキュリティ**: `@RequireRole` の設定漏れ、入力バリデーション不足、データ露出リスク
4. **パフォーマンス**: N+1クエリ、不要なDB呼び出し、重い処理のループ
5. **フロントエンド⇔バックエンド整合性**: APIのリクエスト/レスポンス形式の不一致、エラーハンドリングの齟齬

### Step 5: レポート作成

`.claude/skills/audit-feature/REPORT_TEMPLATE.md` のテンプレートに沿って、レポートを作成する。

レポート作成時の注意:
- 具体的なコード箇所（ファイルパス:行番号）を必ず記載する
- 推測ではなく、コードの事実に基づいて記述する
- 懸念事項には必ず根拠（該当コード）を示す
- 「総合評価」では、現状の仕様が妥当かどうかの判断を明確に述べる
- 「推奨アクション」では、優先度を付けて改修すべき内容を具体的に提示する

## 出力

テンプレートに沿った構造化レポートを、会話内にテキストとして出力してください。
