# 最終チェック・仕上げスキル

`/define-feature` で定義し `/implement` で実装した機能の最終チェックを行う。
要件との整合性、コード品質、ドキュメント更新を確認し、問題があれば修正する。
全チェック通過後、親Issueをクローズして機能を完了とする。

**ユーザーとの対話はすべて日本語で行うこと。**

---

## Step 1: 対象機能の特定

### 引数ありの場合
`$ARGUMENTS` で指定された機能名に対応する `docs/features/<機能名>/` ディレクトリを探す。
見つからない場合はユーザーにエラーを伝える。

### 引数なしの場合
`docs/features/` 配下のディレクトリを走査し、`implementation-plan.md` の全タスクが完了済み（`- [x]` のみ）かつ親Issueが未クローズの機能を一覧表示する。

```
最終チェック可能な機能:
1. <機能名A>
2. <機能名B>
どの機能の最終チェックを行いますか？
```

該当する機能がない場合は、理由を伝える（「全タスク完了済みの機能がありません」等）。

---

## Step 2: 前提条件の確認

以下のファイルを読み込む：
- `docs/features/<機能名>/requirements.md` — 要件定義書
- `docs/features/<機能名>/implementation-plan.md` — 実装手順書

### チェック内容
- `requirements.md` の `status` が `completed` であること
- `implementation-plan.md` の `status` が `completed` であること
- `implementation-plan.md` の全タスクのチェックボックスが `- [x] 完了` であること

### 未完了タスクがある場合
未完了タスクをリストアップし、ユーザーに報告して中断する。

```
以下のタスクが未完了です:
- タスク3: フロントエンド画面実装
- タスク5: テスト追加

先に `/implement <機能名>` で残りのタスクを完了してください。
```

---

## Step 3: 実装の正確性チェック

`requirements.md` の要件と実際のコードベースを突き合わせて、要件通りに実装されているか確認する。

### 3a. API設計の確認
- 要件定義書に記載されたエンドポイントが実装されているか
- URL、HTTPメソッドが一致しているか
- リクエスト/レスポンスの形式が一致しているか

### 3b. DB設計の確認
- 要件定義書に記載されたテーブル・カラムが存在するか
- `database/` 配下のスキーマファイルに反映されているか

### 3c. 画面仕様の確認
- 要件定義書に記載された画面・コンポーネントが実装されているか
- 操作フローが要件通りか（ルーティング、ナビゲーション）

### 3d. ビジネスルール・バリデーションの確認
- 要件定義書に記載されたビジネスルールがコードに反映されているか
- バリデーションルールが実装されているか
- エラーケースの処理が実装されているか

### チェック結果の記録
問題があれば内部的にリストに記録し、Step 6 でまとめて報告する。

---

## Step 4: コード品質チェック

以下のコマンドを順に実行し、結果を確認する。

### 4a. バックエンドビルド
```bash
cd karuta-tracker && ./gradlew build
```

### 4b. バックエンドテスト
```bash
cd karuta-tracker && ./gradlew test
```

### 4c. フロントエンドビルド
```bash
cd karuta-tracker-ui && npm run build
```

### 4d. フロントエンドLint
```bash
cd karuta-tracker-ui && npm run lint
```

**注意:** これらのコマンドはプロジェクト構成に依存する。プロジェクトのルートに `karuta-tracker/` や `karuta-tracker-ui/` が存在しない場合は、実際の構成に合わせて調整する。

### テスト失敗時の対応
テストが失敗した場合は、`test-automator` エージェントを使って原因を分析し修正する。修正後に再度テストを実行し、通過するまで繰り返す。

### セキュリティ関連機能の追加チェック
今回の機能が認証・認可・APIエンドポイント・入力バリデーション等に関わる場合は、`security-auditor` エージェントを使って実装のセキュリティ確認を行う。

### チェック結果の記録
失敗やWarningがあれば内部的にリストに記録し、Step 6 でまとめて報告する。

---

## Step 5: ドキュメント更新チェック

CLAUDE.md のドキュメント更新ルールに基づき、以下のファイルが最新の状態に更新されているか確認する。

### 対象ドキュメント
- `docs/SPECIFICATION.md` — 仕様書
- `docs/SCREEN_LIST.md` — 画面一覧
- `docs/DESIGN.md` — 設計書

### チェック方法
1. 各ドキュメントを読み込む
2. 今回の機能追加・変更に関する記述が含まれているか確認する
3. 新しい画面があれば `SCREEN_LIST.md` に追加されているか
4. 新しいAPIやテーブルがあれば `DESIGN.md` に反映されているか
5. 機能の仕様が `SPECIFICATION.md` に記載されているか

### チェック結果の記録
不足があれば内部的にリストに記録し、Step 6 でまとめて報告する。

---

## Step 6: 問題の報告と修正

Step 3〜5 で検出した全ての問題をまとめてユーザーに報告する。

### 問題がある場合

```
## 最終チェック結果

### 要件との差異（N件）
- [ ] ○○エンドポイントのレスポンスに△△フィールドが不足
- [ ] ○○画面のバリデーションルールが未実装

### コード品質（N件）
- [ ] フロントエンドLintエラー: ○○
- [ ] テスト失敗: ○○

### ドキュメント未更新（N件）
- [ ] SPECIFICATION.md に○○機能の記載がない
- [ ] SCREEN_LIST.md に○○画面が追加されていない

修正を行いますか？
```

ユーザーの確認を得てから、問題を1つずつ修正する。
各修正後、修正内容を簡潔に報告する。

### 問題がない場合
「全てのチェックに通過しました。」と伝え、Step 7 に進む。

---

## Step 7: 仕上げ

### 7a. 修正のコミット
Step 6 で修正を行った場合、変更をコミットする。

```bash
git commit -m "$(cat <<'EOF'
<機能名> 最終チェックでの修正

- <修正内容1>
- <修正内容2>
EOF
)"
```

### 7b. 親Issueのクローズ
`implementation-plan.md` から親Issue番号を特定し、ユーザーに確認の上クローズする。

```
親Issue #<番号> をクローズします。よろしいですか？
```

ユーザーの確認を得てから：

```bash
gh issue close <親Issue番号>
```

### 7c. 完了報告

```
## 最終チェック完了

- 機能: <機能名>
- 要件チェック: OK
- コード品質: OK（ビルド・テスト・Lint通過）
- ドキュメント更新: OK
- 修正コミット: <ハッシュ>（修正があった場合）
- 親Issue: #<番号> クローズ済み

この機能の開発は全て完了しました。
```
